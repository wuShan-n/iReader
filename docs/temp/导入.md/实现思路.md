下面给你一套**生产级、可长期演进**的「导入功能」实现方案（TXT + EPUB + PDF），完全贴合你已经优化过的模块架构（`core/files + core/data + core/database + core/reader/runtime + engines/* + core/work + feature/library`），重点是：

* **可恢复（Resumable）**：进程被杀/重启后能继续导入
* **可观测（Observable）**：UI 随时能看到每一本的状态/进度/错误
* **可扩展（Pluggable）**：新增格式只需要加 `engine-*` + 注册，不改 feature
* **强一致（Idempotent）**：重复导入同一本不会产生脏数据/重复记录
* **不依赖引擎实现细节**：feature 永远不直接依赖 `engines/*`

---

## 1. 设计目标与关键决策

### 1) 最重要的决策：导入后是否“复制到应用私有库”

**我强烈推荐：导入=复制（ingest）到 App 管理的私有存储（或 app-specific external）**，而不是长期依赖 SAF Uri 原地读取。

原因（都是线上会踩的坑）：

* SAF Uri 权限可能丢失（分享 intent 临时授权、ROM bug、用户手动撤销）
* 原文件可能被用户移动/删除，导致库里书“失效”
* 引擎层（尤其 PDF/EPUB）对 random access 更友好的是 **File/FD**，复制后可控、性能稳定
* 做全文索引/缩略图/缓存时更容易做**一致性与清理策略**

> 结论：**“导入”是把外部文件“入库”到你的 BookStore**；外部来源只作为“source 记录”。

## 2. 模块落点（与你目录完全匹配）

把导入分成三层：**文件入库（core/files）→ 书籍落库（core/data + core/database）→ 后处理（core/work + engines）**。

### core/files 负责：把“外部内容”变成“库内文件”

建议新增包：

```
core/files/import/
  ImportSource.kt              // Uri / tree / share intent 等的统一抽象
  UriPermissionManager.kt      // persistable permission 封装
  FormatSniffer.kt             // 扩展名 + mime + magic bytes
  BookFileStore.kt             // 库内存储策略（路径、临时文件、原子落盘）
  FileIngestor.kt              // copy + checksum + progress
  StoragePolicy.kt             // 内部/外部、容量检查、命名策略
```

### core/data 负责：导入状态机 + DB 写入 + 去重策略

建议新增：

```
core/data/import/
  ImportRepository.kt
  ImportRepositoryImpl.kt
  ImportPolicy.kt              // 去重、覆盖策略、是否立即生成封面等
  ImportCoordinator.kt         // “处理一个 item”的纯业务逻辑（可单测）
  ImportProgressBroadcaster.kt // 将 worker progress 映射到 db/flow
```

### core/database 负责：导入可恢复的持久状态

新增/扩展表（下面第 5 节会给结构）

### core/work 负责：批量导入、后处理任务编排

新增 Worker：

* `ImportSessionWorker`：处理一个 session（多文件）
* `PostProcessBookWorker`：封面/缩略图/索引（每本书独立，便于重试）

### core/reader/runtime 负责：引擎选择 + 统一 probe/open

* Import 不应该知道 TXT/EPUB/PDF 的实现细节
* Import 只调用 runtime：`EngineRegistry.select(format)` + `engine.probe()`（或 `open(PROBE)`）

### engines/* 负责：格式内的 metadata/cover/验证能力（对 runtime 暴露统一接口）

* TXT 引擎：编码探测/章节识别（可选）/索引构建（可选）
* EPUB 引擎：OPF metadata + cover + toc
* PDF 引擎：pageCount + docInfo + 首页面渲染缩略图（outline/text 可选）

## 3. 导入流程（强烈推荐：持久化状态机 + WorkManager 可恢复）

导入一定要当作一个**持久化的状态机**来做，而不是“UI 点一下，直接 copy 然后 insert”。

### 状态机（每个文件一条 ImportItem）

建议枚举：

* `PENDING`：已创建 item，等待处理
* `INGESTING`：拷贝中（有字节进度）
* `DEDUP_CHECK`：计算完 hash，查重
* `PROBING`：调用引擎 probe（读 metadata/校验）
* `PERSISTING`：事务写 DB（Book + File + 初始 Progress）
* `POST_PROCESSING`：封面/索引等 worker 阶段
* `DONE`：成功
* `SKIPPED_DUPLICATE`：重复导入，跳过（可返回已存在 bookId）
* `FAILED`：失败（带错误码 + 可重试标识）
* `CANCELLED`

### 整体时序（推荐做法）

1. UI 选文件 → 创建 ImportSession + ImportItems（落 DB）
2. enqueue `ImportSessionWorker(sessionId)`（唯一任务名）
3. Worker 循环处理 items，逐条更新状态 + 进度
4. 每导入成功一本，enqueue `PostProcessBookWorker(bookId)`
5. UI 只需要订阅 DB：`observeImportSession(sessionId)` / `observeItems(sessionId)`

> 关键收益：导入过程与 UI 解耦，杀进程也不丢；导入失败也能精确重试某一本。

## 4. 核心接口设计（推荐落地的 Kotlin 契约）

### 4.1 ImportSource：统一入口（SAF 文件 / 文件夹 / share）

```kotlin
sealed interface ImportSource {
  val displayName: String?
  val sizeBytes: Long?
  val mimeType: String?
  val lastModified: Long?

  data class DocumentUri(val uri: Uri, val persistable: Boolean) : ImportSource
  data class TreeUri(val treeUri: Uri, val persistable: Boolean) : ImportSource
  data class ShareUri(val uri: Uri) : ImportSource // 临时授权，必须立刻 ingest
}
```

TreeUri 的处理：在 `core/files` 做 `DocumentFile.fromTreeUri` 扫描，产出一堆 `DocumentUri`，再走同一 pipeline。

---

### 4.2 FileIngestor：复制 + hash + 原子落盘（关键）

导入最容易出事故的部分：**拷贝半截、hash 算错、写入不原子、磁盘不足、进度乱跳**。

推荐实现要点：

* copy 到 `*.partial` 临时文件
* 边 copy 边计算 `sha256`
* copy 完再 `fsync`（尽量）+ 原子 rename 到最终路径
* 每隔 200~500ms 或每 N MB 更新一次进度到 DB（避免频繁写 DB）
* 遇到取消，立即停止并清理 partial

---

### 4.3 格式识别 FormatSniffer（扩展名 + mime + magic bytes）

推荐规则（稳定且够用）：

* PDF：开头 `%PDF-`
* EPUB：ZIP（`PK..`）且 zip 内 `mimetype` 文件内容为 `application/epub+zip`
* TXT：mime `text/*` 或扩展 `.txt`，否则 fallback

> 不要只信 extension；很多用户下载的文件扩展名是错的。

---

### 4.4 引擎 Probe（强烈建议你在 `core/reader/api` 加这一能力）

导入时你需要**轻量校验 + metadata 提取 + cover 提取**，但又不想真正打开阅读 session。

最推荐的抽象：

```kotlin
data class DocumentProbe(
  val format: BookFormat,
  val title: String?,
  val authors: List<String>,
  val language: String?,
  val identifier: String?,        // epub package id / pdf doc id(若有)
  val pageCount: Int?,            // pdf
  val warnings: List<String>,
  val capabilities: DocumentCapabilities,
  val cover: CoverResult?         // bytes or reference
)

interface ReaderEngine {
  val format: BookFormat
  suspend fun probe(source: Source, options: ProbeOptions = ProbeOptions()): ReaderResult<DocumentProbe>
}
```

`cover` 推荐是 `bytes + mime`（导入层负责统一转成 webp/jpg 存盘），不要让引擎直接写文件。

这样 ImportCoordinator 不关心 EPUB/PDF/TXT 的实现，只拿 probe 结果入库即可。

## 5. 数据库设计（Room，保证可恢复与幂等）

### 5.1 书籍与文件（核心）

**BookEntity**

* `bookId` (UUID, PK)
* `title`
* `authorsJson`
* `format` (TXT/EPUB/PDF)
* `coverPath` (nullable)
* `createdAt`
* `lastOpenedAt`
* `deletedAt`（可选：支持回收站）

**BookFileEntity**

* `fileId` (UUID, PK)
* `bookId` (FK)
* `storagePath`（你库内文件路径）
* `originalDisplayName`
* `mimeType`
* `sizeBytes`
* `sha256`（强烈建议建 unique index）
* `sourceUri`（可选，记录来源；不要依赖它读）
* `importedAt`

> `sha256` 做 unique index：你就能天然实现“重复导入幂等”。

### 5.2 导入会话（关键：恢复、进度、失败可重试）

**ImportSessionEntity**

* `sessionId` (UUID, PK)
* `createdAt`
* `status`（RUNNING/DONE/PARTIAL_FAILED/CANCELLED）
* `totalCount`
* `doneCount`
* `failedCount`

**ImportItemEntity**

* `itemId` (UUID, PK)
* `sessionId` (FK)
* `sourceUri`（字符串）
* `displayName`
* `sizeBytes`
* `sniffedFormat`
* `status`（上面的状态机）
* `progressBytes`
* `sha256`（nullable，ingest 完后填）
* `bookId`（nullable，成功后填 / duplicate 指向已存在 book）
* `errorCode` + `errorMessage` + `stacktraceSnippet`（可选）
* `retryCount`
* `updatedAt`

这样 UI 可以做到：

* “这个 session 导入了 37/120”
* “失败 5 本，点开看原因并逐本重试”
* “重复的 12 本自动合并到已存在书籍”

## 6. WorkManager 编排（不做 MVP 的关键：可恢复 + 可取消 + 可前台）

### 6.1 ImportSessionWorker（一个 session 一个唯一 work）

* UniqueWorkName：`import_session_<sessionId>`
* `ExistingWorkPolicy.KEEP`（避免重复点击造成并发导入同 session）

Worker 行为：

1. 读取 session items：`status in (PENDING, FAILED with retryable)`
2. 按顺序处理（建议默认串行，必要时小并发）
3. 每处理一个 item 更新 DB
4. 成功后 enqueue `PostProcessBookWorker(bookId)`
5. session 完成后更新 session status

**为什么默认串行更推荐？**

* 批量导入本质 IO 密集 + 容易触发磁盘/内存峰值
* 并发过高会让 UI 卡、设备发热、失败率上升
* 真要并发，也建议用 `Semaphore(2)` 限制并发度

### 6.2 ForegroundInfo（必须）

大文件导入（尤其 PDF/大 EPUB）在后台很容易被系统干掉。
推荐对 ImportWorker 使用前台通知（foreground service style）：

* 通知显示：当前导入书名 + 总进度
* 提供“取消导入” action → cancel unique work

## 7. ImportCoordinator：一条 item 的“纯业务处理”步骤

下面是你真正要落地的“最推荐顺序”（每一步都可单测）：

1. **预检**

* 检查可读权限（Uri 可 openInputStream）
* 如果是 persistable Uri：尝试 `takePersistableUriPermission`
* 检查剩余空间（`StatFs`），不足直接失败（错误码 `NO_SPACE`）

2. **INGEST：复制入库 + sha256**

* `BookFileStore.allocateTemp(targetExt)`
* `FileIngestor.copyAndHash(sourceUri, tempFile)`
* 成功：得到 `sha256 + sizeBytes + tempPath`

3. **DEDUP：查重**

* `bookFileDao.findBySha256(sha256)`

  * 找到：删除 tempFile → item = `SKIPPED_DUPLICATE`，绑定 existing `bookId`
  * 未找到：继续

4. **FORMAT：最终格式确认**

* `FormatSniffer`（必要时读取头部/zip mimetype）
* 得到 `BookFormat`

5. **PROBING：引擎 probe**

* `engine = EngineRegistry.get(format)`
* `probe = engine.probe(FileSource(finalFilePath))`
* 若 probe 失败：标记 `FAILED_CORRUPT/UNSUPPORTED/DRM/PASSWORD_REQUIRED`

6. **PERSISTING：事务写 DB**
   Room transaction 内：

* insert BookEntity（title/authors 等优先来自 probe，fallback 到文件名）
* move tempFile → finalPath（或先 move 再 insert，取决于你的原子策略）
* insert BookFileEntity（sha256、path、size、sourceUri）
* 初始化 ProgressEntity（可选：locator = start）

7. **enqueue 后处理**

* `PostProcessBookWorker(bookId)`
* item 标记 `POST_PROCESSING` 或直接 `DONE`（看你是否把后处理当作“导入完成”的一部分）

> 推荐：导入“成功入库”即可算 DONE；后处理失败不影响书能打开，只是缺封面/索引。

## 8. 后处理（PostProcessBookWorker）：封面/缩略图/索引

分三类任务（强烈建议拆开，失败可重试、不会阻塞导入）：

### A) Cover/Thumbnail（建议默认开启）

* TXT：生成“文字封面”（标题+作者），存 webp/jpg
* EPUB：用 probe 返回的 cover bytes，统一转格式并缩放
* PDF：渲染第 1 页生成缩略图（建议 300~600px 边长级别）

### B) 结构信息缓存（可选但推荐）

* EPUB：解析并缓存 toc/outline（如果 probe 没做完整）
* PDF：outline（若你的 pdf 引擎支持）
* TXT：章节识别结果（可选）

### C) 搜索/索引（建议按需或电量充足时做）

* TXT：构建字节 offset 索引（显著提升跳转/搜索性能）
* EPUB：构建 spine → locator 映射缓存
* PDF：文本抽取与索引（非常重，建议开关 + 充电/Wi‑Fi/空闲时）

可用 WorkManager constraints：

* `requiresCharging(true)`
* `requiresDeviceIdle(true)`（谨慎，太严格会导致永远不跑）
* 或你在设置里做策略选择

## 9. 三种格式的“最佳实践细节”

### 9.1 TXT（别低估它的坑）

**核心问题：编码 + 大文件随机访问 + 搜索性能**

推荐策略：

* 编码探测优先级：

  1. BOM（UTF-8/UTF-16LE/UTF-16BE）
  2. 尝试 UTF-8 严格解码（失败则 fallback）
  3. fallback：`GB18030`（中文场景非常重要）
* probe 输出：`encoding`（建议放到 `BookFileEntity.extraJson` 或单独字段）
* 大 TXT：导入后后台建立 `offset index`（每 N KB 记录一个 byte offset），阅读页跳转/分页会稳定很多

### 9.2 EPUB

**核心问题：zip 校验 + 元数据一致性 + cover 获取 + toc**

推荐 probe 做到：

* 校验 `mimetype`（正确 EPUB）
* 解析 container.xml → OPF
* title/authors/language/identifier
* cover：通过 OPF meta 或 manifest item 找 cover
* toc：nav.xhtml 或 toc.ncx（至少拿到章节树）

导入阶段不解压到永久目录：

* 原始 `.epub` 作为“源文件”入库保存
* 解压/资源缓存放 `cache/`，由引擎按需管理与清理

### 9.3 PDF

**核心问题：渲染器选择 + 密码/损坏 + 缩略图/页数**

推荐 probe 做到：

* header `%PDF-` 校验
* pageCount（渲染器/解析器能快速得到）
* 文档信息（title/author，如果有）
* 如果检测到密码保护：明确错误码 `PASSWORD_REQUIRED`（UI 弹出输入，重新 probe）

缩略图生成：

* 用引擎渲染第 1 页到 bitmap，再压缩存储
* 注意：别在 import 主流程里做渲染（会拖慢导入），放到 PostProcess worker

## 10. feature/library 的 UI/交互（不 MVP 的体验标准）

推荐你在 `feature/library` 做三段式体验：

1. **选择来源**

* 文件：`ACTION_OPEN_DOCUMENT` + multi-select
* 文件夹：`ACTION_OPEN_DOCUMENT_TREE`（扫描 tree）
* 分享入口：`ACTION_SEND/ACTION_VIEW`

2. **导入预览页（强烈推荐）**

* 列表展示：文件名、大小、识别格式、可能的标题（快速推断）
* 显示可选策略：

  * 重复文件：跳过 / 覆盖元数据 / 作为新副本（一般不推荐新副本）
  * PDF 密码：如果 probe 失败，允许用户输入后重试

3. **导入进度页（可后台）**

* session 总进度条 + 每本书状态
* 失败项：展示原因 + “重试”按钮（只重试失败项，不重跑整个 session）
* 支持取消 session

UI 数据来源只订阅：

* `ImportSessionDao.observe(sessionId)`
* `ImportItemDao.observeBySession(sessionId)`
* `BookRepo.observeBooks()`（导入完成自动出现在书架）

## 11. 去重策略（推荐默认行为）

> 最常见真实场景：用户从不同地方反复导入同一本书。

默认推荐策略（简单且符合直觉）：

* 以 `sha256` 为唯一标识
* 若重复：

  * ImportItem 标记 `SKIPPED_DUPLICATE`
  * 返回已存在 `bookId`
  * 记录一个新的 `sourceUri`（可选：维护 `BookSourceEntity` 列表）

如果你需要“同一本文件多副本”（很少见），再引入 `contentId` + 引用计数；但对阅读器通常是负收益（管理复杂、删除难）。
