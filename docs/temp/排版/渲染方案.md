下面按“能长期维护、功能能做深、可扩展”的标准给你推荐（不是“凑合能跑”的方案）。核心思路是：**TXT/EPUB 属于可重排（reflowable）内容**，渲染引擎要能随着字号/行距/边距/字体变化重新分页；**PDF 属于固定版式（fixed-layout）**，渲染引擎要能做高性能缩放、分块/缓存渲染、并尽量有文字层能力（选择/搜索/注释）。

---

## 总体架构建议：两条渲染管线 + 统一阅读状态/标注模型

* **ReflowableRenderer（TXT / EPUB reflow）**

  * 关键：分页（pagination）是“函数”：`(内容, 样式参数, 屏幕尺寸) -> PageBreaks`
  * 需要：分页缓存、预分页、样式变更后失效策略、章节/资源级别的拆分加载。
* **FixedLayoutRenderer（PDF / EPUB FXL）**

  * 关键：以“页”为基本单位，缩放时尽量做**tile（分块）渲染**或多级缓存（避免整页超大 bitmap）。
* **统一位置（Location）与标注（Annotation）**

  * TXT：`byteOffset/charOffset + chapterId`
  * EPUB：`Locator/CFI`（Readium 体系）
  * PDF：`pageIndex + quadPoints/rects（文字选择区域）`
  * 用 sealed class/接口封装，别让 UI 层知道各格式细节。

这样你以后加 `mobi/azw3/cbz` 或替换 PDF 引擎，不会全盘重做。

---

## TXT：最推荐“原生排版 + 自研分页/缓存”，不要用 WebView 直接糊

### 推荐方案

**用 Android 原生文字排版（StaticLayout / Compose TextLayout）做分页，按页渲染显示。**

为什么这是 TXT 的最优解：

* TXT 本质是纯文本，原生排版链路最轻，**内存和性能最可控**（尤其大文件）。
* 你可以做到阅读器该有的体验：真分页翻页、智能章节、搜索、进度定位、夜间模式、字体替换、行距段距、首行缩进等。
* 可做“超大 txt”（几十 MB 甚至更大）：用 **mmap/分段读取** + 章节索引，避免一次性读入内存。

### 工程要点（非 MVP 的关键点）

1. **编码检测与规范化**

   * UTF-8/GBK/GB18030 常见；建议：自动探测 + 用户可手动切换。
2. **超大文件读取**

   * 不要 `readText()` 全读；用流式读取 + 索引（或 mmap）。
3. **章节识别**

   * 正则/规则：`第.{1,9}章`、`CHAPTER`、行长度、空行密度等；并允许用户手动“重新分章”。
4. **分页缓存**

   * 样式参数（字号/行距/边距/字体/屏幕宽高）作为 cache key。
   * 保存 page breaks（例如：每页起始 char offset），翻页 O(1)。
5. **搜索**

   * 大文件：增量扫描 + 可选全文索引（如 Room FTS / 自建倒排）。
6. **选中/高亮**

   * 纯原生排版要做“跨页选中”会更复杂，但可实现；或者以“段落块”为单位做可选中区域。

> 什么时候会选 WebView 渲 TXT？
> 如果你强烈想“复用 EPUB 的 CSS 主题、选中/高亮 JS 逻辑”，可以把 TXT **分章转成 HTML** 再用 WebView 渲染。但这对超大 TXT、定位映射（byteOffset ↔ DOM）和性能更难控，一般不如原生分页稳。

---

## EPUB：最推荐 **Readium Kotlin Toolkit（WebView 渲染 + 标准化导航/定位）**

### 推荐方案

**直接上 Readium Kotlin Toolkit 作为 EPUB 引擎**：Streamer 负责解析，Navigator 负责渲染（底层 WebView + JS bridge）。Readium 明确支持 EPUB2/EPUB3，并且提供分页、RTL、装饰/高亮、TTS 等阅读器能力。([Readium][1])

为什么这是“专业方案”：

* **标准兼容**：EPUB3/排版/CSS/资源引用/导航（spine、toc）这些坑非常多，自己实现成本巨大。
* **阅读器核心能力齐全**：分页/滚动、RTL、装饰（高亮等）、TTS、（对文本内容的）搜索等在工具链层面就考虑到了。([Readium][1])
* **工程生态**：Readium 以“阅读器 SDK”方式组织模块，可按需引入（shared/streamer/navigator/opds/lcp）。([Readium][1])
* **许可证友好**：项目本身是 BSD-3-Clause，商业闭源也可用（注意 LCP 另算）。([GitHub][2])

### 工程要点（你做“不要 MVP”时会用到）

1. **阅读偏好（Preferences）统一**

   * 字号、字体、主题色、页边距、行距等，统一映射到 Readium 的 Navigator preferences（以及你 TXT 的分页参数）。
2. **定位/进度**

   * 位置用 Readium 的 `Locator`（通常含 href + progression + CFI），能稳定跨设备同步。
3. **标注体系**

   * Readium 的 Decoration API 很适合做高亮/书签；后端存储要用你自己的统一 Annotation model，再映射回 Decoration。([Readium][1])
4. **字体与排版**

   * 自带字体/用户字体、CJK 排版、竖排、连字等要测试；WebView 渲染要注意字体加载与性能。
5. **DRM（如果你做正版分发）**

   * Readium 支持 LCP，但需要额外的二进制 `liblcp`（EDRLab 提供）。([Readium][1])

## PDF：没有“唯一最推荐”，取决于你要的功能深度 + minSdk；但有清晰的最优选择区间

PDF 的坑比 EPUB 更深：渲染、缩放、文字层、表单、签名、注释、重排……所以“专业方案”通常意味着**选一个成熟 PDF 引擎**，而不是自己从 PdfRenderer 拼起来（除非你只做最基础渲染）。

### A) 你只做“阅读 + 选择 + 搜索”，且目标系统较新：优先看 **AndroidX PDF Viewer**

AndroidX 有官方 `androidx.pdf:pdf-viewer(-fragment)`，提供 Fragment/View/Composable 形态，并在 release notes 里明确包含缩放、快速滚动、文本长按选择、文件内搜索、超链接处理等能力。([Android Developers][3])

但要注意两点（这是“不要 MVP”必须讲清楚的风险）：

* **目前仍是 alpha（到 2026-02-25 还是 1.0.0-alpha14）**。([Android Developers][3])
* **兼容性有门槛**：早期仅 Android V（SDK 35），之后扩到 Android S/T/U/V（大致 Android 12+）。([Android Developers][3])
  如果你要覆盖大量低版本用户，这条路可能不合适。

### B) 你要覆盖广泛 Android 版本（minSdk 21+）、预算偏开源：选 **PDFium（推荐）**

开源路线里，最常见且现实的是 **PDFium**：

* **PdfiumAndroidKt** 提供的是“PDFium 的 Android binding”，专注渲染/能力，不自带 UI（适合你做专业自定义 UI）。([GitHub][4])
* 如果你想快速得到一个可用的 Viewer UI，可以用 **AndroidPdfViewer（基于 PdfiumAndroid）**，带手势、缩放、双击等。([GitHub][5])

专业落地建议：

* 真正做“体验好”的 PDF：不要只做“整页 bitmap 缓存”，要做**分级缓存 + 预渲染**（下一页/上一页）+ 缩放时的清晰度策略。
* 如果你需要文字层（选择/搜索/复制），要确认你选的 PDFium binding 是否暴露了 text extraction / word rects 等 API（否则你要自己补）。

### C) 你要“注释/表单/签名/编辑/企业级全文索引”等：直接上商业 SDK（最省人力、最稳）

如果你的产品目标包含：

* 多类型注释编辑、表单填写、签名、文档编辑、稳定的全文索引搜索、协作等

那最推荐是商业 PDF SDK（这就是“不要 MVP”的现实选择：买成熟能力，省工程年）：

* **Nutrient / PSPDFKit**：强调 viewing、annotate、sign、forms 等能力。([GitHub][6])
* **Apryse（原 PDFTron）**：强调移动端完整文档能力（view/annotate/text editing 等“100+ features”定位）。([Apryse][7])

### D) 你能开源或愿意购买商业授权：MuPDF 是性能很强但许可证敏感

MuPDF 开源是 **AGPL**，闭源商业需要购买授权。([MuPDF][8])

---

## 如果你想 EPUB + PDF 统一在一个“阅读引擎体系”：Readium 也能承载 PDF（通过适配器）

Readium 的 PDF 支持不是自己写引擎，而是**依赖第三方 PDF 引擎**，并提供适配器（pdfium / pspdfkit），也支持你自定义适配器，把别的 PDF 引擎接进 `PdfNavigatorFragment`。([Readium][9])

这对“专业工程”很有价值，因为：

* 你可以把 EPUB 与 PDF 的打开流程、书架/Publication 模型、定位事件、阅读进度回传等尽量统一；
* PDF 引擎可替换（pdfium → 商业 SDK）而不重写上层。

---

## 我给你的“最推荐组合”结论（按常见目标拆两套）

### 1) 目标：覆盖广（minSdk 21+）、以阅读为主、开源优先

* TXT：原生排版分页（StaticLayout/Compose TextLayout）+ 自研分页缓存
* EPUB：Readium Kotlin Toolkit（Streamer + Navigator）([Readium][1])
* PDF：PDFium（PdfiumAndroidKt 自研 UI 或 AndroidPdfViewer 作为基线）([GitHub][4])
  （如果你也用 Readium：优先走 Readium 的 `pdfium` adapter，方便统一进度/事件）([Readium][9])

### 2) 目标：功能做满（注释/表单/签名/稳定全文索引），愿意付费

* TXT：同上（自研最稳）
* EPUB：Readium Kotlin Toolkit（你省掉标准兼容坑）([Readium][1])
* PDF：Nutrient/PSPDFKit 或 Apryse（把工程成本从“年”压到“周/月”）([Nutrient][10])
  （并可通过 Readium 的 `pspdfkit` adapter 做统一）([Readium][9])
