下面给你一套**“可长期维护、可扩展、排版质量可控”**的阅读排版实现方案（按你给的模块化架构落地），重点放在 **排版/分页/定位/交互（选中、高亮、注释）** 的工程实现，而不是 MVP。

---

## 0. 先把“阅读排版”分成两类问题

1. **Reflow（可重排版）**：TXT、EPUB Reflow（绝大多数 EPUB 都是这个）

> 关键难点：断行、连字/字形塑形、行距段距、缩进、两端对齐、分页、定位映射、选区/高亮路径、设置变化后的重分页与缓存。

2. **Fixed-layout（固定页面）**：PDF（以及少量 EPUB FXL）

> 关键难点：渲染性能（缩放/平移/快滚）、瓦片/缓存、目录/搜索/选区（可选）、页码与进度。

你的目录结构已经把“引擎可插拔”拆清楚了。接下来最关键的是：**Reflow 的排版内核要做成“可复用的工程组件”，而不是散在 TXT/EPUB 引擎里。**

---

## 1) 我最推荐的总体落地路线（非 MVP）

### A. EPUB：直接用成熟 Web 排版（Readium / WebView Navigator），别自研 CSS 引擎

原因很现实：EPUB = HTML/CSS/字体/RTL/脚注/图片/复杂样式。你自研“像样”的 CSS 排版引擎，成本极高且坑非常深。

**推荐**：在 `engines/epub` 内集成 Readium Kotlin Toolkit（Navigator），把它当作 EPUB 的排版/定位/高亮能力提供者。Readium Toolkit 官方列出其对 EPUB2/3、分页、RTL、搜索、高亮（Decoration API）等能力的覆盖情况，且对 EPUB reflow/fxl 都有分页能力支持。 ([readium.org][1])
同时 Readium 的 Navigator 架构文档也明确：多数阅读应用在显示 EPUB 资源时依赖一个或多个 **WebView**（native app）来渲染内容并可注入 CSS/JS。 ([readium.org][2])

> 结论：**EPUB 排版 = Web 排版**（WebView + CSS/JS 注入）是“最稳”的工业方案。

---

### B. TXT：自研 Native Reflow 排版引擎（StaticLayout + 分页器 + 选区映射）

TXT 没有 CSS 复杂性，自研排版能做到“体验可控、性能可控、可深度定制”。

你在 `engines/engine-common` 放一个 `reflow-typesetter` 子包即可，TXT 引擎只负责把 TXT 解析成 block 流。

核心依赖：Android 的 `StaticLayout.Builder`（断行/连字/双向文本/两端对齐/自动断词/缩进等能力都在系统文字引擎里）。它支持：

* `setBreakStrategy`：选择断行策略（SIMPLE/BALANCED/HIGH_QUALITY）。 ([Android Developers][3])
* `setHyphenationFrequency`：自动断词（hyphenation）。并且官方提示：**与更高质量断行策略配合能提升版面结构，但会有性能成本**。 ([Android Developers][3])
* `setJustificationMode`：段落两端对齐（inter-word / inter-character）。 ([Android Developers][3])
* `setIncludePad`：是否包含额外字体 padding，文档明确它用于避免某些语言（如 Arabic/Kannada）被裁切。 ([Android Developers][3])

> 结论：**把“字形塑形/断行/对齐/断词”交给系统文字引擎，你只做分页、缓存、定位映射和交互即可**，排版质量会稳定很多。

---

### C. PDF：固定页渲染用 PdfRenderer（必要时加瓦片/缓存；搜索/选区按能力做）

Android 官方 `PdfRenderer` 的 API 语义是：打开 renderer -> openPage -> render -> closePage；并且说明“页面逐个渲染，同一时间只能打开一个 page”。 ([Android Developers][4])
文档也提到它在较新系统（文档写的是 Android V 起）扩展了选择、搜索、快滚、注释等高级能力。 ([Android Developers][4])

> 结论：PDF 先把“渲染 + 平滑滚动/翻页 + 缓存”做好；“文本抽取/搜索/选区”做成 capability 可插拔（你目录里已经这么规划了）。

---

## 2) Reflow（TXT）排版内核：推荐的工程化实现

把这套东西放在 `engines/engine-common/reflow/`，TXT/（自研）EPUB 都能复用。

### 2.1 数据结构：把内容变成“块流 Block Stream”

用 block 来对齐排版与定位（比直接按字符串分页可控得多）：

```kotlin
sealed interface Block {
  val id: BlockId              // 稳定：用于 locator 映射
  val styleKey: StyleKey       // 排版风格：标题/正文/引用/代码
}

data class TextBlock(
  override val id: BlockId,
  val text: CharSequence,      // 建议用 Spanned/AnnotatedString 类似结构承载 spans
  val spans: List<SpanRange>,  // 高亮/链接/注释/搜索命中 等
  override val styleKey: StyleKey
) : Block

data class ImageBlock(
  override val id: BlockId,
  val imageRef: ImageRef,
  val aspectRatio: Float,
  override val styleKey: StyleKey
) : Block
```

**关键点**：`BlockId` 必须可稳定重建（来自章节+段落序号、或正文 hash），这样 locator 才不会因为解析实现变化而漂移。

---

### 2.2 版面参数：RenderConfig 必须可 hash（用于缓存/持久化）

分页结果依赖这些参数：

* viewport：内容宽高（扣除 padding/margin）
* 字体：fontFamily、fontSize、weight
* 行距/段距/首行缩进
* 对齐：left/justified
* 断行策略、断词策略
* 语言/方向（RTL/LTR）

**做法**：把它做成 `RenderConfigHash`（如 xxHash64），作为分页缓存 key。

---

### 2.3 Layout 阶段：TextBlock -> StaticLayout

对每个 TextBlock 生成一个 `ParagraphLayout`（可缓存）：

* 统一用 `StaticLayout.Builder.obtain(...)`
* 设置：

  * `setBreakStrategy(...)`（默认 BALANCED；低端机可以 SIMPLE）
  * `setHyphenationFrequency(...)`（默认 NORMAL；需要更美观用 FULL 但注意性能） ([Android Developers][3])
  * `setJustificationMode(...)`（需要两端对齐时开启） ([Android Developers][3])
  * `setIncludePad(...)`：建议做成“安全模式开关”。文档说明 includePad 用于避免某些语言裁切；你可以默认 `true`（安全），提供“紧凑排版”开关让用户关掉。 ([Android Developers][3])

你最终缓存的是：

```kotlin
data class ParagraphLayout(
  val blockId: BlockId,
  val staticLayout: StaticLayout,
  val baselineInfo: BaselineInfo,
  val lineCount: Int,
  val heightPx: Int,
  val charCount: Int
)
```

**缓存策略（必须做，非 MVP）**：

* `LRU<ParagraphCacheKey, ParagraphLayout>`
* key = `blockId + widthPx + styleKey + renderConfigHashRelevantParts`
* cache 失效：字体/字号/宽度变化时自然 miss；同一配置下翻页非常省 CPU。

---

### 2.4 Pagination 阶段：把“行”切成页（而不是切字符）

分页器只做一件事：**把 Layout 的可视高度按页高切片**。

输出 `PagePlan`：

```kotlin
data class PagePlan(
  val pageIndex: Int,
  val slices: List<Slice>,         // 本页包含哪些 block 的哪些行/区域
  val locatorStart: Locator,
  val locatorEnd: Locator
)

sealed interface Slice { val blockId: BlockId }

data class TextSlice(
  override val blockId: BlockId,
  val lineStart: Int,
  val lineEndExclusive: Int,
  val charStart: Int,
  val charEnd: Int,
  val topPxInBlock: Int            // 画的时候 translate/clip
) : Slice
```

**分页算法建议**（稳定 + 易测试）：

* 逐 block 累积高度
* 若剩余空间不够：

  * TextBlock：按行边界切（`lineStart/lineEndExclusive`）
  * ImageBlock：优先整块换页；若单张图片超过页高，按策略缩放或做“图片页”（固定页模式）
* 处理 widows/orphans（非 MVP 应该要）：例如段落最后一行不允许单独留在新页，至少 2 行；这类规则可以在切行时做回退。

---

### 2.5 Locator 映射：页 <-> 文本位置必须可逆（至少近似可逆）

这决定：

* 进度保存
* 搜索跳转
* 书签/注释定位
* 旋转/字体变化后的“尽量回到原位置”

推荐 locator 采用：

* TXT：`(blockId, charOffsetInBlock)`
* EPUB：保持 CFI（Readium 已有 Locator 概念和 Selection/Decoration API，可直接对接） ([readium.org][5])

分页时每个 page 保存 `locatorStart/locatorEnd`，并在 runtime 维护一个**有序索引**：

* `List<PageAnchor(locatorStart, pageIndex)>`
* locator -> pageIndex：二分查找
* pageIndex -> locator：直接取 page 的 start

---

### 2.6 Render 阶段：绘制必须做到“可复用、可缓存、不卡翻页”

你有两个成熟路线：

**路线 1（推荐，最稳）**：自定义 Android View 渲染 StaticLayout，Compose 用 `AndroidView` 包一层

* 优点：StaticLayout.draw 直接走 Android Canvas，性能路径稳定
* 你还能更容易做手势/选区（同一坐标系）

**路线 2**：Compose Canvas + `drawIntoCanvas { it.nativeCanvas }` 调 `StaticLayout.draw()`

* 优点：UI 更 Compose 原生
* 注意：要用 `drawWithCache` 或自己的 page cache，避免频繁重算

**非 MVP 的缓存建议**：

* “页渲染缓存”与“段落布局缓存”分离：

  * 段落布局缓存：CPU 省（重分页时也能复用部分）
  * 页渲染缓存：GPU/绘制省（翻页更丝滑）
* 页渲染缓存可以用：

  * `RenderNode`（记录绘制命令，重放快）
  * 或 bitmap（但内存压力大，得做严格 LRU + 分辨率策略）

---

### 2.7 选区/高亮/注释：必须基于 Layout 做“范围 -> 路径/矩形集合”

对 TextBlock 的交互统一走：

* `hitTest(x,y) -> (blockId, charOffset)`
* `range(blockId, start..end) -> path/rects`
* `drawDecorations()` 在文字下层绘制

这套在 TXT 自研引擎里用 `StaticLayout`/`Layout` 的行信息做就行；
在 EPUB（Readium）里，直接用 Readium 的 Selection/Decoration 能力去画（它的 API 文档里有 Selection/Decoration 模型）。 ([readium.org][5])

## 3) core/reader/runtime：排版会话与缓存编排（你架构里的“关键层”）

把“排版重活”从 feature/reader 里剥离出来，做到：

* **可取消**：字体/屏幕旋转/设置变化时，旧分页任务必须 cancel（否则卡顿/浪费）
* **可增量**：打开书时只算“当前位置附近 N 页”，后台再补齐后续
* **可持久化**：把 page anchors（locatorStart 列表）落库，下一次打开能秒定位（配置 hash 不变时）

> 这块属于你 `core/reader/runtime` 的“Session + Policy + CachePolicy + PrefetchPolicy”。

---

## 4) EPUB 引擎（Readium）怎么和你的架构对接（推荐做法）

在 `engines/epub`：

* 你实现 `ReaderEngine`（对外入口）
* 内部用 Readium：

  * 打开 publication
  * 创建 navigator（视觉渲染 + 分页）
  * 通过 Locator/Selection/Decoration 提供定位与高亮
* 你的 `core/reader/runtime` 只看你统一的 `ReaderSession`/capabilities

Readium 官方文档列出的功能覆盖（EPUB/PDF 的分页、RTL、搜索、Decoration 高亮等）可以直接转成 `DocumentCapabilities`： ([readium.org][1])

同时，Readium 的 Navigator 架构建议在 native app 里用 WebView/iframe 方案显示资源，并允许注入 CSS/JS，这正好对应你要做“用户字体/主题/行距/翻页方式”的偏好注入。 ([readium.org][2])

---

## 5) PDF 引擎（PdfRenderer）落地要点（不做成 MVP 的关键）

`engines/pdf`：

* Rendering：

  * PdfRenderer：按页 open -> render -> close；同一时间只能开一个 page。 ([Android Developers][4])
* 性能：

  * 强制瓦片化（tile），至少做到：

    * 当前页高清 tile
    * 邻页低清预渲染
    * LRU 释放 bitmap/tile
* 能力分级：

  * Android 新系统（文档写 Android V 起）PdfRenderer 有更高级能力（选择/搜索/注释等）。 ([Android Developers][4])
  * 旧系统：你就按 capabilities 关掉“文本搜索/选中”或走可插拔 text-extraction 模块（你目录里已经预留 `text/` `search/`）。

---

## 6) 最终你会得到的“排版工程能力”清单（符合非 MVP）

1. **排版质量可控**：断行/断词/两端对齐/缩进/段距，全在 StaticLayout/Readium 体系里
2. **翻页性能可控**：段落布局缓存 + 页渲染缓存 + 预取策略
3. **定位稳定**：Locator 基于 blockId/offset（TXT）或 CFI/Readium Locator（EPUB）
4. **设置变化可恢复**：configHash + anchors 让“改字号/旋转屏幕”尽量回到原位置
5. **能力开关清晰**：capabilities 决定 UI 展示与功能启用，避免“PDF 没法选中却硬展示按钮”

## 7) 你现在就可以直接照这个切模块的落地清单

* `engines/engine-common/reflow/`

  * `BlockParser`（TXT -> blocks）
  * `StyleResolver`
  * `ParagraphLayouter(StaticLayout)`
  * `Paginator`
  * `LocatorMapper`
  * `LayoutCache` / `PageRenderCache`
* `engines/txt/`

  * 只做：解析、章节切分、提供 blocks、接入 reflow 内核
* `engines/epub/`

  * 只做：Readium 适配器（open/navigate/selection/decorations/preferences）
* `engines/pdf/`

  * 只做：PdfRenderer 渲染 + tile/cache + outline（可选）+ text/search（可选）
* `core/reader/runtime/`

  * `SessionManager`
  * `PrefetchPolicy`（前后 N 页）
  * `CachePolicy`（内存 LRU + DB anchors）
  * `RenderPolicy`（分页模式/滚动模式统一策略）
