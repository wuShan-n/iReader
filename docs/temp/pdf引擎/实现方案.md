下面给你一套**在你当前契约（ReaderDocument/Session/Controller + RenderContent.Tiles/BitmapPage + Providers）下最推荐的 PDF 引擎实现方案**。核心思路是：**PDF 永远走 fixed-layout；渲染统一走 Tiles（缩放不卡、内存稳）；“高级能力”用后端分层（平台 PdfRenderer 新能力 / 旧系统用 Pdfium 可选补齐）**。

---

## 0) 先定结论：PDF 引擎的最佳形态

### 渲染（必须）

* **RenderContent.Tiles** 做默认输出（即使 fitWidth 的 1.0 倍也可以 tiles=单块），因为 PDF 天然需要缩放与大页渲染；Tiles 让你能：

  * 先 DRAFT 快速出图，再 FINAL 补清晰
  * 只渲染可见区域（放大时关键）
  * 更好做邻页预取与缓存淘汰

### 能力（按后端决定，可降级）

* `fixedLayout=true, reflowable=false`
* `outline/search/textExtraction/links`：**强烈建议“有就开、没有就关”**（你能力字段已经支持）
* `annotations=true`：固定版式标注先做 **FixedRects**（矩形高亮/下划线/笔记钉），后续再加“基于文本选择”的 rects

## 1) 后端分层：一个 PDF Engine，内部两套 backend（推荐）

你现在 `EngineRegistry` 一个 format 只能绑定一个 engine，非常好：**不要注册多个 PDF 引擎**，而是在 `PdfEngine` 内部做后端选择。

### 1.1 平台优先：android.graphics.pdf.PdfRenderer（推荐作为主渲染后端）

* `PdfRenderer` 是官方 API（API 21+），并且在 **Android V（API 35）** 增强后，文档明确提到它支持 **selection/search/annotations/fast scrolling** 等高级能力。([Android Developers][1])
* 同时它不引入 `.so`，天然避开 “16KB page size 对 native libs 的兼容性要求” 风险（见 1.3）。

> 注意：`PdfRenderer` 文档也强调 **同一时间只能打开一个 Page**，渲染要严格 openPage/render/closePage。([Android Developers][1])
> 这直接决定你 TileProvider 里要做**串行化 page 打开**。

### 1.2 旧系统补齐能力：Pdfium（可选但我最推荐）

如果你产品必须做：**目录/链接热区/文本抽取/搜索/加密 PDF（密码）**，那在 API < 35 上，单靠旧版 PdfRenderer 很难补齐。

* PdfiumAndroid 提供：

  * bookmarks（目录树）
  * links（每页链接 + bounds + 坐标映射）([GitHub][2])
  * 渲染到 bitmap
* 基于 Pdfium 的生态（AndroidPdfViewerV2 等）支持 **加密 PDF 的 password 打开**（密码校验在底层 Pdfium）。([DeepWiki][3])

**推荐策略**

* `backend = if (SDK >= 35) PlatformPdfRendererBackend else PdfiumBackend`
* 或者：即便 SDK>=35，你也可以先用 Pdfium 做“能力一致性”，等你完全验证平台新能力稳定后再切回。

### 1.3 重要：Google Play 16KB page size 要求（只影响你带 native libs 的方案）

从 **2025-11-01** 起，面向 Android 15+（64-bit）的新上架/更新需要支持 **16KB memory page size**；如果你（或你间接依赖的 SDK）带 `.so`，需要确保对齐与构建链满足要求。([Android Developers][4])

因此：

* **纯 PdfRenderer 路线最省心**
* 如果你引入 Pdfium/MuPDF（有 native），必须把 16KB 对齐当成 CI 检查项（但这不是不可做，只是工程项要补齐）

```text

    ├── open/

    │   └── PdfDocumentImpl.kt        # 实现 ReaderDocument
    ├── backend/
    │   ├── PdfBackend.kt             # 抽象：pageCount/metadata/renderTile/text/search/outline/links
    │   ├── PlatformPdfBackend.kt     # android.graphics.pdf.PdfRenderer
    │   └── PdfiumBackend.kt          # PdfiumAndroid（可选）
    ├── session/
    │   └── PdfSessionImpl.kt         # 实现 ReaderSession
    ├── render/
    │   ├── PdfController.kt          # 实现 ReaderController
    │   ├── PdfTileProvider.kt        # 实现 TileProvider（串行 openPage）
    │   └── FitTransform.kt           # fitWidth/fitPage/zoom/rotation -> Matrix/viewport 映射

    │   ├── PdfOutlineProvider.kt
    │   ├── PdfSearchProvider.kt

```

## 3) 关键实现点（按你的契约逐个落地）

### 3.1 打开：必须拿到“可随机访问”的 ParcelFileDescriptor

* `PdfRenderer` 需要 seekable FD（官方 sample 也强调需要 seekable file descriptor）。([Android Git Repositories][5])
* `DocumentSource.openFileDescriptor()` 可能返回 null（SAF 某些 provider），所以 `PdfOpener` 要有兜底：

  1. 先试 `openFileDescriptor("r")`
  2. 失败就把 `openInputStream()` 拷贝到 `cacheDir/pdf/xxx.pdf` 再 reopen FD

### 3.2 capabilities 的计算：永远以“实际可用”为准

建议这样填：

* **最低保证**（只做渲染）：

  * fixedLayout=true
  * outline/search/textExtraction/links=false
  * annotations=true（至少支持 FixedRects：用户框选区域高亮/笔记）
* **如果 backend 支持**（Pdfium / API35+ PdfRenderer）：

  * outline=true（目录树）
  * links=true（热区跳页/外链）
  * textExtraction=true（复制/分享、选中文本）
  * search=true（全文搜索）

> `PdfRenderer` 在 API35 文档中写到支持 searching/selection/annotations。([Android Developers][1])
> Jetpack `androidx.pdf` 也强调 viewer 具备 search/zoom/annotations/stylus，并支持 password 文档。([Android Developers][6])
> 但 `androidx.pdf` 更偏 UI 组件（Fragment/Compose），你当前 engine 架构更适合做“底层渲染+provider”。

### 3.3 ReaderController：PDF 只实现 fixed-page 导航模型

* Locator：`scheme=pdf.page, value=pageIndex(0-based)`
* `next/prev`：pageIndex ±1
* `goToProgress`：`pageIndex = floor(percent * pageCount)`
* `RenderState.progression`：

  * percent = pageIndex / (pageCount-1)
  * label = “{pageIndex+1}/{pageCount}”

### 3.4 render()：强烈推荐一律返回 Tiles

返回：

```kotlin
RenderPage(
  id = PageId("pdf:$docId:$pageIndex:${configKey}"),
  locator = Locator(LocatorSchemes.PDF_PAGE, pageIndex.toString()),
  content = RenderContent.Tiles(pageWidthPx, pageHeightPx, tileProvider),
  links = linkProvider?.linksFor(pageIndex) ?: emptyList(),
  decorations = annotationsProvider?.decorationsFor(pageIndex) ?: emptyList(),
  metrics = ...
)
```

#### TileProvider 的关键技巧：用 Matrix 平移让“只渲染局部”成立

对 `PdfRenderer.Page.render(bitmap, destClip, matrix, mode)` 的常见做法是：

* 创建 **tile 大小** bitmap
* matrix 设为：

  * scale = request.scale * zoom * densityFactor
  * translate = `(-leftPx, -topPx)`（把目标区域移动到 tile 原点）
* destClip = `Rect(0,0,width,height)`
  这样你等价于“只画需要的区域”，而不是先整页再裁剪（整页会爆内存）。

另外，PdfRenderer 文档明确：一次只能开一个 Page。([Android Developers][1])
所以 `PdfTileProvider.renderTile()` 内部要：

* 用 `Mutex` 或单线程队列串行：

  * `openPage(pageIndex)` -> `render` -> `close`

### 3.5 缓存策略（配合 RenderPolicy）

你已有 `RenderPolicy(quality, allowCache, prefetchNeighbors)`，非常适合 PDF：

* **TileCache（LRU）**：

  * key：`(pageIndex, scaleBucket, rect, rotation, quality)`
  * scaleBucket：把 scale 量化（比如 1.00、1.25、1.50…）减少缓存碎片
* **BitmapPool**：

  * 复用 tile bitmap，避免频繁 GC
* **prefetchNeighbors**：

  * 预取邻页的“低清 DRAFT”（或只预取首屏 tile）
  * 用户停住后再补当前页 FINAL

> 多分辨率/瓦片策略是业界做 PDF 流畅缩放的常见路线（先低清再补高请、tile grid）。([Nutrient][7])

## 4) Outline / Links / Search / Text：怎么在你当前模型里表达（不强改 model）

你现在 PDF 的 Locator 只定义到 page，这没问题：**把更细粒度的信息塞进 extras**（或把 rects 直接给 UI）。

### 4.1 Links（最推荐先做）

PdfiumAndroid 支持 `getPageLinks()` 并给出 link bounds 和坐标映射方法。([GitHub][2])
实现上：

* 把 bounds 转成 `NormalizedRect`（0..1 相对页宽高）
* `DocumentLink(bounds=rects, target=Internal(pageLocator) / External(url))`
* 这样 UI 点击热区就能跳转

### 4.2 Outline（目录树）

Pdfium 能拿 bookmarks tree。([GitHub][2])
映射为 `OutlineNode(title, locator=pdf.page)`。

### 4.3 Search + Text extraction

两种推荐路径：

**A) API35+ 走 PdfRenderer 新能力**
PdfRenderer 文档已写明从 Android V 起支持 searching/selection/annotations。([Android Developers][1])
你可以在 `PlatformPdfBackend` 里实现：

* `search(query)` -> 命中页 + rects
* `getTextAround(pageLocator)` -> 页文本片段

**B) 旧系统用 Pdfium**

* `searchProvider.search()`：按页扫描（先粗略按页文本匹配，再拿命中区域）
* `SearchHit.range`：

  * start/end 都用同一个 page locator
  * 把命中的 `rects`（normalized）塞进 `range.extras["rectsJson"]=...`，UI 直接高亮
* `TextProvider.getTextAround(locator)`：返回该页 text 的截断片段即可（你的契约允许）

### 4.4 Password（加密 PDF）

* Jetpack `androidx.pdf` release notes 提到 `PdfViewerFragment` 对 password 文档会弹框。([Android Developers][8])
* Pdfium 生态支持传入 password 打开加密 PDF。([DeepWiki][3])

如果你走 Pdfium backend：

* 在 `PdfOpener` 捕获 `PdfPasswordException`（或类似）映射成 `ReaderError.InvalidPassword`

## 5) 关于 MuPDF：不推荐默认（除非你能接受许可证）

MuPDF 官方文档明确：开源版本是 **AGPL**，闭源应用一般需要商业授权。([MuPDF][9])
所以在你的工程里，MuPDF 更像是“商业版/企业版后端”，不建议作为默认最小依赖。

## 6) 最推荐的落地顺序（能最快跑通 + 未来不返工）

1. **PdfRenderer + Tiles + 缓存 + 翻页/缩放/旋转**（capabilities：search/text/outline/links 先 false）
2. **补 Links（Pdfium getPageLinks）+ Outline（bookmarks）**（capabilities 打开 links/outline）([GitHub][2])
3. **补 Text/Search（优先用 API35 PdfRenderer 新能力；旧系统用 Pdfium）**([Android Developers][1])
4. **标注：先 FixedRects（框选）→ 再“基于文本选择”的 rects（如果 text/selection 可用）**
