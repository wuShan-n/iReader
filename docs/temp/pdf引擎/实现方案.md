## 一句话推荐：做成“**双后端 + 同一引擎契约**”的 PDF Engine

**最推荐的专业实现**是：

* **后端 A（优先）**：在系统支持的设备上使用 **Jetpack `androidx.pdf` 的 processing 层（pdf-document-service / PdfDocument 等）**

  * 这是 Google 正在推进的官方 PDF 方案，提供“查看 + 搜索/选择 + 注释 + 手写/触控”等完整交互能力方向。官方介绍明确列出：打开查看、搜索选择、缩放滚动、保存注释、手写笔交互等能力。([Android Developers][1])
  * 它的 `PdfViewerFragment` 也揭示了官方推荐的渲染策略：**两段式渲染（layout + render）、渐进式加载、只渲染可见页、不可见页释放 bitmap**，这是你要做“专业性能”的关键参考。([Android Git Repositories][2])
  * 但它有**平台/扩展版本要求**：`PdfViewerFragment` 标注了 `@RequiresExtension(S, 13)`，并且相关服务对 API/扩展有依赖。([Android Git Repositories][2])
  * `pdf-document-service` 的定位就是：**解析/处理 PDF，并且“sandboxing”底层 `PdfRenderer`**（安全性更好）。([Maven Repository][3])

* **后端 B（兜底兼容）**：对低版本/不满足扩展要求的设备，用 **PDFium（推荐选择活跃维护的 Android binding）**

  * PDFium 体系能做到你真正需要的：渲染、元信息、目录（bookmarks/outline）、链接等（很多 binding 已经封装）。例如 PdfiumAndroid 文档就明确覆盖了 metadata、toc/bookmarks、hyperlinks 等信息抽取。([DeepWiki][4])
  * 这样你就不会被平台版本锁死，也不用把业务/feature 和底层绑定在一起。

> 这套“**双后端**”的意义：
>
> 1. 你能用到官方的安全与新能力（在支持设备上）；
> 2. 你又能覆盖广泛机型（用 PDFium 兜底）；
> 3. 你引擎层代码只写一套，后端可替换，完全符合你“feature 不依赖 engines”的架构目标。

## 你现有 Reader API 下，PDF Engine 的“专业能力定义”

基于你 `DocumentCapabilities`，我建议 PDF 引擎至少做到：

### 1) Fixed layout 渲染（必须）

* `capabilities.fixedLayout = true`
* `RenderConfig.FixedPage` 支持：

  * FitMode：`FIT_WIDTH / FIT_HEIGHT / FIT_PAGE / FREE`
  * zoom：连续缩放
  * rotation：0/90/180/270（或任意角度，建议先支持 90 的倍数）

### 2) 渲染输出形态：优先 Tiles（专业必备）

你 API 已经支持 `RenderContent.Tiles` + `TileProvider`，这非常专业：

* PDF 页在高倍缩放下，如果直接渲染整页 bitmap，**内存会瞬间爆**。
* Tiles 是最正确的做法（地图式渲染）。

**推荐策略：**

* 默认返回 `Tiles`（尤其是 zoom>1 或大页）
* 小页/低 zoom 可返回 `BitmapPage`（优化首屏速度/简单 UI）

### 3) 目录（Outline / Bookmarks）

* 有就提供 `OutlineProvider`
* 映射到你的 `OutlineNode(title, locator)`，locator 用 `pdf.page`

### 4) 链接热区（Links）

* 每页解析 URI link / internal goto
* 映射 `DocumentLink(bounds=NormalizedRect…)`，bounds 用页内 0..1 归一化坐标

### 5) 文本抽取 + 搜索（强烈建议做）

这是你产品从“能看”到“专业阅读器”的分水岭：

* `TextProvider`：至少支持“整页文本/选区文本”
* `SearchProvider`：支持跨页搜索、返回 excerpt、定位到页

> 官方 Jetpack PDF viewer 也强调“Search and select content”。([Android Developers][1])
> 你自己实现引擎时，不做搜索基本就会被用户认为“不专业”。

### 6) 标注（Annotation）

最推荐的落地方式（阅读器行业通用）：

* **标注不直接写回 PDF 文件**（那是“编辑器”级复杂度），而是：

  * 存到你 `core/database` / `AnnotationRepo`
  * 渲染时叠加为 `Decoration.Fixed`（基于 page + rects）
* 这样：

  * 同一个 PDF 文件在不同设备/不同渲染后端仍然可用
  * 无需处理 PDF 内部 annotation 写入/增量保存/线性化等复杂坑

## 关键难点：PDF 后端与 Reader API 的“正确映射”

### Locator 设计（你现有方案很好）

* 页面定位：`LocatorSchemes.PDF_PAGE`，value = 0-based pageIndex
* 进度：

  * `Progression.percent = pageIndex / (pageCount - 1)`
  * label：`"${pageIndex+1}/${pageCount}"`

### 坐标体系统一（非常关键）

你使用 `NormalizedRect(0..1)` 很正确。建议定义一套统一规范：

* PDF 后端通常给你的是“PDF 坐标（points）”，原点可能在左下角
* UI 需要的是“屏幕坐标（px）”，原点左上
* 你存的是“归一化到页面内容坐标（0..1）”

**建议：**

* 在 pdf/internal 里做一个 `PdfCoordinateMapper`：

  * 输入：pageWidthPts/pageHeightPts + rotation + cropBox/mediaBox + zoom + fitMode + viewport
  * 输出：

    * pdfPts -> normalizedRect
    * normalizedRect -> screenPxRect（用于绘制 overlay）
* 所有 links / text selection bounds / annotation bounds 都走这一套，避免各处各算各的导致漂移。

## 打开文档：为什么必须做“seekable”与缓存拷贝

无论你用 `PdfRenderer` 还是很多 PDF 库，几乎都要求底层输入是**可随机访问**的。

Android 官方 PdfRenderer 的 sample 也明确说：

* 需要一个 **seekable 的 ParcelFileDescriptor**
* 资产里的 PDF 默认压缩，`PdfRenderer` 不能直接打开，sample 通过 **复制到 cacheDir** 解决。([Android Git Repositories][5])

所以你引擎必须有一层 `PdfSourceResolver`：

**输入**：`DocumentSource (Uri)`
**输出**：

* 优先：`ParcelFileDescriptor`（已 seekable）
* 否则：拷贝到 app cache（文件名用 hash），再用 `ParcelFileDescriptor` 打开

> 这一层做得好，你后面换 PDF 后端几乎不动。

## 渲染：专业级必须有“渲染管线（pipeline）”

别把渲染写成“来一个请求渲一个”的同步函数；专业阅读器一定要有调度：

### 1) 请求模型

定义渲染请求的 key（决定缓存命中）：

* docId
* pageIndex
* rotation
* zoom（建议做 bucket，例如 1.0/1.25/1.5…，避免 cache 被连续 zoom 打爆）
* fitMode
* viewportPx（决定最终输出尺寸）
* quality（DRAFT/FINAL）

### 2) 优先级队列 + 取消

场景：用户快速翻页/滚动时

* 旧请求必须被取消（否则 CPU 全浪费）
* 当前页 > 邻页预取 > 远页

官方 pdf 相关代码里也出现过“用 PriorityDispatcher 把请求排队并顺序执行”的思路（这是正确方向）。([Android Git Repositories][6])

### 3) 两段式渲染（强烈推荐）

借鉴 `PdfViewerFragment` 的描述：

* Layout pass：先拿到 page size，算出在 viewport 里的显示尺寸
* Render pass：只对可见页做 bitmap/tile 渲染，不可见释放 bitmap。([Android Git Repositories][2])

你虽然是单页阅读为主，也建议保留这个思想：

* `render()` 先快速返回 DRAFT（低分辨率）提升体感
* 后台再补 FINAL（高分辨率），通过 `events` 发 Rendered 事件刷新

### 4) PdfRenderer 的并发限制

Android 的 `PdfRenderer` 文档明确提到：页是“一页一页渲”，**同一时刻只能打开一页**。([EManual][7])
所以如果你用 framework 后端，渲染 dispatcher 建议：

* `limitedParallelism(1)` 或单线程 executor
* 或者 `Mutex` 保护“openPage/render/closePage”临界区

PDFium 后端并发可以更激进一点，但也建议受控（2 线程足够）。

## Tiles：怎么把你的 TileProvider 做到“真好用”

你的 `TileRequest` 有 `(left, top, width, height, scale, quality)`，很适合 PDF：

**推荐实现细节：**

* Tile 坐标定义为**渲染后的 page 像素坐标**（已包含 zoom/rotation 影响）
* TileProvider 内部：

  * 计算 tile 对应的 PDF 坐标区域
  * 调后端 “renderRegion(pageIndex, rect, scale)” 输出 bitmap
* TileCache：

  * key：pageIndex + zoomBucket + rotation + tileRect + quality
  * memory LRU：按 bitmap bytes 计

**阈值建议：**

* 当“整页 bitmap 预计占用 > 16~24MB”时强制 Tiles

  * 例如：`w*h*4`（ARGB_8888）
* 否则直接整页 bitmap（更简单，且更少 tile 拼接开销）

## Outline / Links：把 PDF 的“信息抽取”当成一等公民

### OutlineProvider

* 打开文档时一次性解析（或 lazy 第一次请求时解析）
* 缓存为树结构（通常不会变）
* locator：pdf.page

### Links

最好在渲染页时一起返回：

* `RenderPage.links`：由后端拿到当前页的 link annotations（URI/GoTo）并转换
* bounds：转换成 `NormalizedRect` 存在模型里，UI 就可以直接做点击热区

PDFium bindings 通常能提供 hyperlinks/bookmarks 等信息抽取接口能力（至少在一些实现里已有封装）。([DeepWiki][4])

## 搜索：专业方案一定要“分层”

PDF 搜索的坑在于：

* 有的 PDF 是纯图片（扫描件），没有文本
* 有的文本抽取很慢
* 你还希望结果能“定位 + 高亮”

我建议分三层：

### 层 0：能力探测

打开文档后，快速判断：

* 是否存在文本（抽几页试探）
* 是否支持后端原生 search

决定 `capabilities.search` / `capabilities.textExtraction`。

### 层 1：即时搜索（用户输入就要出结果）

* 用后端原生 search（如果 Jetpack/PDFium 提供）
* Flow 按页流式 emit `SearchHit`：

  * range：start/end = pdf.page（同页就一样）
  * excerpt：从该页文本切片
  * sectionTitle：可用 outline 最近标题（可选）

### 层 2：索引加速（大型文档）

当文档较大时（比如 > 500 页 或 > 50MB）：

* 交给 WorkManager 后台做“页文本抽取 + 索引”
* 用 Room FTS 或 AppSearch 进行全文索引
* 搜索时先用索引快速筛页，再对命中页做精确定位（生成高亮 rect）

> 这就是“能用”与“专业”的差距：
> 没索引的大 PDF 搜索会卡到用户崩溃。

## 标注：推荐“外置存储 + 叠加渲染”，并与 PDF 引擎解耦

### 你现有模型非常适合 PDF 标注

`AnnotationAnchor.FixedRects(page, rects)` 就是 PDF highlight 的正确数据结构。

**推荐实现：**

* `PdfAnnotationProvider` 不写 PDF 文件，只负责：

  * CRUD -> `core/data` 的 AnnotationRepo
  * `decorationsFor(page=locator)` -> 转成 `Decoration.Fixed` 返回给 UI
* UI 渲染：

  * 用 `NormalizedRect -> screenRect` 映射在 page 上画高亮/下划线/笔记图标

如果你未来要支持“写回 PDF”（真正保存进 PDF 文件）：

* 可以做成独立能力（capability flag），完全不影响现有架构
* 甚至可以换商业 SDK 也不动 feature

## Jetpack `androidx.pdf` 后端：怎么用得“专业且不被绑死”

你**不要直接用它的 UI 组件（PdfView/PdfViewerFragment）**来替代你的 Reader UI；你要的是“引擎层”。

推荐你这样用它：

1. 在 `PdfBackend` 层实现一个 `JetpackPdfBackend`：

   * 用 `pdf-document-service`（它的定位就是处理层，并且 sandbox `PdfRenderer`）。([Maven Repository][3])
2. 运行时判断可用性：

   * 需要满足 API/Extension 条件（官方 Fragment 是 `S ext 13` 起）。([Android Git Repositories][2])
3. 不可用就自动 fallback 到 PDFium 后端。

这样你不会把产品绑死在 alpha 库上，但能在支持的设备上吃到官方能力红利。

## PDFium 后端：专业落地注意事项

**工程化要点：**

* ABI：arm64-v8a/armeabi-v7a/x86_64 体积控制
* 渲染线程：限制并发（CPU + native heap）
* 资源释放：document/page/textPage 必须严格 close（建议封装成 Kotlin `use` 风格）

**能力实现：**

* render page / render region（tile）
* getPageSize / pageCount
* outline/bookmarks
* links（URI + internal）
* text extraction / search（能做就做，做不到就把 capabilities 置 false）

## 最后：给你一个“落地顺序”但不是 MVP（是专业交付的拆分）

你可以按模块交付，但每一阶段都是“可上线质量”：

1. **open + 渲染（Tiles + 缓存 + 取消）**

   * 打开/关闭无泄漏
   * 翻页 60fps 体感（DRAFT->FINAL）
2. **links + outline**

   * 热区点击正确、目录跳转正确
3. **text extraction + selection（至少整页）**
4. **search（Flow 流式结果 + 高亮 rect）**
5. **annotations（外置存储 + 叠加渲染）**
6. **后台索引（WorkManager + FTS/AppSearch）**
7. **benchmark + baseline profile**（你已经预留模块）
