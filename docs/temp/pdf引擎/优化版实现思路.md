下面给你一套**最推荐、最贴合你现有 reader/api + TileProvider 架构**的 PDF 引擎实现思路：一句话总结就是——

**PDF = 固定版式（Fixed Layout）+ 瓦片渲染（Tiled Rendering）+ 坐标型 Locator + 分层可选的“文本/搜索/链接/批注”能力。**
并且**优先用 Android Framework / Jetpack 的 PDF 能力**，尽量避免早早引入 PDFium 这类 native 方案（后面说原因）。

---

## 1) 先选“最稳的底座”：优先走 Framework / Jetpack PDF 能力（2026 的现实最优解）

### 为什么推荐

从 Android V（API 35）开始，`PdfRenderer` / `PdfRenderer.Page` 增强了**文本提取、搜索、内容选择、链接、表单、写回（保存编辑/批注）**等能力；并且还提供了 `PdfRendererPreV`，用于在 Android R~U（API 30–34）上暴露这些 Android V 的能力。([Android Developers][1])

也就是说：

* **API 21+**：至少能渲染（最基本 MVP）
* **API 30–34**：有机会用 `PdfRendererPreV` 拿到“接近 API35 的高级能力”([Android Developers][1])
* **API 35+**：直接用增强版 `PdfRenderer` / `PdfRenderer.Page` 的完整能力([Android Developers][1])

另外 Jetpack 还有 `androidx.pdf`：提供现成 Viewer UI、以及 document-service 等拆分组件（包括 sandbox/隔离处理层）。([Android Developers][1])

### 为什么不建议“一上来就 PDFium”

不是说不能用，而是**在 2025/11/1 起 Google Play 对 16KB page size 兼容有硬性要求**，会影响所有包含 native `.so` 的依赖（包含很多 PDFium 封装库）。([Android Developers][2])
如果你引入 native 方案，后续会多一条长期维护线：对齐、打包、ABI、NDK、依赖升级、Play Console 合规检查等。

**因此最推荐策略**：

> **默认用 framework/Jetpack（纯 Java/Kotlin 调用、无 native 负担）**；
> 只有在你必须支持 “API < 30 的文本搜索/文本提取” 或必须实现更深层 PDF 特性时，再考虑 PDFium 作为可插拔后端。

---

## 2) 引擎形态：把 PDF 当“图片化页面流”，用 TileProvider 做瓦片渲染

PDF 最大的不同：它不是可重排文本，而是**固定版式**。所以你应该把 PDF 引擎做成“页面渲染器”，而不是“章节排版器”。

### 你引擎内部最关键的抽象：`PdfBackend`

做一个内部接口，把平台差异收进去：

* `LegacyBackend`：仅渲染（API 21+，用老 `PdfRenderer`）
* `PreVBackend`：渲染 + 文本/搜索/链接（如果 `PdfRendererPreV` 可用）([Android Developers][3])
* `VBackend`：渲染 + 文本/搜索/链接（API 35+，用增强 `PdfRenderer.Page`）([Android Developers][4])

> 注意：`PdfRenderer` / `PdfRendererPreV` 都强调**线程安全但内部锁串行执行**，并且**同一时刻只能打开一个 Page**（你必须快速 open/render/close）。([Android Developers][5])

### TileProvider 的推荐设计

把每页视为一个“无限放大”的平面，用瓦片（tile）渲染来解决内存：

* TileKey: `(pageIndex, zoomBucket, tileX, tileY, tileSizePx)`
* 渲染请求由 UI 发起：只渲染当前 viewport 可见区域对应的 tile
* 缓存：

  * **内存 LRU**：存 Bitmap（最重要）
  * **磁盘缓存（可选）**：存 page 缩略图 / 常用 tile
* **渐进渲染**（体验好）：

  * 手势缩放时：先把已有低分辨率 tile 进行拉伸显示（立即反馈）
  * 停止缩放后：触发高分辨率 tile 重渲染替换

Android 官方也明确：`PdfRenderer.Page#render()` 可以渲染**整页或部分区域**，缩放场景可以渲染“某一部分到指定大小 bitmap”。([Android Developers][1])

## 3) Locator 设计：PDF 最推荐用“页 + 坐标”，而不是纯进度百分比

你的 `core/model/Locator` 很关键。对 PDF 推荐一个 scheme：

* `pageIndex: Int`
* `pos: (xRatio, yRatio)` 或 `rect: (left, top, right, bottom)`（都用 0..1 归一化到页面宽高）
* 可选：`zoomHint`（不参与定位稳定性，只做体验）

用途：

* 阅读进度：存“当前页 + 页内顶部 yRatio”
* 书签：存“页 + 一个点 / 一个小矩形”
* 选中文本高亮：存“页 + 多个 rects（或 quads）”

这样 Locator **不会被屏幕尺寸/缩放影响**，并且对跨设备同步更稳。

## 4) 文本、搜索、链接：用 Android V 的内容 API 做 Provider（能力可降级）

这是 2026 的甜点：你可以直接用 `PdfRenderer.Page` 的内容方法来实现 `TextProvider / SearchProvider / ResourceProvider` 的大部分需求：

### 4.1 TextProvider（可选能力）

`PdfRenderer.Page.getTextContents()` 会返回页面文本块及其边界框（按从左到右、从上到下排序）。([Android Developers][4])
你可以把它转成：

* 页面全文字符串（用于“复制/分享/摘要”）
* `TextMapping`（字符范围 ↔︎ 矩形区域）
* 命中测试：点到哪个文本块（用于选择/菜单）

> API < 30：可能只有渲染，没有文本；这时 TextProvider 直接返回“不支持”。

### 4.2 SearchProvider（强烈推荐做）

`PdfRenderer.Page.searchText(query)` 可以直接返回所有匹配的边界框集合。([Android Developers][4])

实现策略（很重要）：

* **不要并发扫全书**：文档里明确 searchText 如果之前对任何页面发起过搜索，会等待上一次完成再开始。([Android Developers][4])
  所以你应该做成：

  * 单通道、可取消的顺序搜索（Coroutine + Mutex / single-thread dispatcher）
  * 每扫完一页就 emit 局部结果（Flow），让 UI “边搜边出结果”
* 搜索结果 Locator：

  * `pageIndex + rects`（可直接高亮框）

### 4.3 LinkProvider / OutlineProvider（可选但很值）

* `getLinkContents()`：拿到页面上所有 URL link 的 bounds + URL。([Android Developers][4])
  你可以在 UI 上做透明点击层：点击矩形区域 -> 打开外链
* `getGotoLinks()`：拿到页面内的 “书签/内部跳转链接”。([Android Developers][4])
  你可以把 destination 映射为 Locator（页 + 坐标），实现“目录/内部引用跳转”。

> 真正“文档级目录树”（Outline）在不同实现里可能来源不同：
>
> * 若 `getGotoLinks()` 能覆盖你的需求（很多 PDF 的目录/内部跳转就是 goto link），那可以先当 MVP。
> * 更完整的 outline 可能需要额外解析（这时可以考虑 `androidx.pdf` 的处理层/服务，或后续再引入更重的解析器）。

## 5) 批注/高亮：推荐“先自管叠加”，再考虑写回 PDF

现实做法（对阅读器最稳）：

### MVP（最推荐先做）

* 批注/高亮**存到你自己的数据库**（core/database + core/data 的 AnnotationRepo）
* UI 渲染时，在页面 tile 上方叠加绘制 highlight/underline/notes（基于 PDF 页坐标 -> 屏幕坐标的变换矩阵）

优点：

* 兼容所有 Android 版本（不依赖新 API）
* 不改动原 PDF 文件（对用户更安全）
* 同步/备份简单

### 进阶：写回 PDF（新 API 路线）

Android V/新 API 有 `PdfRenderer.write(...)` 用于把“当前状态（包含编辑/批注/表单等）”写回到输出流。([Android Developers][1])
并且 `PdfRenderer.Page` 在较新 API 中还出现了获取/移除页面批注等能力（文档里能看到 `getPageAnnotations()` 等新增）。([Android Developers][4])

建议把它当成 **“后续高级功能开关”**：

* 开关打开：允许导出“带批注的新 PDF”
* 默认：仍以你自己的 annotation 系统为准

## 6) 打开策略与安全：PDF 一定要保证“seekable FD”，并考虑隔离处理

`PdfRendererPreV` 文档明确：传入的 `ParcelFileDescriptor` **必须可 seek（随机访问）**，否则会报错。([Android Developers][3])
所以你的 `PdfOpener` 必须有一条“兜底”：

* 如果 `DocumentSource` 是 SAF Uri：

  * 尝试 `contentResolver.openFileDescriptor(uri, "r")`
  * 如果不满足 seekable / 性能很差：**复制到 app cache 私有文件**，再用 file FD 打开

安全方面：
`PdfRendererPreV` 文档也建议：如果文件来自不可信来源，推荐在“隔离进程/最小权限进程”里跑渲染，降低 PDF 解析漏洞影响。([Android Developers][3])
你可以两条路：

* **简单可控**：你自己做 `:pdf` isolated process + Binder（成本高但最强）
* **借力 Jetpack**：`androidx.pdf:pdf-document-service` 目标之一就是 sandbox 底层 PdfRenderer 的解析处理层。([Maven Repository][6])

## 7) 推荐的 engines/pdf 内部结构（与你的风格一致）

```text
engines/pdf/
├── PdfEngine.kt                         # 实现 ReaderEngine：声明 capabilities
└── internal/
    ├── open/
    │   ├── PdfOpener.kt                 # DocumentSource -> seekable FD / cache copy
    │   └── PdfDocument.kt               # 实现 ReaderDocument
    ├── backend/
    │   ├── PdfBackend.kt                # 抽象：render / text / search / links
    │   ├── PdfRendererBackend.kt        # API21+: render-only
    │   ├── PdfRendererPreVBackend.kt    # 若可用：R~U 的高级 API :contentReference[oaicite:19]{index=19}
    │   └── PdfRendererVBackend.kt       # API35+: 完整高级 API :contentReference[oaicite:20]{index=20}
    ├── render/
    │   ├── PdfTileProvider.kt           # 实现 TileProvider：瓦片渲染 + 缓存 + 预取
    │   ├── PdfViewportMapper.kt         # page<->viewport 的 Matrix/坐标转换
    │   └── PdfThumbGenerator.kt         # 缩略图（页列表/封面）
    ├── provider/
    │   ├── PdfTextProvider.kt           # getTextContents -> TextMapping（可选）:contentReference[oaicite:21]{index=21}
    │   ├── PdfSearchProvider.kt         # searchText -> match rects（可选）:contentReference[oaicite:22]{index=22}

        └── TileDiskCache.kt             # 可选
```

## 8) 如果你想“最快落地”（不追求引擎统一抽象）的备选方案

如果你的目标是 **先把 PDF 能看、能搜、能选**，并且允许 PDF 走独立 UI：

* 直接用 Jetpack `androidx.pdf` 的 `PdfViewerFragment` / Compose 组件嵌入阅读页。
  它本身就主打：打开/浏览、搜索与选择、缩放滚动、保存批注、手写笔交互等。([Android Developers][1])

代价是：

* 你的 reader/api 统一抽象会被绕开一部分（PDF 变成“特殊页面”）
* 但上线速度快、功能完整度高

## 结论：我最推荐你走的路线

**最佳推荐（长期正确 + 贴合你架构）：**

1. `PdfRenderer` 做最低保真渲染（API 21+）；
2. **优先接入 `PdfRendererPreV / Android V 的 PdfRenderer.Page 内容 API`**实现文本/搜索/链接 provider（API 30+ 体验直接上一个档次）；([Android Developers][1])
3. 批注先“自管叠加”，写回 PDF 当高级开关；
4. 若考虑安全与隔离，优先评估 `androidx.pdf:pdf-document-service` 或独立进程渲染。([Maven Repository][6])

如果你愿意，我可以基于你 `core/reader/api` 里现有的 `TileProvider / RenderConfig / Locator` 的具体定义，再给你一份更贴代码的：

* `PdfTileProvider` 的 TileKey 设计 + 缓存策略参数
* `PdfViewportMapper` 的矩阵/坐标换算公式（页 points ↔︎ 像素）
* `SearchProvider` 的 Flow 分页产出与取消策略（避免 searchText 串行阻塞）
