下面给你一套**我最推荐、可长期演进**的“书籍导入”方案：**两阶段导入 + WorkManager 管线 + app 私有存储落盘 + 引擎探测/轻量解析能力可插拔**。它能完美贴合你现在的模块边界与依赖规则（feature 不碰 engines）。

---

## 目标与原则（先定基调）

1. **导入后可离线、可稳定打开**：不要依赖外部 Uri 永久在线（用户撤权/SD 卡拔出/云盘失效很常见），导入成功即**拷贝到 app 私有目录**作为 canonical 文件。
2. **导入快、富化慢**：UI 选择文件后立刻入队，快速完成“落盘+建档”；封面/目录/全文索引等“慢活”异步做。
3. **格式扩展零改 feature**：格式识别 + 引擎选择全部走 `core/reader/runtime` + `EngineRegistry`，导入逻辑只认 `reader/api` 契约。

---

## 总体架构（模块映射到你现有目录）

### 1) feature/library（只负责“入口与展示”）

* UI：

    * “选择文件”(`ACTION_OPEN_DOCUMENT`) / “选择文件夹”(`ACTION_OPEN_DOCUMENT_TREE`)
    * “导入队列/进度”列表、失败重试、取消
* domain/usecase：

    * `StartImport(uris)`：发起导入
    * `ObserveImportJobs()`：订阅导入任务进度
    * `RetryImport(jobId)` / `CancelImport(jobId)`
* **注意**：feature 不做任何 IO/解析，全部调用 core 层服务。

### 2) core/files（导入核心编排：权限、拷贝、扫描）

新增/完善这些组件（都放在 `core/files`）：

* `ImportManager`：导入入口（给 feature 调用）
* `UriPermissionStore`：封装 `takePersistableUriPermission()`、权限恢复/校验
* `FileStaging`：落盘策略（临时文件 -> 原子 rename）
* `TreeScanner`：文件夹扫描（递归 DocumentFile、过滤扩展名/MIME）
* `Fingerprint`：复制时顺带做 hash（用于去重/一致性）

### 3) core/reader/runtime（格式探测 & 引擎能力调用）

导入过程中需要两件事：

* `BookFormatDetector`：根据 mime/扩展名/魔数/zip 内部判断（你已经规划了）
* **轻量元数据探测（推荐加一个能力）**：避免导入时完整 open + 渲染初始化
  我建议在 `core/reader/api` 增加一个“可选能力”接口（由 engines 实现）：

```kotlin
interface MetadataProbe {
  suspend fun probe(source: DocumentSource): ReaderResult<DocumentMetadata>
}
```

引擎实现：

* EPUB：读 OPF/metadata
* PDF：读 info dictionary（可选）
* TXT：文件名推断标题 + 简单编码探测（可选）

> 没实现 probe 的引擎：导入阶段只建档（title=文件名），富化阶段再慢慢补。

### 4) core/database + core/data（持久化与业务仓库）

推荐表/字段（核心是“书籍实体”和“导入任务实体”）：

* `BookEntity`

    * `id`
    * `format`
    * `title`, `authors`, `language`（可空）
    * `createdAt`, `updatedAt`
    * `coverPath`（指向私有存储缓存图）
    * `canonicalFilePath`（私有目录真实文件路径）
    * `originalUri`（可选，仅用于“回溯来源/重新导入”，不要依赖它阅读）
    * `fingerprint`（sha256/xxhash64）
    * `fileSize`, `modifiedAt`

* `ImportJobEntity`

    * `jobId`
    * `status`：QUEUED/RUNNING/SUCCEEDED/FAILED/CANCELLED
    * `total`, `done`, `currentTitle`
    * `errorMessage`（可空）
    * `createdAt`, `updatedAt`

`core/data` 提供：

* `BookRepo.upsertBook(...)`
* `ImportJobRepo.create/update/observe(...)`

### 5) core/work（可选但我强烈建议用）

* `ImportWorker`：复制 + 建档（快）
* `EnrichWorker`：封面/目录/全文索引（慢，可分多个 worker）
* 使用 **WorkManager + ForegroundInfo**（大文件/批量导入避免后台被杀）

## 推荐的导入管线（两阶段）

### 阶段 A：Ingest（强一致：落盘 + 建档）

**输入**：用户选择的一批 `Uri` 或 `treeUri` 扫出来的一批文件 Uri
**输出**：本地 canonical 文件 + 最小可用的 Book 记录

步骤：

1. `feature/library` 调用 `ImportManager.enqueue(request)`
2. `ImportManager`：

    * 记录 `ImportJobEntity(QUEUED)`
    * 为每个文件生成 `WorkRequest`（或一个 worker 内部顺序处理）
3. `ImportWorker`（每本书的核心流程）：

    * 申请/恢复 Uri 权限（SAF）
    * 打开输入流 `ContentResolver.openInputStream(uri)`
    * **边复制边计算 fingerprint**（避免二次读）
    * 落盘到：`/data/data/.../files/books/{bookId}/original.{ext}`
      采用：`tmp -> fsync(可选) -> rename` 保证原子性
    * `BookFormatDetector.detect(source)` 得到 format
    * 去重策略（推荐默认“跳过重复”）：

        * 如果 fingerprint 已存在：标记该项为 SKIPPED，并推进进度
    * 写入/更新 `BookEntity`（title 暂用文件名）
    * 更新 `ImportJobEntity(done++)`

> 这样做的好处：导入完就能打开阅读（至少 TXT/PDF/EPUB 可以），后面的富化失败也不影响阅读。

### 阶段 B：Enrich（弱一致：补齐元数据/封面/目录/索引）

触发时机：

* Ingest 成功后自动 enqueue
* 或设备空闲/充电时批处理（可由策略控制）

`EnrichWorker` 做：

* 如果引擎实现了 `MetadataProbe`：补齐 `title/authors/identifier` 等
* 提取封面（EPUB）或渲染首屏缩略图（PDF）到 `coverPath`
* 抽取目录/章节（OutlineProvider）
* 如果你做全文检索：构建索引（可拆成单独 worker）

## SAF/权限与来源处理（导入最容易踩坑的点）

**推荐策略：导入即拷贝到私有目录；原 Uri 只做“来源记录”。**

* 用户从系统文件管理器选的 Uri：

    * `ACTION_OPEN_DOCUMENT` 返回的 Uri：调用 `takePersistableUriPermission()`
    * 但你仍然应该拷贝到私有目录，避免未来权限变化、云盘失效。
* 用户选文件夹：

    * `ACTION_OPEN_DOCUMENT_TREE` 获取 treeUri，持久化权限
    * 扫描时用 `DocumentFile.fromTreeUri` 递归列举
* 删除书籍时：

    * 删除私有 canonical 文件
    * 可选：如果没有其它书引用同一个 treeUri，再清理权限记录（不要随意 revoke，可能还有别的导入任务在用）

## 与 EngineRegistry 的关系（保证 feature 永远不改）

导入逻辑只依赖：

* `core/reader/runtime`：`BookFormatDetector`、`EngineRegistry`
* `core/reader/api`：`ReaderEngine` +（可选）`MetadataProbe`

在 `app` 做装配：

* engines 通过 Hilt multibinding 注册到 `EngineRegistryImpl`
* runtime 通过 registry 找到能处理该 format 的引擎

这样未来新增 `mobi/azw3`：

* 只加 `engines/mobi` + app 注册
* `feature/library` 和 `core/files` 不动

---

## 默认去重/冲突策略（我建议的“最省心”默认值）

1. **fingerprint 已存在**：默认“跳过”，并在 UI 显示“已存在（跳过）”
2. 文件名相同但 fingerprint 不同：当作新书（可在书架显示“(1)”或保留原名）
3. 失败可重试：保留失败项的 error（ReaderErrorMapper 统一映射）

## 你可以直接落地的接口形状（建议放 core/files）

```kotlin
data class ImportRequest(
  val uris: List<Uri> = emptyList(),
  val treeUri: Uri? = null,
  val strategy: DuplicateStrategy = DuplicateStrategy.SKIP
)

enum class DuplicateStrategy { SKIP, REPLACE, KEEP_BOTH }

interface ImportManager {
  fun enqueue(request: ImportRequest): String // jobId
  fun observe(jobId: String): Flow<ImportJobState>
  suspend fun cancel(jobId: String)
}
```

---

## 为什么这是“最推荐”的方案

* **稳定性第一**：导入后阅读不依赖外部 Uri，少 90% 的线上坑。
* **性能与体验兼顾**：快路径只做复制建档，慢路径后台富化；大批量也不怕 ANR/被杀。
* **架构一致性**：完全遵守你的依赖规则；feature 不碰 engines；扩展格式成本最低。
* **可测试性强**：`ImportWorker`、`FormatDetector`、`Fingerprint` 都能在 `core/testing` 做假实现和 fixture。
