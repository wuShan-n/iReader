下面给你一套**“最推荐、优先级最高、落地性强”**的优化方案（按 **P0→P1→P2** 排序）。这些建议完全基于你贴出来的 `feature/reader` 源码，重点解决：**卡顿/阻塞翻页、潜在崩溃（Bitmap/TileProvider 生命周期）、主线程 I/O、事件堆积、Compose/AndroidView 性能** 这五类核心问题。

## P0（必须优先做）1：FixedLayout（PDF/固定排版）两段渲染“锁死翻页” + 渲染并发风险

### 你现在的关键问题

在 `ReaderViewModel.navigate()` 里：

* 固定排版时你会 `renderWithFinalPass()` 做 **DRAFT → FINAL** 两次渲染。
* 但这段逻辑被 `render.withNavigationLock { ... }` 包住了——**FINAL 渲染期间 mutex 一直被占用**。
* 结果：用户快速翻页时，下一次 `Next/Prev` 会被锁等待 FINAL 完成，体验就是“翻页粘滞/卡住”。

同时还有一个隐患：
`RenderCoordinator` 触发的 `renderCurrentPageImmediate()` **没有使用同一把 navigationMutex**，它可能在导航期间并发调用 `controller.render()`（取决于 controller 的线程安全实现）。这类并发通常是渲染系统的大雷点。

### 最推荐的改法（最小改动、收益最大）

**把 FINAL 渲染变成“可取消的后台 job”，并且 FINAL 不占用导航锁**，导航只做“快动作 + 草稿页”，FINAL 在用户停下来后补帧。

#### 目标行为

* 翻页：立即返回 DRAFT（快），UI立刻更新；
* FINAL：异步补渲染；如果用户又翻页，**立即 cancel 上一次 FINAL**；
* 全部对 `controller.*` 的调用仍然串行（避免并发），但 FINAL 不再“堵住用户操作”。

#### 示例实现（核心结构）

在 `ReaderViewModel` 增加：

```kotlin
private var finalRenderJob: Job? = null

private fun cancelFinalRender() {
    finalRenderJob?.cancel()
    finalRenderJob = null
}
```

改造固定排版的导航逻辑（伪代码，重点在结构）：

```kotlin
private suspend fun navigate(
    direction: PageTurnDirection? = null,
    block: suspend (ReaderController, RenderPolicy) -> ReaderResult<RenderPage>
): Boolean {
    val sessionHandle = session.currentHandle() ?: return false

    // 新导航开始前，取消上一轮 FINAL 补帧
    cancelFinalRender()

    var success = false
    render.withNavigationLock {
        val fixed = sessionHandle.document.capabilities.fixedLayout
        val actionPolicy = if (fixed) RenderPolicy(quality = RenderPolicy.Quality.DRAFT) else RenderPolicy.Default

        when (val result = block(sessionHandle.controller, actionPolicy)) {
            is ReaderResult.Err -> {
                ui.update { it.copy(error = errorMapper.map(result.error)) }
                success = false
            }
            is ReaderResult.Ok -> {
                if (!fixed) {
                    replacePage(result.value, direction)
                    success = true
                } else {
                    // 1) 先上 DRAFT
                    replacePage(result.value, direction)
                    success = true

                    // 2) 再启动 FINAL 补帧（可取消）
                    val expectedPageId = result.value.id
                    ui.update { it.copy(isRenderingFinal = true) }
                    finalRenderJob = viewModelScope.launch {
                        // 可选：小延迟，避免用户连续翻页时浪费 FINAL
                        kotlinx.coroutines.delay(120L)

                        val finalResult = render.withNavigationLock {
                            sessionHandle.controller.render(RenderPolicy(quality = RenderPolicy.Quality.FINAL))
                        }

                        if (finalResult is ReaderResult.Ok) {
                            // 只在“页面没变”的情况下替换 FINAL，避免旧 FINAL 覆盖新页
                            if (ui.state.value.page?.id == expectedPageId) {
                                replacePage(finalResult.value, direction = null)
                            }
                        } else if (finalResult is ReaderResult.Err) {
                            // FINAL 失败一般不应阻断阅读，可考虑仅 toast/snackbar
                        }

                        ui.update { it.copy(isRenderingFinal = false) }
                    }
                }
            }
        }
    }
    return success
}
```

再把 `renderCurrentPageImmediate()`（布局变化/设置变化触发的 render）也改成同样模式：

* 固定排版：DRAFT 立即上屏 + 异步 FINAL（可取消）
* 全程用同一个 `navigationLock` 包住 `controller.render()`，避免与 navigate 并发。

✅ 这个改动的收益非常大：**固定排版翻页顺滑度提升最明显**，而且还顺带把“渲染并发”风险压掉。

## P0 2：资源生命周期（Bitmap recycle / tileProvider.close）存在“提前释放/重复释放”风险

### 你现在的风险点

`ReaderViewModel.replacePage()` 在更新 state 之前就会：

```kotlin
releasePageContent(previous)
```

`releasePageContent()` 会：

* `BitmapPage.bitmap.recycle()`
* `Tiles.tileProvider.close()`

但 UI（Compose）对上一页的绘制/动画/下一帧是否还会触碰到旧对象，是不完全可控的。
尤其是你后续若给 Bitmap/Tiles 也加动画，或者 Compose 在帧调度上出现边界情况，**“recycled bitmap 被绘制”**这种 crash 很常见。

另外：`TilesPage` 自己也 `DisposableEffect` 里 close 了 `tileProvider`，ViewModel 也 close——属于重复释放。

### 最推荐方案：让 UI 成为资源的唯一释放者（基于 DisposableEffect）

原则：**谁持有在屏幕上展示的对象，谁负责释放**。
因此把 `Bitmap.recycle` / `tileProvider.close` 从 ViewModel 移到 UI 侧，在页面离开 composition 时释放。

#### 具体落地（建议组合拳）

1. **ViewModel 里删掉** `replacePage()` 中对 previous 的 `releasePageContent(previous)`
2. **UI 侧**在 `PageRenderer` 按 pageId 做 `DisposableEffect`，释放上一页资源
3. Tiles 的 provider 释放**只保留一个地方**：

   * 要么保留 `TilesPage` 里已有的 `DisposableEffect`（推荐，局部自洽）
   * 要么统一到 `PageRenderer`，但那要调整 TilesPage 现有逻辑

#### PageRenderer 添加 Bitmap 回收（示例）

在 `PageRenderer` 最开始拿到 `page` 后加：

```kotlin
val page = state.page
if (page != null) {
    DisposableEffect(page.id.value) {
        onDispose {
            val content = page.content
            if (content is RenderContent.BitmapPage) {
                runCatching {
                    if (!content.bitmap.isRecycled) content.bitmap.recycle()
                }
            }
            // Tiles 的 close 建议只做一次：如果 TilesPage 已经 close，这里不要再 close
        }
    }
}
```

然后把 ViewModel 的 `releasePageContent()`（至少对 Bitmap/Tiles 那部分）逐步移除或缩小责任范围。

✅ 这个优化的目标是：**避免“提前 recycle/close”导致的随机崩溃**，同时消除重复 close。

## P0 3：打开书籍流程里存在主线程 I/O 风险（容易造成打开时卡顿）

在 `open(args, password)` 中你直接调用：

* `bookRepo.getRecordById`
* `sourceResolver.resolve(book)`
* `progressRepo.getByBookId`
* `bookRepo.setIndexState(...)`
* `bookRepo.touchLastOpened(...)`

这些很可能涉及 DB/文件系统，哪怕 repo 内部有调度，也建议在 ViewModel 层**明确把“读取 book / resolve source / 读 progress”放到 IO dispatcher**，避免任何实现差异导致 UI 卡顿。

### 推荐改法：把“查书/查进度/resolve source”集中放进 IO

结构上保持你现在的逻辑，只把数据准备部分包到 `withContext(Dispatchers.IO)`：

```kotlin
val prepared = withContext(Dispatchers.IO) {
    val book = bookRepo.getRecordById(args.bookId) ?: return@withContext null
    val source = sourceResolver.resolve(book) ?: return@withContext Pair(book, null)
    val historyLocator = progressRepo.getByBookId(book.bookId)?.locatorJson?.let(locatorCodec::decode)
    Triple(book, source, historyLocator)
}
```

然后在主线程上做 UI state 更新和 session attach。

✅ 这个优化主要提升：**打开/切书/恢复进度时的流畅度**，减少 ANR 风险。

## P1 4：事件通道（Channel.UNLIMITED）可能导致内存增长 + 输入延迟；Effect tryEmit 可能丢关键事件

### 你现在的风险点

* `private val intents = Channel<ReaderIntent>(capacity = Channel.UNLIMITED)`
  任何高频事件（drag/tap/layoutChanged）都可能堆积，导致：

  * 内存上涨
  * 事件延迟（用户操作反馈变慢）

* `ReaderUiReducer.emit(effect)` 使用 `tryEmit`，缓冲满会丢。
  Snackbar 丢还能忍，但 **Back / OpenAnnotations / Share** 这类不该丢。

### 最推荐改法

**Intent 用有限缓冲或 SharedFlow drop oldest；Effect 分级处理（关键 effect 不丢）**

#### Intent：把 UNLIMITED 改为 BUFFERED + 丢弃策略

最简单且安全：

```kotlin
private val intents = Channel<ReaderIntent>(capacity = Channel.BUFFERED)
```

或者更可控：SharedFlow + DROP_OLDEST（适合高频 UI 事件）

```kotlin
private val intents = MutableSharedFlow<ReaderIntent>(
    extraBufferCapacity = 64,
    onBufferOverflow = BufferOverflow.DROP_OLDEST
)
```

#### Effect：关键事件用 emit（挂起保证送达）

把 `ReaderUiReducer.emit()` 设计成 suspend 或者在 VM 里针对关键 effect 使用：

```kotlin
viewModelScope.launch { effectStore.emit(ReaderEffect.Back) }
```

并保留 `tryEmit` 只给非关键（如频繁 Snackbar）。

✅ 这个优化的目标：**防止输入堆积导致的“越用越慢”**，并确保导航类 effect 不会丢。

## P1 5：TextPage（AndroidView + TextView）更新成本偏高：每次 recomposition 都重建 Spannable + requestLayout

### 你现在的情况

`TextPage.update` 每次都会：

* `buildDisplayText(...)` 新建 `SpannableString` + setSpan
* setPadding / setLineSpacing / setTypeface / setTextColor / setBackgroundColor
* `textView.requestLayout()`

如果页面装饰（高亮、链接）多，或者频繁刷新页面，会导致明显的 UI 负担。

### 最推荐优化（不改功能，只降成本）

1. 把 `buildDisplayText` 移到 Compose 层用 `remember` 缓存
2. 在 update 里只对“确实变化”的属性 set
3. 尽量去掉 `requestLayout()`（TextView setText 本身会触发布局）

示例：

```kotlin
val displayText = remember(content.text, links, decorations, content.mapping) {
    buildDisplayText(content, links, decorations)
}

AndroidView(
  ...,
  update = { textView ->
      // typography 相关变化才更新
      // displayText 变化才 setText
      if (textView.text != displayText) {
          textView.text = displayText
      }
  }
)
```

Typeface 也建议缓存：按 fontFamilyName 做 `remember(fontFamilyName)`，避免每次 create。

✅ 这个优化提升：**文本页翻页/刷新更稳、更省电**。

## P2（可选但很值得）6：一些“顺手就能做”的质量优化

### 6.1 `ReaderFeatureModule` 可移除

`ReaderUiErrorMapper` 自己有 `@Inject constructor()`，Hilt 可以直接构造。
现在 module 只是多一层维护成本（而且还 `@Singleton`，但 mapper 是纯函数无状态）。

### 6.2 `applyLayout()` 可以加“相同约束不重复 set”

`LayoutChanged` 可能频繁触发（字体缩放、窗口 inset、旋转等）。
加一行短路就能省渲染：

```kotlin
if (constraints == render.currentLayout()) return
```

### 6.3 `cleanupScope` 在 onCleared 末尾 cancel

你创建了 `CoroutineScope(SupervisorJob() + Dispatchers.IO)`，但没有 cancel。
建议在 `onCleared()` 里最后：

```kotlin
cleanupScope.cancel()
```

避免 scope 持有 ViewModel 引用链更久。

### 6.4 persist=true 时避免“双重 setConfig”

`applyConfig(persist=true)` 会：

* 写 DataStore
* 立刻 `controller.setConfig(config)`
* 同时 `settingsJob` 监听到 DataStore 改变又 `setConfig`

可以通过“如果 controller 当前 config 已经是目标值就跳过”来避免重复：

```kotlin
if (sessionHandle.controller.state.value.config == config) return
```
