下面给你一套**“可长期维护、可扩展、可测试、可替换引擎、可做性能与质量门禁”的 feature 模块实现方案**。它完全贴合你现在的工程分层（core / engines / feature / app）、`ReaderRuntime` 入口、`ReaderResult/ReaderError` 体系，以及你已经写好的依赖规则。**不做 MVP 思路**，按“专业线上项目”的标准来落地。

---

## 0. 目标与总原则

### 你现在的架构已经很接近“正确答案”

* `feature/*` **只依赖** `core/*`（含 `reader/api` 或 `reader/runtime`），绝不直连 `engines/*`
* `app` 做唯一组装：DI 装配、EngineRegistry 注册、NavHost
* `reader/runtime` 已经提供了对 feature 的单一入口 `ReaderRuntime`：**feature 只关心 openSession / controller / providers**

### Feature 实现的核心原则（建议写进 docs/architecture.md）

1. **UDF（单向数据流）**：UI -> Intent(Action) -> VM -> State 更新 -> UI 渲染

   * 避免 UI 直接调用 repo、直接管理协程、直接接触异常
2. **Feature VM 只做编排，不做底层实现**

   * 底层：`core/data`, `core/datastore`, `core/reader/runtime`, `core/files`
3. **错误统一**：所有 IO/引擎/DB 错误最终落到 `ReaderError` / `DomainError`，再映射到 UI 文案/动作
4. **可测试**：每个 feature 的 domain/usecase 必须可单测；VM 必须可用 fake runtime/repo 测；UI 可 compose 测
5. **生命周期正确**：Reader 的 session/document 必须“成对关闭”，且能抵抗配置变更、返回栈切换、进程重建

## 1. Feature 模块统一模板（强烈建议标准化）

每个 `feature/<name>` 都建议按你给的结构，但我建议再补一层**“contract / internal”边界**（不是多模块，而是包层级边界），以防 feature 内部越写越乱：

```text
feature/<name>/
└── .../<name>/
    ├── navigation/          # route/args + graph
    ├── ui/                  # Compose UI（只渲染，不写业务）
    ├── presentation/
    │   ├── <Name>ViewModel.kt
    │   ├── <Name>UiState.kt
    │   ├── <Name>Intent.kt
    │   └── <Name>Effect.kt  # Snackbar/Toast/Jump 等一次性事件
    ├── domain/
    │   ├── usecase/
    │   └── model/           # feature 私有模型（尽量少）
    ├── di/
    └── internal/            # 可选：复杂 feature 的内部实现（如 reducer、paging source 等）
```

### 推荐的“通用基建”放在 core（减少重复）

建议在 `core/common-android` 或 `core/designsystem` 增加这些“所有 feature 都会用”的小基建：

* `UiText`：支持 stringRes / rawString（统一处理多语言与格式化）
* `UiMessage`/`SnackbarEvent`：一次性消息模型
* `UiErrorMapper`：`ReaderError/Throwable -> UiText + 可执行动作（重试/打开设置/申请权限）`
* `LoadingState`：Idle/Loading/Refreshing/Error 的统一模型

你已经有 `core/navigation`，建议：

* **Route 常量与 Args 解析**放 `core/navigation`（或 feature/navigation 内，但必须是稳定 API）

object ReaderRoute {

    fun create(bookId: String, locator: String? = null): String =

            append("reader?bookId=").append(bookId)
            if (locator != null) append("&locator=").append(locator)

fun NavGraphBuilder.readerGraph(

            onBack = onBack,

## 3. Presentation：统一采用 UDF（比“随手写 VM”更抗演进）

### 推荐状态模型：State + Intent + Effect

* `UiState`：**可持久化/可重建**（可放 SavedStateHandle 的关键字段）
* `Intent`：用户动作（点击、翻页、跳转、改设置）
* `Effect`：一次性事件（弹 Snackbar、导航、打开系统设置）

data class ReaderUiState(

sealed interface ReaderEffect {

## 4. Reader Feature：最关键的专业实现方案（完全贴合你的 runtime/api）

### 4.1 ReaderViewModel 的职责边界（非常重要）

Reader VM 只做这些事：

1. 根据 bookId -> 从 `BookRepo/Files` 得到 `DocumentSource`
2. 从 `ProgressRepo` 读取初始 locator（或 route 传入的 locator）
3. 调用 `ReaderRuntime.openSession(...)` 得到 `ReaderSessionHandle`
4. 订阅：

   * `controller.state` -> 更新 `renderState`
   * `controller.events` -> 处理 Rendered/Error/PageChanged
   * `settings flow` -> 推送 `setConfig`
   * `annotations flow` -> `invalidate()`
5. 翻页/跳转/渲染：只通过 `controller` 调用
6. 生命周期：`onCleared()` 必须 close handle（你已经提供了 Closeable handle，非常专业）

> VM **不应该**：解析格式、操作引擎内部类、处理 PDF tile 细节（这些由 controller/renderContent 决定）

### 4.2 强烈建议把 RenderSurface 做成“可类型识别”的契约（你现在是空接口）

你现在的 `RenderSurface` 是空接口，实际会逼迫引擎去强转 Any，类型风险高。
建议在 `core/reader/api` 增加一个标准实现（仍然不依赖 engines）：

```kotlin
// core/reader/api/render/AndroidViewRenderSurface.kt
class AndroidViewRenderSurface(
    val container: android.view.ViewGroup
) : RenderSurface
```

这样 EPUB/WebView 引擎就能安全判断：

```kotlin
when (surface) {
  is AndroidViewRenderSurface -> attachWebView(surface.container)
  else -> ReaderResult.Err(ReaderError.Internal(message="Unsupported surface"))
}
```

### 4.3 ReaderScreen：布局变化/绑定 surface 的专业做法

Reader 的坑在于：Compose 重组、横竖屏、返回栈切换会反复发生。建议用：

* `collectAsStateWithLifecycle`
* `DisposableEffect` 做 bind/unbind
* `onSizeChanged` 或 `BoxWithConstraints` 计算 `LayoutConstraints`

示例（关键结构）：

```kotlin
@Composable
fun ReaderScreen(
    onBack: () -> Unit,
    onOpenAnnotations: (String) -> Unit,
    vm: ReaderViewModel = hiltViewModel()
) {
    val state by vm.uiState.collectAsStateWithLifecycle()

    // 1) Surface（用于 Embedded / Html-WebView 等引擎）
    AndroidView(
        factory = { ctx -> FrameLayout(ctx) },
        modifier = Modifier.fillMaxSize(),
        update = { container ->
            vm.bindSurfaceIfNeeded(container)
        }
    )

    // 2) 布局约束
    BoxWithConstraints(Modifier.fillMaxSize()) {
        val density = LocalDensity.current
        LaunchedEffect(maxWidth, maxHeight, density) {
            vm.dispatch(
                ReaderIntent.UpdateLayout(
                    LayoutConstraints(
                        viewportWidthPx = with(density) { maxWidth.toPx().toInt() },
                        viewportHeightPx = with(density) { maxHeight.toPx().toInt() },
                        density = density.density,
                        fontScale = density.fontScale
                    )
                )
            )
        }

        // 3) UI 渲染（工具栏/菜单/页面）
        ReaderScaffold(
            state = state,
            onBack = onBack,
            onNext = { vm.dispatch(ReaderIntent.Next) },
            onPrev = { vm.dispatch(ReaderIntent.Prev) },
            onOpenAnnotations = { onOpenAnnotations(state.bookId) }
        )
    }
}
```

`bindSurfaceIfNeeded(container)` 推荐实现为：

* 首次创建 session 后才 bind
* 只 bind 一次；surface 变化（container identity 变化）才重新 bind
* `DisposableEffect(container)` 中 unbind

### 4.4 ReaderViewModel 核心实现建议（编排清晰、可测试、抗异常）

关键点：

* `openSession` 成功后：先 `setLayoutConstraints`、`setConfig`、再 `render()`
* `progress 保存`：监听 `controller.events` 的 PageChanged，**debounce/throttle** 写入 `ProgressRepo`
* `annotations 变化`：如果 session 有 annotations provider，订阅 repo 的 annotations flow -> `controller.invalidate(CONTENT_CHANGED)`
* 所有调用都走 `ReaderResult`，不用 throw

伪代码（结构比细节重要）：

```kotlin
@HiltViewModel
class ReaderViewModel @Inject constructor(
    private val runtime: ReaderRuntime,
    private val bookRepo: BookRepo,
    private val progressRepo: ProgressRepo,
    private val settings: ReaderSettingsStore,
    private val uiErrorMapper: UiErrorMapper,
    savedStateHandle: SavedStateHandle
) : ViewModel() {

    private val _uiState = MutableStateFlow(ReaderUiState())
    val uiState: StateFlow<ReaderUiState> = _uiState.asStateFlow()

    private val _effect = Channel<ReaderEffect>(Channel.BUFFERED)
    val effect = _effect.receiveAsFlow()

    private var handle: ReaderSessionHandle? = null
    private var boundContainer: ViewGroup? = null

    fun dispatch(intent: ReaderIntent) {
        when (intent) {
            is ReaderIntent.Start -> open(intent.bookId, intent.initialLocator)
            ReaderIntent.Next -> launch { controller().next().updatePage() }
            ReaderIntent.Prev -> launch { controller().prev().updatePage() }
            is ReaderIntent.GoTo -> launch { controller().goTo(intent.locator).updatePage() }
            is ReaderIntent.UpdateConfig -> launch { controller().setConfig(intent.config); controller().render().updatePage() }
            is ReaderIntent.UpdateLayout -> launch { controller().setLayoutConstraints(intent.constraints); controller().render().updatePage() }
            ReaderIntent.ToggleMenu -> _uiState.update { it.copy(menuVisible = !it.menuVisible) }
            ReaderIntent.Retry -> retry()
        }
    }

    fun bindSurfaceIfNeeded(container: ViewGroup) {
        boundContainer = container
        val h = handle ?: return
        launch {
            h.controller.bindSurface(AndroidViewRenderSurface(container))
        }
    }

    private fun open(bookId: String, locator: Locator?) = launch {
        _uiState.update { it.copy(bookId = bookId, isLoading = true, error = null) }

        // 1) source
        val source = bookRepo.getSource(bookId) // 你在 core/data 定义
            ?: return@launch fail("Book source not found")

        // 2) initial locator (route > saved progress)
        val initial = locator ?: progressRepo.getLastLocator(bookId)

        // 3) open session
        when (val r = runtime.openSession(source, initialLocator = initial)) {
            is ReaderResult.Err -> fail(r.error)
            is ReaderResult.Ok -> {
                handle?.close()
                handle = r.value

                _uiState.update {
                    it.copy(
                        isLoading = false,
                        capabilities = r.value.document.capabilities
                    )
                }

                observeController(r.value)
                observeSettings(r.value)
                // 初次渲染
                controller().render().updatePage()
            }
        }
    }

    private fun observeController(handle: ReaderSessionHandle) {
        // state
        viewModelScope.launch {
            handle.controller.state.collect { st ->
                _uiState.update { it.copy(renderState = st, title = st.titleInView) }
            }
        }
        // events -> save progress
        viewModelScope.launch {
            handle.controller.events
                .filterIsInstance<ReaderEvent.PageChanged>()
                .map { it.locator }
                .debounce(500)
                .collect { locator ->
                    progressRepo.save(handle.document.id, locator)
                }
        }
    }

    private fun observeSettings(handle: ReaderSessionHandle) {
        viewModelScope.launch {
            settings.observeRenderConfig()
                .distinctUntilChanged()
                .collect { cfg ->
                    handle.controller.setConfig(cfg)
                    handle.controller.render().updatePage()
                }
        }
    }

    private suspend fun ReaderResult<RenderPage>.updatePage() {
        when (this) {
            is ReaderResult.Ok -> _uiState.update { it.copy(page = value, error = null) }
            is ReaderResult.Err -> fail(error)
        }
    }

    private fun fail(err: ReaderError) {
        _uiState.update { it.copy(isLoading = false, error = uiErrorMapper.map(err)) }
    }
    private fun fail(msg: String) { /* ... */ }

    private fun controller(): ReaderController =
        requireNotNull(handle).controller

    private fun retry() { dispatch(ReaderIntent.Start(uiState.value.bookId, null)) }

    private fun launch(block: suspend () -> Unit) =
        viewModelScope.launch { block() }

    override fun onCleared() {
        handle?.close()
        handle = null
    }
}
```

> 这套结构的关键收益：**Reader 可以完全靠 fake runtime/controller 做 VM 单测**，而不用启动真正引擎。

## 5. Library / Import Feature：专业级落地（不只是“选文件+入库”）

### 5.1 推荐把“导入流程”做成强一致的 pipeline

导入不是 UI 行为，而是一个可恢复、可追踪的业务流程：

**Usecase 建议：**

* `PickDocuments`（UI 层触发 SAF）
* `ImportDocuments`（domain：复制到 app 私有目录/建索引/入库）
* `ProbeBook`（调用 `ReaderRuntime.probe()` 拿 metadata/capabilities/format）
* `UpsertBook`（写入 Room）
* `ScheduleIndexing`（可选：WorkManager）

你现在的 `ReaderRuntime.probe()` 已经很适合导入阶段：

* 在不创建 session 的情况下判断格式、读 metadata、capabilities（并 close document）

### 5.2 导入任务一定要后台化（WorkManager）

不只是为了“体验”，而是为了：

* 大文件复制不会被系统杀死导致半成品
* 批量导入可恢复（网络/存储/权限变化）

建议在 `core/work`：

* `ImportWorker`：输入为 Uri 列表（或 SAF persistable permission token），输出为入库结果
* `IndexWorker`：全文索引（如果做 search）

UI：library 只展示导入任务进度（flow from WorkManager）

## 6. Annotations Feature：跨格式统一的关键点

你已经把 annotation 分成：

* 引擎侧 `AnnotationProvider`（面向当前 session 的读写）
* app 侧 `AnnotationStore`（持久化抽象，Room/sync 实现）
  这很专业。**建议进一步明确职责：**

### 推荐职责分配

* `AnnotationStore`（core/data 实现）：负责 CRUD + observe（DB/sync）
* `AnnotationProvider`（引擎 session）：负责把“引擎定位模型”与 “store 模型”对齐

  * reflow（EPUB/TXT）：LocatorRange
  * fixed（PDF）：page locator + rects
* feature/annotations：只做 UI + usecase（列表、筛选、跳转、编辑）

### “跳回阅读页”统一方式

* annotations 列表项必须能生成一个**可导航的 locator**
* 点击后 `navigate(ReaderRoute.create(bookId, locator=serialize(locator)))`

## 7. Search Feature：分两条路，别混在一起

### 7.1 “当前书内搜索”（强烈建议先做这个）

直接走 session `SearchProvider`：

* 特点：实时流式、按章节/范围定位、可中断
* UI：输入 query -> collect flow -> list hits -> 点击 hit -> `controller.goTo(hit.range.startLocator...)`

VM 要点：

* query 变化要 cancel 上一个搜索：`flatMapLatest`
* hits 过多要分页/限制：你的 `SearchOptions.maxHits` 很好

### 7.2 “全库全文搜索”（如果要做，一定要建索引）

不要让 UI 去遍历每本书打开 session 搜索（会炸）。

* 用 WorkManager 建索引（按章节/页分片写入 FTS 或 Lucene/SQLite FTS5）
* `core/database`：FTS 表
* feature/search：只查索引表，点击结果再打开 reader 跳转 locator

## 8. Settings Feature：必须是“即时生效”的响应式

你已经有 `core/datastore`，建议：

* settings feature 只管 UI + usecase `UpdateReaderSettings`
* reader feature 通过 `ObserveReaderSettings`（flow）订阅并 `controller.setConfig()`

专业项目要注意：

* `distinctUntilChanged()` 防抖
* 有些设置影响 layout（padding/fontSize）要触发 `invalidate(CONFIG_CHANGED)` 或直接 `render()`

## 11. 你现在这套 reader/runtime 代码，我建议再补 3 个“专业增强点”

不改变总体架构，但能显著提升工程质量。

### 11.1 ReaderRuntime.openSession 增加自动恢复策略（可选）

* 从 ProgressRepo 自动取 locator（runtime 不依赖 data，不能做）
* 正确做法：保持 runtime 纯粹；由 feature 的 usecase `OpenBook` 负责读取 progress 并传入 initialLocator
  ✅ 你已经倾向这种做法，保持即可。

### 11.2 RenderSurface 类型化（前面讲过）

把 `RenderSurface` 从空接口升级为至少 1 个标准 surface（AndroidView 容器）。

### 11.3 controller.render 的错误事件要“结构化”

你有 `ReaderEvent.Error(val throwable: Throwable)`，建议长期演进为：

* `ReaderEvent.Error(val error: ReaderError)`
  这样 UI 永远不需要处理 Throwable（避免泄漏底层实现）

## 结论：我最推荐的 feature 实现方案（一句话）
