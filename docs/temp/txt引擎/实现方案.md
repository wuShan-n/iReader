下面给你一套**可以直接落地、面向“长期维护 + 大文件性能 + 正确性”的 TXT 引擎实现方案**，完全贴合你现在的 `core/reader/api + core/reader/runtime + engines/*` 架构（feature 不依赖 engines，app 做组装）。

我会把方案拆成：**打开/预处理管线、文本存储、分页与定位、搜索、目录(章节)、缓存与一致性、测试与基准**。这是“专业版”的实现，不是 MVP。

# 0. 目标与关键决策

## 你想要的 TXT 引擎应达到的能力

* **超大 TXT（几十 MB~数百 MB）可用**：打开不 OOM、翻页稳定、搜索可取消、不会卡死 UI。
* **编码可靠**：UTF-8/UTF-16/GBK/GB18030/Big5 等，BOM + 自动探测 + 用户指定都能工作。
* **定位稳定**：书签/进度/标注 anchor 在重启、设置变化后仍然正确（至少在同一份文本内容上）。
* **分页一致**：同一 `RenderConfig + LayoutConstraints` 下分页确定；支持缓存；横竖屏、字体变化能正确失效。
* **搜索专业**：流式输出结果（Flow），支持从某个 locator 起搜；可做到“首次可用 + 可选索引加速”。

## 最关键的工程决策：TXT 的 Locator 选什么单位？

你在模型里写了 `txt.offset`（建议字符偏移或字节偏移）。**我推荐：txt.offset 使用“UTF-16 code unit 的字符偏移（Long）”**，理由：

* Kotlin/Android 的 `CharSequence`、`StaticLayout`、选择范围、本地化断行，都天然是 UTF-16 语义。
* 如果你用“原文件字节偏移”，分页需要精确的“字节↔字符”映射，否则页尾定位会漂；UTF-8/GBK 这种可变长会让实现复杂且容易出错。
* 使用 UTF-16 偏移后，**只要我们把文本预处理成一个统一的 UTF-16 存储（缓存文件）**，定位、分页、搜索都变成可控问题。

> 结论：引擎内部建立一个**规范化后的 UTF-16LE 无 BOM 文本缓存**作为“事实源”，`txt.offset` 就是对这个事实源的 char offset（UTF-16 code unit offset）。

# 2. 打开流程：TxtEngine.open 的专业实现

TXT 打开不是“读成一个 String”，而是一个**可复用的预处理管线**：

## 2.1 缓存目录与指纹（保证复用与正确失效）

引擎需要一个可注入的文件系统/目录提供者（放 `core/files` 最合适）：

```kotlin
interface EngineFileSystem {
  fun engineCacheDir(engine: String, documentId: DocumentId): File
  fun engineDataDir(engine: String, documentId: DocumentId): File // 可持久化：分页索引、搜索索引
}
```

然后 `TxtCacheManager` 做：

* `documentId`：建议 `hash(uri + sizeBytes + displayName + (first64KB hash))`
* `fingerprint`：至少包含

  * `uri`（string）
  * `sizeBytes`
  * `displayName`
  * `detectedEncoding`
  * `normalizerVersion`（你改归一化逻辑时可强制失效）
  * `appVersionCode`（可选）
* 缓存文件：

  * `content.u16`（UTF-16LE 无 BOM）
  * `line.idx`（换行位置/行起始，压缩）
  * `softbreak.idx`（软换行位置/类型）
  * `chapters.json`（可选，或二进制）
  * `pagemap_<profile>.bin`（可选，分页索引）
  * `trigram.idx`（可选，搜索索引）
  * `meta.json`（fingerprint + 统计信息）

> 关键：**缓存复用必须基于 fingerprint**，而不是只看文件名。

## 2.2 编码探测（EncodingDetector）

优先级建议：

1. `OpenOptions.textEncoding`（用户指定）
2. BOM：UTF-8 / UTF-16LE / UTF-16BE / UTF-32(可选)
3. 采样探测：读取前 64KB~256KB，使用探测器（两种路线）

   * 路线 A（推荐、工程成熟）：引入 `juniversalchardet`（体积小、效果可接受）
   * 路线 B（更“纯”）：自己做启发式（UTF-8 合法性、零字节分布、常见中文编码候选）
4. fallback：UTF-8（或 GB18030，取决于你产品面向人群）

探测输出不仅是 Charset，还要输出 **confidence**（用于 UI 给出“编码可能不对”的提示/重选编码）。

## 2.3 规范化转换（TextNormalizer）

你最终要的事实源是 `content.u16`。转换时做：

* 换行归一：

  * `\r\n`、`\r` → `\n`
* 过滤/替换控制字符：

  * 保留 `\n`、`\t`（可转换为 4 空格）、其余 `\u0000..` 控制字符可替换为空格或删除（删除会改变 offset，不推荐）
* 可选：去掉文本头部 BOM（如果上一步没处理）
* 统计：

  * 总 char 数（UTF-16 code unit count）
  * 换行数量、平均行长、行长方差（用于硬换行检测）

写出 `content.u16` 的方式：

* 用 `CharsetDecoder` 从源流解码到 `CharBuffer`
* 规范化后写入 `FileChannel`，`ByteBuffer.order(LITTLE_ENDIAN)` + `putChar(c)`（不写 BOM）
* 全程 streaming，不把全书加载到内存

> 这一步是专业 TXT 引擎的核心：一次性把“杂乱的外部世界”收敛成“引擎可控的内部世界”。

# 3. “硬换行消除”不是可选项：SoftBreakIndex（专业阅读体验的分水岭）

大量 TXT（尤其爬虫/老站文本）是**按固定列宽硬换行**的；如果你原样渲染，阅读体验会很差。

但你又不能直接把 `\n` 删除，否则 locator/标注会漂移。

## 3.1 我推荐的做法：把每个换行分类，而不是删字符

* 保留文本内容不变（`\n` 仍然占一个 char offset）
* 建一个索引：哪些 `\n` 是 **SoftBreak**（行内硬换行），哪些是 **ParagraphBreak**（段落换行）
* 渲染时：

  * SoftBreak：显示为 `空格` 或 `ZWSP(零宽空格)`（**不改变长度**）
  * ParagraphBreak：显示为 `\n`（换段）

这样：

* locator 仍然稳定
* 视觉上实现“消除硬换行”
* 用户可在设置里切换策略（显示 softbreak 为换行 / 空格 / 零宽），不影响定位

## 3.2 SoftBreak 判定算法（足够工程化，效果好）

预处理时基于 line index 做一次扫描：

1. 判定“该文本是否硬换行占主导”

* 行长中位数在一个区间（如 30~120）
* 行长方差小
* 单行结尾不是句末标点的比例高
* 空行比例低（硬换行通常段落不靠空行分隔）

2. 如果是硬换行文本，对每个换行做分类：

* ParagraphBreak if：

  * 连续空行（`\n\n`）
  * 下一行像章节标题（第X章/Chapter X/...）
  * 下一行有明显段首缩进（中文“　　/  ”、或若干空格）
  * 当前行末是强段末标点（`。！？!?`）并且下一行首不是延续标志
* 其余为 SoftBreak

> 这套规则不用“完美”，但要**可解释、可调参、可通过样本回归测试**。

索引存储：

* 记录 SoftBreak 的全局 char offset（Long/Int），delta 编码压缩到文件
* 也可用 BitSet，但 delta 编码更省内存且易持久化

# 4. 文本存储：Utf16TextStore（可随机访问、支持并发读）

目标：给分页/搜索/摘录提供高性能随机读取，不持有全文字符串。

## 4.1 存储格式

* 文件：`content.u16`（UTF-16LE，无 BOM）
* `lengthChars = fileSizeBytes / 2`

## 4.2 读取 API（建议）

```kotlin
class Utf16TextStore(private val channel: FileChannel) : Closeable {
  val lengthChars: Long

  fun readChars(start: Long, count: Int): CharArray
  fun readString(start: Long, count: Int): String
  fun readAround(center: Long, before: Int, after: Int): String
}
```

关键点：

* 用 `channel.read(buffer, positionBytes)`（带 position 参数）避免共享 position，支持并发 prefetch/search
* start/count 需要做 **边界裁剪** 和 **代理项(surrogate)安全**：

  * 如果 start 落在 surrogate pair 中间，向前/向后修正 1 个 char
* 用 direct ByteBuffer 降低拷贝
* 大部分读都控制在几十 KB 级别（页级）

# 5. 分页：StaticLayout 驱动的按需分页 + 可持久化 PageMap

## 5.1 分页的一致性要求

分页必须完全由以下输入决定：

* `RenderConfig.ReflowText`（字号、行高、段距、padding、字体…）
* `LayoutConstraints`（viewport/density/fontScale）
* 你的 SoftBreak 渲染策略（空格/零宽/保留换行）

把它们 hash 成一个 `PaginationProfileId`，并作为：

* 页缓存 key
* PageMap 文件名的一部分
* 失效判断依据

## 5.2 TxtPaginator.pageAt(startOffset) 的实现要点

流程（核心是“**读一段 → 生成可显示文本 → StaticLayout测量 → 得到页尾 charEnd**”）：

1. sanitize startOffset（0..lengthChars，surrogate 安全）
2. 读取窗口：例如 `windowChars = 48_000`（根据字体大小动态调）
3. 应用 SoftBreak 替换（只替换 `\n` 的显示字符，不改长度）
4. 创建 `StaticLayout`：

   * width = viewportWidth - padding*2（转 px）
   * `setLineSpacing(extra, multiplier)`
   * `includePad=false`
   * hyphenation 按 config 开关
5. 找到高度内最后一行：

   * 遍历 lineBottom，找到 `bottom <= contentHeightPx`
   * `endInWindow = layout.getLineEnd(lastLineIndex)`
6. 得到：

   * `pageStart = startOffset`
   * `pageEnd = startOffset + endInWindow`
7. 输出 `PageSlice(start, end, text, mapping, metrics)`

> 专业点：`pageEnd` 建议做“段内修正”，避免把页面截在 `\n` 的前后导致视觉空行。常见策略：若页尾落在换行附近，尝试向前回退到更自然的断点（但不要过多回退导致页内内容太少）。

## 5.3 prev 的专业实现（不依赖全量分页）

你不能只用 “currentStart - window” 就简单 layout，因为会不稳定。推荐：

* 维护一个 `avgCharsPerPage`（滚动更新）
* prev 时：

  1. 估算 anchor = max(0, currentStart - avgCharsPerPage * 2)
  2. 从 anchor 开始不断 `pageAt(x)` 前向分页，直到某一页的 `end >= currentStart`
  3. 那一页的 `start` 就是 prevStart

这在没有全量 PageMap 时非常实用，且体验稳定。

## 5.4 PageMap（强烈推荐：持久化分页索引）

为了：

* 进度条跳转更精准（按页而不是按 char）
* 显示 “currentPage/totalPages”
* 下次打开秒开分页结果

实现：

* `PageMap` 存 `LongArray pageStarts`（每页 start offset）
* 二进制存储，delta + varint 压缩（非常省空间）
* 构建策略：

  * 前台：按需构建（用户翻到哪算到哪）
  * 后台：空闲时 WorkManager 把整本书分页跑完（可暂停/可取消）
* 命中策略：

  * profile 完全一致则加载
  * 不一致则丢弃（或做近似迁移，但不建议）

# 6. ReaderController：TxtController 的并发与缓存策略

## 6.1 控制器内部建议

* 一个 `CoroutineScope(SupervisorJob + dispatcher)`，session close 时 cancel
* 双 dispatcher：

  * IO：读 store
  * Default：layout 测量（StaticLayout 构建也可放 Default）
* LRU 页缓存：

  * 缓存 `RenderPage`（或至少缓存 `PageSlice(start,end,text)`）
  * 典型大小：3~7 页（当前+前后邻居）
* 预取：

  * `prefetchNeighbors(n)`：并行算 next/prev 的 pageAt 并放进缓存
  * 与你的 `RenderPolicy.prefetchNeighbors` 对齐

## 6.2 state 更新策略（避免 UI 抖动）

* `render()`：

  * 若缓存命中：快速返回，并 `RenderMetrics(cacheHit=true)`
  * 否则：测量 + 更新 state + 发 event
* `next()/prev()`：

  * 先算目标页 slice
  * 再一次性更新 `RenderState(locator, progression, nav, config)`，最后 emit `PageChanged`

# 7. TextMapping：TXT 最简单但要“正确且 Long 化”

页面级 `TextMapping` 很简单，但要注意全局 offset 是 Long：

* locatorAt(charIndex: Int)：

  * global = pageStart + charIndex
  * return `Locator("txt.offset", global.toString())`

* rangeFor(startChar, endChar)：

  * start = pageStart + startChar
  * end = pageStart + endChar
  * return LocatorRange

* charRangeFor(range):

  * parse range.start/end -> Long
  * 若都在 `[pageStart, pageEnd]`，返回 IntRange

> 额外建议：给 locator extras 写入 `profileId`（分页 profile）是有价值的（可选）。但真正稳定的仍是 `txt.offset`。

# 8. 搜索：默认流式扫描 + 可选 Trigram 索引（专业版）

你的 `SearchProvider.search()` 返回 `Flow<SearchHit>`，这非常适合“先出结果再继续”。

## 8.1 默认实现：Streaming substring scan（必须做到可取消）

* 支持 `SearchOptions.startFrom`
* 支持 `caseSensitive`
* 对英文可用 Boyer-Moore，泛用用 KMP 也行
* 不要一次读全书：用滑动窗口扫描（例如每次读 256KB chars）
* 处理跨窗口匹配：窗口尾部保留 `query.length - 1` 的 overlap
* 每命中一次，构建 `SearchHit`：

  * `range = LocatorRange(txt.offset start..end)`
  * `excerpt`：从 hit 附近读 `±N chars`，应用 SoftBreak 显示策略（更符合用户看到的内容）
  * `sectionTitle`：若有章节索引，按 offset 查最近章节标题

专业细节：

* 每扫描若干 KB `ensureActive()`，保证取消及时
* `maxHits` 达到后停止
* 如果 query 很短（1~2），命中可能非常多：要有节流/上限与提示

## 8.2 可选加速：TrigramIndex（推荐给“真正专业”的 TXT 搜索）

如果你追求“几秒内搜完整本百 MB 小说”，建议做三元组索引：

* 预处理时（WorkManager 后台）：

  * 对规范化文本做 trigram（连续 3 个 UTF-16 char）
  * `map<trigram, positions[]>`
  * positions 用 delta+varint 压缩写盘
* 搜索时：

  * 若 query.length >= 3：

    * 把 query 分成 trigrams
    * 取每个 trigram 的 positions 列表做交集（偏移对齐）
    * 再回读文本验证（避免误报）
  * 若 query.length < 3：fallback streaming scan

优点：

* 语言无关（中文也好用）
* 子串搜索天然支持（FTS 不擅长）

缺点：

* 索引构建与存储更复杂（但这正是“专业版”的价值）

# 9. OutlineProvider：章节识别（启发式 + 可控误报）

TXT 没有结构，你只能启发式。

实现建议：

* 基于 `LineIndex` 逐行扫描（避免 regex 扫全字符串）
* 一组 pattern：

  * 中文：`^第[零一二三四五六七八九十百千万0-9]{1,9}[章节回卷部].{0,30}$`
  * 英文：`^(Chapter|CHAPTER)\s+\d+.*$`、`^(Prologue|Epilogue)$`
* 过滤误报：

  * 同一标题重复出现过多（跳过）
  * 标题间距过小（例如 < 200 chars）可能是目录页/广告
* 输出 OutlineNode：

  * title = 行文本（trim）
  * locator = txt.offset（该行起始 offset）
* 缓存 chapters（`chapters.json` 或二进制）

capabilities：

* 若章节数 >= 某阈值（例如 3），`outline=true`，否则 false

---

# 10. TextProvider：复制/分享/摘录要稳定

* `getText(range)`：从 store 读 `[start, end)`，注意：

  * 限制最大长度（比如 200k chars）防止内存炸
  * 返回前把 SoftBreak 的显示替换应用一次（用户复制看到的应该接近阅读看到的）
* `getTextAround(locator)`：center ±N chars，并做边界/代理项修正

# 12. 缓存与资源管理：必须“可控 + 可清理”

## 12.1 多进程/并发打开

* 同一本书同时打开（或快速重复打开）会竞争缓存构建：

  * 用文件锁（`FileChannel.lock()`）或 `Mutex` + “lock 文件”方案，保证只构建一次
  * 其他调用等待或复用

## 12.2 清理策略

* `cacheDir`：可随时删（content.u16 也可放 cache，但重建成本高）
* `dataDir`：更持久（分页索引、搜索索引），可在设置里提供“清理索引/缓存”

推荐：

* `content.u16` 放 `dataDir`（重建代价大）
* `pagemap`、`trigram` 放 `dataDir`
* 临时测量缓存放内存 LRU

# 13. 与 runtime 的协作点（你现有设计几乎不用改）

* `TxtEngine` 注册到 `EngineRegistry`（app 多绑定）
* `DefaultReaderRuntime` 会：

  * formatDetector 探测 TXT
  * engine.open
  * document.createSession(initialLocator, initialConfig)
* TXT 引擎只要正确实现 api 即可，feature 永远不关心实现细节

> 注：你现在的 `RenderDefaults.configFor()` 对 TXT 会返回 `ReflowText()`，OK。

# 14. 你应该把哪些东西放到 engines/engine-common（非常关键）

为了保证 TXT/EPUB 一致性与可维护性，我建议把以下抽到 `engines/engine-common`：

* `TextPaintFactory`（字体、字号、fontScale、粗细、lineHeight 统一）
* `StaticLayoutMeasurer`（给定 text + width + config + constraints -> layout 结果）
* `PaginationProfile`（hash/序列化）
* `LruPageCache`（线程安全）
* （可选）`DeltaVarintCodec`（索引压缩）

这样 TXT/EPUB 的 reflow 引擎不会各写一套“测量分页”，后期维护成本会低很多。
