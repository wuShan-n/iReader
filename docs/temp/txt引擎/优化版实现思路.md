下面给你一套**我最推荐**、并且能和你现有 `core/reader/api + engines/*` 体系无缝对齐的 **TXT 引擎实现思路**。我会按“能直接落地”的角度讲：**怎么打开、怎么分页、怎么定位、怎么搜索、怎么做缓存和性能**，以及你应该把代码放在哪些类里。

# 1) TXT 引擎的“正确定位”：它本质是 Reflow 文本引擎

TXT 没有固定版式（不像 PDF），所以 TXT 引擎的核心不是“渲染”，而是：

* **文本解码与规范化**（encoding + newline）
* **稳定定位（Locator）**：能保存进度、能跳转、能标注
* **分页（pagination）**：给定 `LayoutConstraints + RenderConfig`，算出 page breaks
* **轻量渲染（page content）**：按页给出文本段落/范围
* **搜索 /（可选）章节识别**：OutlineProvider

你现有 `engines/txt` 的目标是实现 `core/reader/api` 的 `ReaderEngine/ReaderDocument/ReaderSession/ReaderController` 等即可，**feature 永远只通过 runtime 调用**。

# 2) 我最推荐的“分页与渲染一致性”方案（关键选择）

TXT 最容易踩坑的是：**引擎用 A 算分页，UI 用 B 渲染文本**，最终导致“分页算出来能放下，但 UI 实际渲染溢出/留白”。

所以我给你两条路线，你只需要选其一；我会标出我更推荐哪条。

## 路线 A（更推荐，稳定）：引擎与 UI 都走 Android `StaticLayout`

* 引擎内部使用 `android.text.StaticLayout` 来计算换行与分页。
* UI 渲染文本页时，也用同样的 `StaticLayout`（比如在 `AndroidView` 里用一个 `TextPageView` 渲染）。
* 优点：**分页=渲染 100%一致**，定位映射（y->offset）也很好做，性能稳定。
* 缺点：阅读页里 TXT 会用到 `AndroidView`（但你 EPUB 可能也会用 WebView，本来就会混合）。

> 这是我最推荐的路线：**准确、少坑、工程成本低**。

## 路线 B（Compose 纯血）：引擎分页用 Compose TextMeasurer，UI 用 Compose Text

* 引擎分页依赖 Compose 的 text layout 能力（TextMeasurer/Paragraph）。
* UI 用同样的字体解析/密度配置去渲染。
* 优点：纯 Compose。
* 缺点：引擎更“UI 依赖”，测试与后台线程使用要更小心；不同设备字体 fallback 细节更容易出差异。

如果你没强制“必须纯 Compose”，**选路线 A**，你会省掉 80% 的分页一致性问题。

下面我按**路线 A**讲一套完整实现（路线 B 只是替换布局测量器即可）。

# 4) 打开 TXT：解码 + 规范化（一定要做，不然定位会乱）

## 4.1 解码（EncodingDetector）

最实用的策略（不用上来就引入大依赖）：

1. **先看 BOM**

   * UTF-8 BOM
   * UTF-16LE/BE BOM
2. 没 BOM：先尝试 UTF-8

   * 如果 decode 出现大量替换字符 `�`（U+FFFD），判定失败
3. 失败后：提供几个常用 fallback（按你用户群）

   * 中文用户常见：GBK / GB18030
   * 兜底：ISO-8859-1（不推荐，但能“显示点东西”）

> 最好在 `OpenOptions` 里支持 `preferredCharset`（用户手动选择编码时能重开文档）。

## 4.2 规范化（Normalization）

打开后立刻做一次“规范化”，并保证定位以规范化后的文本为准：

* 换行统一：`\r\n`、`\r` → `\n`
* 删除开头可能存在的 BOM 字符（有些文件会把 BOM 当成正文）
* 统一 Tab：`\t` → 若干空格（比如 4 个），否则排版会很怪
* 可选：去掉过多的控制字符（0x00-0x08 等），避免布局器异常

最终得到：

```kotlin
data class TxtContent(
  val text: String,          // 规范化后的全文
  val originalLength: Int,   // 可选：用于调试
  val charset: Charset
)
```

# 5) Locator 设计：我推荐“全局字符偏移 + snippet 锚点”

TXT 的最佳 Locator（够稳定、够快、易实现）：

* 主定位：**全局字符偏移 `offset`（0..text.length）**
* 辅助：`progression = offset / text.length`
* 防止文件轻微变化导致偏移失效：保存一个**snippet 锚点**（比如 offset 附近 32~64 个字符的片段 + 片段前后少量上下文哈希）

### Locator 示例（概念）

* scheme: `txt-offset`
* locations:

  * `position = offset`
  * `progression = ...`
* text:

  * `before/after/snippet`（你 core/model 的 Locator 若已有相关字段就用，没有就放到 “extras”）

### 重定位策略（打开时发现文件变了）

当 saved offset 超出范围或文件更新：

1. 先 clamp 到 `[0, text.length]`
2. 如果有 snippet：在 `text` 里 `indexOf(snippet)` 找最近位置（可限定搜索窗口）
3. 找不到就退回 progression

这套能让“文件被加了几行广告/前言”时，用户进度仍然大致正确。

# 6) 分页（Pagination）：用 StaticLayout 做“chunk 分页”，别整本 layout

TXT 最重要的性能点：**不要对整本书创建一个 StaticLayout**（大文本会非常慢且占内存）。推荐“chunk 分页 + 懒加载 + 预取”。

## 6.1 你要输出什么：PageMap

分页的产物我建议是：

```kotlin
data class PageMap(
  val pageStarts: IntArray,  // 每页的 startOffset（全局 offset）
  val configHash: Long,      // RenderConfig+Constraints 的 hash
  val textLength: Int
) {
  fun pageIndexForOffset(offset: Int): Int // 二分查找
  fun startOffsetOf(pageIndex: Int): Int
  fun endOffsetOf(pageIndex: Int): Int     // 下一页 start 或 textLength
}
```

> `pageStarts` 用 IntArray 很省内存：1万页也就 40KB。

## 6.2 Chunk 分页算法（核心思路）

给定 `startOffset`，取一个足够大的文本片段（chunk，比如 20k~50k 字符），对它创建 StaticLayout，然后从 layout 的 line 信息里切出多个页。

伪代码（关键逻辑）：

```kotlin
class TxtPaginator(
  private val fullText: String,
  private val paint: TextPaint,
  private val widthPx: Int,
  private val heightPx: Int,
  private val spacingAdd: Float,
  private val spacingMult: Float,
) {
  fun paginateChunk(chunkStart: Int, chunkSize: Int): IntArray {
    val end = min(chunkStart + chunkSize, fullText.length)
    val chunk = fullText.substring(chunkStart, end)

    val layout = StaticLayout.Builder.obtain(chunk, 0, chunk.length, paint, widthPx)
      .setLineSpacing(spacingAdd, spacingMult)
      .setIncludePad(false)
      .build()

    val pageStarts = ArrayList<Int>(16)
    pageStarts += chunkStart

    var pageStartLine = 0
    val lineCount = layout.lineCount

    for (line in 0 until lineCount) {
      val top = layout.getLineTop(pageStartLine)
      val bottom = layout.getLineBottom(line)

      if (bottom - top > heightPx) {
        // 新页从当前 line 开始
        val newPageStartInChunk = layout.getLineStart(line)
        val newPageStart = chunkStart + newPageStartInChunk

        // 防止死循环：至少要前进 1 个字符
        if (newPageStart > pageStarts.last()) {
          pageStarts += newPageStart
          pageStartLine = line
        } else {
          pageStarts += (pageStarts.last() + 1)
          pageStartLine = line
        }
      }
    }

    return pageStarts.toIntArray()
  }
}
```

### 这套算法的特点

* 一次 layout 能切出**多页**，而不是“一页一个 layout”，速度明显更好
* 懒加载：你只对当前附近的 chunk 做 layout
* 配合预取：翻页体验会很顺

## 6.3 如何把 chunk 分页拼成全书 PageMap？

思路：维护一个可增长的 `IntArray`（或 IntArrayList），不断把新的 pageStart 追加进去。

* 初始只算从 `offset=0` 开始的几块 chunk（比如覆盖前 50 页）
* 用户跳转到中间 offset 时：先用 `progression` 估算一个 offset，再从该 offset 往前对齐到最近已知 pageStart，然后向后扩展，直到覆盖目标 offset

> 你在 `core/reader/runtime` 已经有 PrefetchPolicy，这里很容易做成“确保 [current-2, current+6] 页存在”。

## 6.4 分页一致性：必须把 RenderConfig 的排版参数纳入 hash

凡是会影响换行/高度的参数，都要进 `configHash`：

* 字体大小、字重、字体 family
* 行距（spacingAdd / spacingMult）
* 段间距（如果你做）
* 页边距（影响可用宽高）
* 语言相关（如果你做了标点挤压/禁则）

一旦 configHash 变化：PageMap 全部失效（或标记为旧的，后台重新算）。

# 7) RenderPage 设计：页内容只给“文本范围 + 样式信息 + 映射数据”

TXT 引擎渲染页不需要给 bitmap（没必要），推荐返回：

* `range: IntRange`（startOffset..endOffset）
* `text: String`（该页 substring）
* `locatorStart/locatorEnd`
* `textMapping`（可选，但强烈建议为标注/选择做）

### 7.1 TextMapping 怎么做（用于点按/选择/高亮）

如果你希望 TXT 支持标注与选择，最实用的方法是：

* 每页渲染时创建一个 StaticLayout（只针对该页文本）
* 用 `layout.getLineForVertical(y)` + `layout.getOffsetForHorizontal(line, x)` 做坐标到字符 offset 的映射
* 把映射逻辑封装成你 core/api 的 `TextMapping`（不要直接暴露 StaticLayout）

> 也就是说：分页可以 chunk-layout，渲染页可以 page-layout（小 layout，开销可控）。

# 8) SearchProvider：直接全文 indexOf 就够好用（先别过度工程）

TXT 搜索最推荐的 MVP：

* 全文 String 搜索：`indexOf(query, start, ignoreCase)`
* 逐个 match 产生 `Locator(offset)`，用 Flow 发射（避免一次性返回巨大列表）
* 支持“向前/向后找下一处”比“列出所有结果”更实用

你可以先做：

* `search(query, fromLocator, direction)`：返回下一处
  然后再扩展：
* `searchAll(query)`：Flow<SearchHit>

性能上：几 MB 的小说，这完全够用。

# 9) OutlineProvider：TXT 没目录，但可以“可选自动识别章节”

最常见的自动目录生成规则（足够提升体验）：

* 英文：`^\\s*(chapter|CHAPTER)\\s+\\d+`
* 中文：`^\\s*第[一二三四五六七八九十百千万0-9]+章`
* 卷/部：`^\\s*第[一二三四五六七八九十百千万0-9]+卷`

实现方式：

1. 对全文按行扫描（或按 `\n` 分割但注意内存）
2. 匹配到章节行时记录 offset
3. 生成 `OutlineNode(title, locator=offset)`

**建议把它做成 capability：**

* 默认关闭或“智能开启”（文本长度超过某阈值才做）
* 识别结果缓存到 DB（bookId -> outline json），避免每次打开都扫一遍

# 10) 缓存与并发：TXT 引擎的体验好坏就在这里

## 10.1 建议两级缓存

* `PageMapCache`：按 `configHash` 缓存 `pageStarts`（session 生命周期内）
* `PageLruCache<pageIndex, RenderPage>`：缓存最近 N 页（比如 8~20 页）
* `ChunkLayoutCache<chunkStart, chunkResult>`（可选）：避免来回翻页重复 chunk 分页

## 10.2 协程与取消

* 分页、搜索必须可取消：用户拖动进度条时会频繁触发
* `flatMapLatest` 或 session 内维护一个 `Job`，新请求进来取消旧的

# 11) 大文件策略（别一开始就被“超大 TXT”拖死）

TXT 可能被用户拿来打开日志/超大文本。建议做一个“分级策略”：

* `<= 10~20MB`：全文读入 `String`（体验最佳，工程最简单）
* `> 20MB`：提示“文件过大，建议导入为…/开启大文件模式”

  * 大文件模式可选：

    * 用 byte offset 建索引，只解码需要的窗口（实现复杂）
    * 或直接禁止分页式阅读，改成滚动式（如果你产品允许）

我建议先把主路径做好（小说 TXT），大文件模式后加。

# 13) 最小可用闭环（推荐你按这个顺序实现）

1. **打开 + 解码 + 规范化**（能显示全文）
2. **Locator = offset**（能保存进度、能跳转）
3. **分页 PageMap（懒分页）**（翻页顺畅）
4. **渲染页（substring）+ LRU cache**（不卡）
5. **SearchProvider（next match）**
6. （可选）Outline 自动章节
7. （可选）TextMapping（点按定位、标注范围）

## 最终总结（给你一句“实现标准”）

> TXT 引擎最推荐的做法是：
> **用“规范化后的全文 String”作为基准，Locator 用全局字符 offset 表示；分页用 StaticLayout 做 chunk-layout 得到 PageMap；渲染页按 PageMap substring 输出，并用 page-layout 提供坐标映射与高亮；搜索用 indexOf 流式返回。**
