## 1) TxtController：把「links」和「decorations」缓存拆开（高收益，立竿见影）

### 现状问题

你现在 `pageExtrasCache` 的 key 里包含 `annotationRevision`，导致：

* **每次批注变更都会让 links 也失效**（其实 links 和 annotations 无关），从而频繁重跑正则。
* `pageExtrasFor()` 在渲染锁内做链接检测 + decorations 查询，属于**重活 + 锁内执行**的典型卡顿来源。

### 优化目标

* `links` 缓存只按 `(startOffset, endOffset)`，**批注变更不清**；
* `decorations` 缓存按 `(startOffset, endOffset, annotationRevision)`，批注变更只清 decorations；
* 维护一个 `hasAnyAnnotations`，当没有任何批注时 **直接跳过 range query**（大量页面=0批注时省非常多）。

### 增量改动（在 TxtController 内）

新增 key + 两个 cache + 小工具：

```kotlin
private data class PageRangeKey(val startOffset: Long, val endOffset: Long)
private data class DecorKey(val startOffset: Long, val endOffset: Long, val rev: Long)

private val pageLinksCache = LruCache<PageRangeKey, List<DocumentLink>>((maxPageCache * 3).coerceAtLeast(8))
private val pageDecorCache = LruCache<DecorKey, List<Decoration>>((maxPageCache * 3).coerceAtLeast(8))

private var hasAnyAnnotations: Boolean = false

private inline fun <K, V> LruCache<K, V>.getOrPut(key: K, producer: () -> V): V {
    val cached = this[key]
    if (cached != null) return cached
    val v = producer()
    this[key] = v
    return v
}
```

在你现有的 `observeAnnotationChangesIfNeeded()` 里，只清 decorations，并记录是否有批注：

```kotlin
provider.observeAll().collect { list ->
    mutex.withLock {
        annotationRevision++
        hasAnyAnnotations = list.isNotEmpty()
        pageDecorCache.clear()
        // pageLinksCache 不动
    }
}
```

在 `setLayoutConstraints / setConfig / invalidate(CONFIG/LAYOUT)` 里清两份缓存（因为页边界 start/end 会变）：

```kotlin
pageLinksCache.clear()
pageDecorCache.clear()
```

然后在 `pageExtrasFor(...)` 里改成分开取：

* links：`pageLinksCache.getOrPut(PageRangeKey(start,end)) { LinkDetector.detect(...) }`
* decorations：当 `annotationProvider==null || !hasAnyAnnotations` -> empty；否则 `pageDecorCache.getOrPut(DecorKey(..., annotationRevision)) { query }`

> 这个改完之后：用户做高亮/删除批注时，不再导致 links 重跑；而且无批注的书，decorations 查询直接不走，渲染明显更稳。

## 2) TxtController：Page-map completion 任务减少锁占用 + 避免污染 page cache

### 现状问题

`pageCompletionJob` 会在后台推进分页索引，但它在 `mutex` 内：

* 一次 batch build 多页（默认 4），容易在用户翻页时产生锁争用；
* `allowCache=true` 会把后台推进的页写进 `sliceCache`，**挤掉用户正在看的页**（maxPageCache 默认 7 时更明显）。

### 建议改动

1. completion 里构建 slice 时用 `allowCache = false`（后台索引不要污染用户 cache）
2. batch 改成更小（1 或 2），并在批次之间 `yield()`，给前台 render 抢锁机会

#### 增量修改点

* 在 `completePageMapBatchLocked(...)` 里把：

```kotlin
allowCache = true
```

改成：

```kotlin
allowCache = false
```

* 在 `completePageMapForward(...)` 的 while 循环每轮 batch 后加：

```kotlin
import kotlinx.coroutines.yield
...
yield()
```

* 你现在 `maybeSchedulePageCompletionLocked()` 传入的 `batchSize = PAGE_COMPLETION_BATCH_SIZE`，建议把 `PAGE_COMPLETION_BATCH_SIZE` 降到 1~2（我更偏向 1，完全避免一次持锁太久）。

> 这套改法对“翻页时偶发卡顿/掉帧”的改善通常非常明显，而且风险极低（只影响后台索引速度）。

## 3) SoftBreakIndexBuilder：两个“隐藏大头”——每字符 ensureActive + 每换行创建 classifier context

这块在大书上是纯 CPU 热点。

### 3.1 把 `ensureActive()` 从“每字符”降到“每 N 字符”

你现在在 `for (c in chunk)` 内每个字符都 `ensureActive()`，这非常贵。

改成按步进检查（比如 4096）：

```kotlin
val baseOffset = globalOffset
for (i in chunk.indices) {
    if ((i and 0xFFF) == 0) coroutineContext.ensureActive() // 每 4096 字符
    val c = chunk[i]
    val offset = baseOffset + i
    ...
}
globalOffset += chunk.size.toLong()
```

同时把 `pending = Pending(newlineOffset = globalOffset, ...)` 改成 `newlineOffset = offset`（用上面 offset）。

### 3.2 复用 SoftBreakClassifierContext（避免每换行分配对象）

你现在每次 `isSoftBreak(line0, line1)` 都 new 一个 `SoftBreakClassifierContext(...)`。
对于几十万/上百万换行，这就是巨量对象 + 构造开销。

做法：典型长度不是每行都要更新，按窗口更新即可：

```kotlin
var cachedTypical = 72
var cachedContext = SoftBreakClassifierContext(
    typicalLineLength = cachedTypical,
    hardWrapLikely = meta.hardWrapLikely,
    rules = ruleConfig
)
var updateCountdown = 0

fun isSoftBreak(line0: SoftBreakLineInfo, line1: SoftBreakLineInfo): Boolean {
    if (updateCountdown-- <= 0) {
        val typical = estimateTypicalLineLength()
        if (typical != cachedTypical) {
            cachedTypical = typical
            cachedContext = SoftBreakClassifierContext(
                typicalLineLength = cachedTypical,
                hardWrapLikely = meta.hardWrapLikely,
                rules = ruleConfig
            )
        }
        updateCountdown = 256 // 每 256 行更新一次
    }
    return SoftBreakClassifier.classify(line0, line1, cachedContext).isSoft
}
```

> 两个改完，soft-break index 构建速度通常会有明显提升，且 GC 压力显著下降。

## 4) EncodingDetector：把 UTF-8 校验改成“零分配状态机”（避免 CharBuffer allocate + trim copy）

### 现状问题

你现在 `checkUtf8()`：

* `trimUtf8LeadingContinuationBytes()` 会 `copyOfRange`（最多拷贝 128KB 的数组）；
* 用 `CharsetDecoder` + `CharBuffer.allocate(trimmed.size)`，每 sample 都会分配大对象。

### 建议：手写 UTF-8 验证状态机

核心目标：

* 不拷贝 bytes（只算一个 startIndex）
* 不创建 CharBuffer / Decoder（纯扫描）

你只需要替换 `checkUtf8()` + `trimUtf8LeadingContinuationBytes()` 逻辑为：

* `trimStartIndex(bytes): Int` 返回 start index（0~3）
* `validateUtf8(bytes, startIndex): Boolean` 做严格合法性校验（处理 E0/ED/F0/F4 的边界，避免 overlong、代理区、>U+10FFFF）

**注意**：你原逻辑 `utf8.all(valid) && utf8.any(hasNonAscii)` 才判定 UTF-8。
我建议改成：

* `all(valid)` => **优先 UTF-8**
* 如果 `none(nonAscii)`（纯 ASCII）也用 UTF-8，但 `confidence` 降一点（比如 0.7），否则仍可能落到 scorer 产生不一致的 meta.originalCharset。

这块改动对打开书时的瞬态内存峰值很友好，尤其在 Android 低端机上。

## 5) cache 正确性：sampleHash 只取头 64KB 容易“误命中旧缓存”（建议升级为多窗口采样）

### 现状风险（不是性能，而是专业级正确性）

`computeSampleHash()` 只 hash 前 64KB：

* 文件尾部被改动（追加/替换）但头部不变时，**缓存会误判有效**；
* 后续 content.u16、bloom、softbreak、outline 都可能复用旧结果（甚至 lengthChars 也可能没变）。

### 建议：多窗口采样 hash（头 + 中 + 尾）

策略：

* 若 `sizeBytes` 可得且大于 `3 * chunk`：读 3 段（0、mid、tail）更新 digest
* 若 size 不可得：保留头部 hash（fallback）

实现上可复用你已有的 `readSampleAt`（偏向 fd 读取），把 digest.update 三次即可。

> 这类改动很“非 MVP”，但对线上缓存一致性非常关键，尤其你用的是 `sourceUri + size + displayName + sampleHash` 生成 documentId。

## 6) TxtOpener：meta.json / outline.json 写入也建议原子化（减少损坏概率）

你 content.u16 用了 `replaceFileAtomically`，但 `metaJson.writeText(...)`、`outlineJson.writeText(...)` 仍然可能在崩溃/断电/kill 时留下半文件。

做法：

* 写到 `*.tmp`，再 `replaceFileAtomically(tmp, target)`
* 同时把 `tryLoadCached()` / `loadFromCache()` 的 JSON parse 用 `runCatching` 包起来，解析失败当 cache miss（你上次已经走向这个方向）

这会把“偶发打不开书/报错”降到“自动重建缓存”。

## 7) LinkDetector：加 cheap prefilter，避免对绝大多数页面跑 regex

页面大多数没有链接。regex 即使不匹配也要扫描。

在 `detect()` 里 `source = text.toString()` 后加：

```kotlin
val maybeUrl = source.indexOf("http", ignoreCase = true) >= 0 ||
    source.indexOf("www.", ignoreCase = true) >= 0
val maybeEmail = source.indexOf('@') >= 0
if (!maybeUrl && !maybeEmail) return emptyList()
```

然后：

* `maybeUrl` 才跑 urlRegex.findAll
* `maybeEmail` 才跑 emailRegex.findAll

> 这是非常“划算”的优化：几乎零风险，但能减少很多无意义扫描。

## 8) TxtSelectionManager：补齐 progression extras（你现在拿 startLocator.extras 基本是空的）

你现在 selection extras 里：

```kotlin
LocatorExtraKeys.PROGRESSION to startLocator.extras[LocatorExtraKeys.PROGRESSION].orEmpty()
```

但 `startLocator` 是你新建的 locator，它根本没 progression。

建议在 buildSelection 里按 `startOffset / store.lengthChars` 直接计算：

```kotlin
val percent = if (store.lengthChars == 0L) 0.0 else startOffset.toDouble() / store.lengthChars.toDouble()
val extras = mapOf(
    LocatorExtraKeys.PROGRESSION to String.format(Locale.US, "%.6f", percent),
    "selectionStartOffset" to startOffset.toString(),
    "selectionEndOffset" to endOffset.toString()
).filterValues { it.isNotBlank() }
```

这会让“复制/分享/批注锚点”等依赖 progression 的上层逻辑更稳定。

## 9) TxtTextProvider.getTextAround：给 maxChars 做 hard cap（防止上层误用导致 OOM）

现在 `maxChars` 理论上可以非常大。建议加上：

```kotlin
private const val MAX_AROUND_CHARS = 200_000
...
val capped = maxChars.coerceIn(32, MAX_AROUND_CHARS)
val half = (capped / 2).coerceAtLeast(16)
```

这属于线上防御型优化。

## 10) Outline 检测（可选但专业）：避免 `carry + chunk` 的大字符串拼接

`TxtOutlineProvider.detectOutline()` 现在每 chunk `merged = carry + chunk` 会复制字符串，书越大越浪费。

更专业的写法是：

* 用 `store.readChars()` 拿 `CharArray`
* 用一个 `StringBuilder line` 流式扫描（遇到 `\n` 结束一行）
* 只在 `detector.isChapterTitle(...)` 需要时才把 builder 转 String

这能把 outline 检测的峰值内存和复制次数降很多。

# 建议你按这个顺序落地（收益/风险比最高）

1. **TxtController：links/decors cache 拆分 + hasAnyAnnotations 跳过查询**
2. **Page completion：allowCache=false + batch=1 + yield**
3. **SoftBreakIndexBuilder：ensureActive 降频 + classifier context 复用**
4. **EncodingDetector：UTF-8 状态机零分配**
5. **多窗口 sampleHash + JSON 原子写**
6. LinkDetector prefilter、Selection progression、TextAround cap（小改动但很稳）
