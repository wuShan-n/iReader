下面按“**不重复你已有代码**、只给**最值的优化点 + 需要改哪些地方（含少量新增/改动片段）**”来讲。
  你当前 TXT 引擎能跑，但主要瓶颈在：

  * **打开阶段**：`readAllBytes -> String -> normalize 二次扫描`，大文件容易内存峰值/GC 抖动
  * **分页阶段**：`ensurePageMap()` 会做“全书分页”，且你现在把重活放在 mutex 里（会卡住其它请求）
  * **渲染阶段**：每页 `substring`/`subSequence` 仍可能产生大量临时 String
  * **取消与预取**：分页计算不响应取消；预取是同步串行，容易拖慢翻页

  我按优先级给你优化方案。

  ---

  ## P0（立刻收益最大，且风险最低）

  ### 1) 打开：改成“采样探测编码 + 流式解码 + 流式规范化”，避免 ByteArray 全量驻留

  **问题**
  你现在是：一次性把全部 bytes 读进内存 -> 再解码成 String -> 再 normalize 扫一遍。
  这会造成**内存峰值≈ 文件字节 + 字符串 + normalize 新字符串**（大 TXT 直接爆）。

  **目标**

  * 编码探测只读**前 64KB**样本
  * 解码用 `InputStreamReader` **流式**读进 `StringBuilder`
  * 规范化也做成**流式**（顺手处理 `\r\n/\r`、`\t`、控制字符、BOM）

  **改动点**
  只需要改 `TxtOpener.open()` 的实现，新增一个 streaming normalizer 函数即可。

  **新增片段（示意）**：

  ```kotlin
  private const val SAMPLE_BYTES = 64 * 1024

  // 只读样本探测编码，不再把全文件读到 ByteArray
  private fun BufferedInputStream.readSample(): ByteArray {
      mark(SAMPLE_BYTES + 8)
      val buf = ByteArray(SAMPLE_BYTES)
      val n = read(buf)
      reset()
      return if (n <= 0) ByteArray(0) else buf.copyOf(n)
  }

  // 流式解码 + 流式规范化：避免二次遍历 & 避免 bytes 全量驻留
  private fun decodeAndNormalize(
      input: InputStream,
      charset: Charset
  ): String {
      val reader = InputStreamReader(input, charset)
      val out = StringBuilder(256 * 1024)

      val buf = CharArray(16 * 1024)
      var prevWasCR = false
      var isFirstChar = true

      while (true) {
          val read = reader.read(buf)
          if (read <= 0) break

          for (i in 0 until read) {
              var ch = buf[i]

              // BOM 只在开头去掉
              if (isFirstChar) {
                  isFirstChar = false
                  if (ch == '\uFEFF') continue
              }

              // \r\n / \r -> \n（注意跨 buffer 边界）
              if (prevWasCR) {
                  prevWasCR = false
                  if (ch == '\n') continue
                  out.append('\n')
              }
              if (ch == '\r') {
                  prevWasCR = true
                  continue
              }

              // tab -> 4 spaces
              if (ch == '\t') {
                  out.append("    ")
                  continue
              }

              // 控制字符过滤（保留 \n）
              if (ch != '\n') {
                  val code = ch.code
                  if (code in 0x00..0x08 || code in 0x0B..0x1F) continue
              }

              out.append(ch)
          }
      }

      // 文件以 \r 结尾的情况
      if (prevWasCR) out.append('\n')

      return out.toString()
  }
  ```

  然后在 `open()` 里：

  * `BufferedInputStream` 包一层，`sample = bis.readSample()`
  * `charset = detector.detect(sample, options.textEncoding)`
  * `text = decodeAndNormalize(bis, charset)`（同一条流即可，不必 reopen）

  这样打开大文件时，**内存峰值明显下降**。

  ### 2) 分页：不要在 Mutex 里做重活；用 “单飞构建任务” 合并并发 + 可取消

  **问题**
  你现在 `ensurePageMap()` 里“拿锁 -> withContext 做分页 -> 出锁”，意味着：

  * render/next/prev/goTo 等都会在锁上排队
  * 用户快速拖动进度条时会触发重复构建
  * invalidate/config/layout 变化时无法及时 cancel 正在构建的分页

  **目标**

  * 同一 `(configHash)` 同时只允许一个构建任务
  * 其它请求 `await` 同一个任务
  * `invalidate/setConfig/setLayout` 立刻 cancel 正在构建任务
  * **构建分页不占用 mutex**

  **改动点**
  在 controller 加一个 `building: Deferred<PageMap>?`（带 hash），改造 `ensurePageMap()`。

  **新增片段（结构示意）**：

  ```kotlin
  private data class Building(val hash: Long, val job: Deferred<PageMap>)
  private var building: Building? = null

  private suspend fun ensurePageMap(): PageMap {
      val spec = buildLayoutSpec()
      val hash = configHash(spec, config)

      // fast path
      pageMap?.takeIf { it.configHash == hash && it.textLength == fullText.length }?.let { return it }

      val job: Deferred<PageMap> = mutex.withLock {
          // 复用正在构建的同 hash 任务
          building?.takeIf { it.hash == hash }?.job
              ?: scope.async(Dispatchers.Default) {
                  // 建议 paginator 里加 ensureActive（下一条）
                  paginator.buildPageMap(fullText, spec, hash)
              }.also { building = Building(hash, it) }
      }

      val built = job.await()

      mutex.withLock {
          // 如果等待期间 config/layout 变了，就别覆盖
          val currentHash = configHash(buildLayoutSpec(), config)
          if (currentHash == hash) {
              pageMap = built
          }
          if (building?.hash == hash) building = null
      }

      return pageMap ?: built
  }
  ```

  并且在 `invalidate/setConfig/setLayoutConstraints` 里补：

  ```kotlin
  mutex.withLock {
      building?.job?.cancel()
      building = null
      pageMap = null
      pageCache.clear()
  }
  ```

  这套改完，**分页不会把 controller 锁死**，并且能**避免重复构建**。

  ### 3) 分页/目录计算支持取消：在分页循环中 `ensureActive()`

  **问题**
  用户改字体/旋转屏幕/快速退出时，分页仍可能在后台跑完才停止，浪费 CPU。

  **目标**
  分页每处理一个 chunk 或一定次数就检查取消。

  **改动点**
  把 `TxtPaginator.buildPageMap()` 改成 `suspend`（或注入一个 `checkCancelled()` 回调），每轮 while 做：

  ```kotlin
  coroutineContext.ensureActive()
  ```

  以及在拆行循环里每 N 行检查一次：

  ```kotlin
  if ((line and 0x1FF) == 0) coroutineContext.ensureActive()
  ```

  收益非常明显：**旋转/改字号不会“旧分页还在烧 CPU”**。

  ---

  ### 4) 预取：不要阻塞当前 render；改成后台并发预热（带并发上限）

  **问题**
  你现在 prefetch 是同步串行调用 `renderPageAt`，并且 render 里还会等待 prefetch 完成（会拖慢首屏/翻页）。

  **目标**

  * render 返回当前页后立刻响应
  * 预取在 `scope.launch` 后台跑
  * 并发上限（2~3）防止抢占太多 CPU

  **改动点**
  把 render 内的预取改成：

  ```kotlin
  if (policy.prefetchNeighbors > 0) {
      scope.launch {
          // 忽略返回值即可，必要时打日志
          prefetchNeighbors(policy.prefetchNeighbors)
      }
  }
  ```

  并在 `prefetchNeighbors` 内部做并发上限（伪代码）：

  ```kotlin
  val sem = Semaphore(permits = 2)
  coroutineScope {
    for (i in from..to) launch {
      sem.withPermit {
        // 若已缓存直接跳过（可在这里先查 pageCache）
        renderPageAt(map, i, allowCache = true)
      }
    }
  }
  ```

  ### 5) 渲染：减少 `substring` 产生的临时 String（引擎侧先把 “页文本” 做成零拷贝 CharSequence）

  **问题**
  每次 `substring` 都会产生新 String（现代 Android/Java 基本都会复制），翻页多了 GC 压力上来。

  **目标**
  引擎内部用一个**视图型 CharSequence** 代表页文本，避免立刻复制。UI 若用 TextView/AndroidView，可直接吃 CharSequence。

  **新增类（很小）**：

  ```kotlin
  internal class TextSlice(
      private val source: String,
      private val start: Int,
      private val end: Int
  ) : CharSequence {
      override val length: Int get() = end - start
      override fun get(index: Int): Char = source[start + index]
      override fun subSequence(startIndex: Int, endIndex: Int): CharSequence =
          TextSlice(source, start + startIndex, start + endIndex)
      override fun toString(): String = source.substring(start, end)
  }
  ```

  然后在 `renderPageAt` 里把：

  * `fullText.subSequence(start, end)`
    改为
  * `TextSlice(fullText, start, end)`

  > 注意：如果你的阅读 UI 最终把 `CharSequence.toString()` 喂给 Compose `Text()`，那复制还是会发生；但如果你 TXT 阅读页
  走 `AndroidView(TextView)`，这个优化会非常有用（并且分页与渲染一致性也更好）。

  ## P1（中等改动，收益大，适合第二轮）

  ### 6) 增加 “reflow.page locator” 支持，让翻页/恢复更快、更稳定

  你 core/model 已经有 `LocatorSchemes.REFLOW_PAGE`。TXT 可以做：

  * 渲染第 `pageIndex` 页时，把 `RenderState.locator` 的 `extras` 填上：

    * `"pageIndex" -> idx`
    * `"pageCount" -> total`
  * 并在 `goTo(locator)` 时支持：

    * `scheme == reflow.page`：直接按页跳转（O(1)）
    * `scheme == txt.offset`：用 offset 跳转（你已有）

  这样做的好处：

  * 未来你做**分页缓存落盘**时，可以直接用 pageIndex 恢复
  * 用户“快速拖进度条”可以按页跳，不必每次 offset->pageIndex 反查

  > 记得把 `breakStrategy/hyphenation/padding/fontSize` 等都纳入 `configHash`，否则 pageIndex 缓存会错。

  ---

  ### 7) `StaticLayout.Builder` 加上断行策略/连字符（API 23+），提升排版质量

  你 config 有 `hyphenation`，现在没用。可以在创建 layout 时加：

  * `setBreakStrategy(Layout.BREAK_STRATEGY_HIGH_QUALITY)`
  * `setHyphenationFrequency(Layout.HYPHENATION_FREQUENCY_NORMAL)`（当 hyphenation=true）

  ⚠️ 这会影响分页结果，所以必须把这些选项纳入 `configHash`（否则会复用错误的 PageMap）。

  ---

  ### 8) annotations 查询从 O(N) 降到近似 O(logN + K)

  目前每页取 decorations 都会扫所有 annotations。少量没问题，但如果用户标注很多会慢。

  做法：在 `TxtAnnotationStore` 内维护一个按 startOffset 排序的 `IndexedAnnotation` 列表：

  * create/update/delete 时同步更新索引
  * query(range) 时：

    * 二分找到第一个 `start < rangeEnd`
    * 向后遍历直到 `start > rangeEnd`

  这样页渲染 decorations 会更稳。

  ## P2（较大重构，但能把体验做成“商业级”）

  ### 9) 分页“懒构建 + 后台补全”

  把 “全书 PageMap” 改成：

  * 首屏只算**当前 chunk**（比如覆盖前 30~60 页）
  * 用户继续翻页时按需扩展 pageStarts
  * 同时后台低优先级补全全书 pageCount（给 UI 完整 “12/320”）

  配合 `Progression.total` 可为 null，你可以先显示百分比，再补全页数。

  ---

  ### 10) 大文件模式（>20MB）：不要强行全量 String

  这个是终局方案：

  * 建一个 “byte offset -> char window” 的解码器（按窗口解码）
  * locator 用 byte offset（或 char offset + 再定位锚点）
  * 分页/搜索都在窗口上做 + 维护索引

  实现复杂，但能支持日志/超大文本。

  优化完成后，你应该看到：

  * render 首页不再被 mutex 阻塞
  * 旋转/改字号能立刻取消旧分页，不再“后台烧 CPU”
