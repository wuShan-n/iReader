下面给你一套**我最推荐、且和你当前模块划分“完全契合”的书架（library）实现方案**。核心目标是：**书架永远以 DB 为真相源**（Source of Truth），文件系统/SAF 只是“来源”，解析引擎只是“能力提供者”，并且严格遵守你写的依赖规则：`feature/library` 永远不碰 `engines/*`。

---

# 0. 一句话总方案

**书架 = “本地索引（Room） + 后台索引器（WorkManager） + 引擎能力（通过 core/reader/runtime 间接使用）”**
UI 只展示索引、触发动作；所有导入/扫描/去重/元数据抽取/封面缓存/丢失检测都在 core 层完成。

---

# 1. 书架功能边界与分层职责

## 1.1 feature/library 应该只做什么

* 展示：网格/列表、分组（合集/标签）、排序筛选、搜索
* 操作：导入、删除、加入合集、改名（仅“显示名”）、标记已读/在读/未读、收藏
* 触发后台：点击“重新解析元数据/重新生成封面/重新定位文件”
* 跳转：打开阅读（导航到 `feature/reader`）

✅ **feature/library 不做：**

* 不直接扫描文件系统（最多调用 usecase）
* 不直接解析 EPUB/PDF/TXT
* 不直接访问 `WorkManager`（可选，最好不碰）
* 不写 SQL 拼接（属于 data 层）

## 1.2 core 层该做什么

* `core/database`：实体、DAO、migrations（不含业务）
* `core/data`：BookRepo/ProgressRepo + “索引构建器/元数据抽取器”的编排（业务都放这里）
* `core/files`：SAF 权限、DocumentSource、目录扫描、文件指纹（hash/size/mtime）
* `core/work`：批量导入、批量索引、丢失检测（可选但强烈推荐）
* `core/reader/runtime`：格式探测 + 通过 EngineRegistry 选择引擎，提供**“无 UI 的 metadata probe”**能力给 core/data/core/work

---

# 2. 最关键的设计选择：DB 是书架真相源（不是实时扫描）

如果书架每次进入都扫描目录，你会立刻遇到：

* SAF 目录扫描慢、权限复杂、厂商实现差异
* 大书库性能崩、列表抖动
* 去重/元数据/封面没法稳定缓存

所以我建议你把书架建成一个**离线索引系统**：

1. 导入/扫描时，把文件登记到 `books` 表（先放“基础信息 + pending 状态”）
2. 后台 worker 批量解析元数据、封面、能力（capabilities）并更新表
3. UI 只订阅 DB：Paging + Flow，永远稳定、可缓存、可增量更新

> 你已经有 `core/work` 和 `core/files`，这个路线与你的结构是“天作之合”。

# 3. 数据模型：我推荐的 Room Schema（够用、可扩展）

下面是“书架索引”最实用的一套表结构（你可以按需裁剪）。

## 3.1 books（书的主表）

**建议用内部 `bookId: Long` 做主键**（避免 URI 变化导致主键不稳定），同时保留一个“外部唯一键”用于去重/重定位。

**关键字段：**

* `bookId: Long` PK
* `documentId: String` UNIQUE（你 core/model 里已有 `DocumentId` 概念，建议序列化成 string）
* `sourceUri: String`（DocumentSource 的 Uri 字符串）
* `sourceType: Int`（SAF/File/Asset…）
* `format: Int`（BookFormat）
* `fileName: String`
* `fileSize: Long`
* `lastModified: Long`
* `fingerprint: String?`（可选：hash/快速指纹，用于去重和重定位）
* `addedAt: Long`
* `lastOpenedAt: Long?`
* `title: String?`
* `authors: String?`（简单存一条字符串，展示够用；更复杂再拆表）
* `series: String?`
* `description: String?`
* `coverPath: String?`（缓存到 app 私有目录，比如 `files/covers/{bookId}.webp`）
* `readingStatus: Int`（UNREAD/READING/FINISHED）
* `favorite: Boolean`
* `indexState: Int`（PENDING/INDEXED/ERROR/MISSING）
* `indexError: String?`（可选）

## 3.2 progress（进度表）

* `bookId: Long` FK UNIQUE
* `locatorJson: String`（序列化你的 `Locator`）
* `progression: Double`（0..1，便于 UI 快速显示）
* `updatedAt: Long`

> 你已有 `ProgressRepo`，直接对齐。

## 3.3 collections / tags（可选但很值）

* `collections(collectionId, name, createdAt, sortOrder)`
* `book_collection(bookId, collectionId)` 多对多
* `tags(tagId, name)`
* `book_tag(bookId, tagId)` 多对多

先做 collections 就够了，tag 可以后加。

## 3.4 搜索（推荐 FTS）

书架内搜索（标题/作者/系列/文件名）建议用 Room FTS（常见是 FTS4）：

* `book_fts`（外部内容表指向 `books`）
* DAO 提供 `MATCH` 查询，配合排序即可

# 4. 书架的“导入/索引”流水线：两阶段导入（强烈推荐）

这是书架成败的关键。

## 4.1 阶段 A：快速登记（UI 立刻可见）

当用户从 SAF 选择文件（或目录扫描发现文件）：

1. `core/files` 负责拿到 `DocumentSource`（Uri + 持久权限）
2. `core/data` 的 `BookRepo.registerSources(sources)`：

   * 读取文件基础属性：name/size/lastModified/mime
   * 探测 format（优先扩展名 + mime，必要时魔数/zip 内判断 —— 你已经有 `DefaultBookFormatDetector`）
   * 插入 `books`（`indexState=PENDING`）
3. 立刻返回 `bookId` 列表
   ✅ UI 能马上看到“正在导入/待解析”的卡片（占位封面 + 文件名）

## 4.2 阶段 B：后台丰富（元数据/封面/能力）

对每本 `PENDING`：

* WorkManager 任务（建议 `core/work`）负责调用一个“索引器”：

  * `BookIndexingService.index(bookId)`
* 索引器做：

  1. 通过 `ReaderRuntime` 打开文档的**轻量模式**

     * 只读取 metadata / cover，不进入重排渲染、不做分页
  2. 更新 `books.title/authors/coverPath/capabilities/indexState`
  3. 失败则 `indexState=ERROR` 并写 `indexError`，UI 可展示“点击重试”

✅ 这样导入体验会非常丝滑：UI 不阻塞、导入越多越稳。

# 5. 引擎如何服务书架（又不破坏依赖规则）

你的硬约束是：`feature/*` 不依赖 `engines/*`。
所以书架要拿元数据/封面，必须走：

**feature/library → core/data → core/reader/runtime → EngineRegistry → engines/**

## 5.1 我建议你在 core/reader/runtime 增加一个“Probe API”

不要让 library 去 open session 正常阅读；它只需要 probe。

示例（伪代码）：

```kotlin
// core/reader/runtime
data class BookProbeResult(
    val format: BookFormat,
    val metadata: DocumentMetadata?,
    val cover: ByteArray?,          // 或者返回一个可读取的 Resource 引用
    val capabilities: DocumentCapabilities?
)

interface ReaderRuntime {
    suspend fun probe(source: DocumentSource): ReaderResult<BookProbeResult>
}
```

实现上：

* 通过 `BookFormatDetector` 得到 format
* 找到对应 engine
* engine 提供一个“轻量读取 metadata/cover”的路径
  （如果你现有 ReaderDocument 打开很重，就新增 OpenOptions 让引擎可选跳过排版/分页）

## 5.2 封面缓存策略（推荐）

* Probe 返回 `ByteArray`（或 InputStream），`core/data` 负责落盘到私有目录
* `books.coverPath` 存本地路径
* UI 用 Coil 加载 `File(coverPath)`（有磁盘缓存更快）

这样封面加载完全不依赖引擎，也不需要 SAF 再读一遍。

# 6. 去重与“文件重定位”：别一开始就做很重的 hash

书架最常见的坑：重复导入 + 文件移动导致条目丢失。

## 6.1 去重推荐策略：分层指纹

我建议你采用“轻到重”的指纹策略（避免导入时卡死）：

* **一级（超快）**：`(sourceUri)` 唯一
  同一个 Uri 重复导入直接 ignore/update

* **二级（快）**：`(fileSize + lastModified + fileName)` 组合
  用于“疑似重复”的提示，不用于强去重

* **三级（慢，后台）**：`contentHash`（比如 sha256，但在 worker 做）
  用于真正的强去重 + 重定位（文件移动后还能找回）

落地方式：

* 导入阶段只做一级/二级
* `IndexWorker` 在后台补算 `fingerprint/hash`，再做强去重合并

## 6.2 重定位（文件丢失）推荐方案

当打开书失败（SAF 失效/文件被移动）：

* 将 `books.indexState=MISSING`
* UI 给“重新定位文件”按钮
* 让用户重新选择文件后：

  * 比对 `hash` 或 `fileSize+mtime+name`
  * 更新 `sourceUri`，保留 `bookId` 和进度/标注

这能极大提升“书库长期使用”的体验。

# 7. 查询与列表：Paging + 动态排序筛选（建议 RawQuery）

书架列表的排序筛选会越来越多（最近阅读、标题、作者、加入时间、进度、已读/未读、合集过滤、搜索关键字…）。

**推荐做法：**

* DAO 提供一个 `@RawQuery` 的 PagingSource
* `core/data` 里做 `LibrarySqlBuilder` 拼 query（这属于业务逻辑，放 core/data 合规）

示意：

```kotlin
@Dao
interface LibraryDao {
  @RawQuery(observedEntities = [BookEntity::class, ProgressEntity::class])
  fun paging(query: SupportSQLiteQuery): PagingSource<Int, BookWithProgress>
}
```

`LibraryQuery`（domain）建议长这样：

* `keyword: String?`
* `sort: SortOption`
* `filterStatus: Set<ReadingStatus>`
* `collectionId: Long?`
* `onlyFavorites: Boolean`

然后 `feature/library` 的 VM：

* 把 UI 的筛选/排序保存到 `core/datastore`
* 合并成 `queryFlow`
* `flatMapLatest { repo.paging(query) }`

# 9. 与阅读页的衔接：用 bookId 作为唯一链路

强烈建议：**书架到阅读页只传 `bookId`**，不要传 uri/metadata。

打开流程：

* Library 点击卡片 → `navigate(ReaderRoute(bookId))`
* `feature/reader` 通过 `BookRepo.getBook(bookId)` 拿到 `DocumentSource`
* `ReaderRuntime.open(source)` → 引擎打开
* 退出阅读页 → `SaveProgress(bookId, locator, progression)`

这样书架条目即使重定位了 uri，阅读页也永远正确。

# 10. 你现在最值得优先实现的“最小闭环”（MVP 顺序）

为了最快做出可用书架，我建议按这个顺序（每一步都可上线）：

1. **DB + BookRepo + Paging 列表**（空书架也要能跑）
2. **SAF 选择文件导入**：只登记 `PENDING`，UI 显示占位卡片
3. **IndexWorker**：后台 probe 元数据 + 落封面 + 更新 `INDEXED`
4. **打开阅读 + 保存进度**：书架卡片显示进度条/在读状态
5. **排序/筛选/书架内搜索**（FTS 或简单 like）
6. **批量导入/目录扫描**（DocumentTree）
7. **丢失检测 + 重定位**
8. **合集/标签**

# 11. 我最推荐你加的两个“防后期返工”接口

这俩做了，后面扩展会非常省：

## 11.1 core/data：单独抽出索引器接口

```kotlin
interface BookIndexer {
    suspend fun index(bookId: Long): Result<Unit>
    suspend fun reindex(bookId: Long): Result<Unit>
}
```

* 实现里用 ReaderRuntime.probe + files + database
* Worker/Usecase 都调用它

## 11.2 core/files：统一文件属性读取

```kotlin
data class FileProps(
  val name: String,
  val size: Long,
  val lastModified: Long,
  val mime: String?
)

interface FilePropsReader {
  suspend fun read(source: DocumentSource): FileProps
}
```

这样未来从 SAF → File path → 云盘 provider 都能平滑替换。
