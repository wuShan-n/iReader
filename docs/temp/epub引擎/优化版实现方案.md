下面给你一套**我最推荐、也最容易在工程上“长期不后悔”**的 EPUB 引擎实现思路：**以 Web 技术栈（HTML/CSS/JS）为渲染核心，以 CFI 做定位，以“Positions/Locator”做全书进度**，并且强烈建议用 **Readium Kotlin Toolkit** 做底座（解析、资源组织、导航、偏好、搜索、标注能力都比较完整），你只需要在 `engines/epub` 里把它“适配”到你自己的 `core/reader/api`。([W3C][1])

---

## 1) EPUB 引擎本质要解决的 6 件事（别上来就写渲染）

EPUB 不是“文件 = 一段文本”，它是一个 **打包的 Web 应用集合**：XHTML + CSS + 图片/字体/SVG + 导航 + 元数据，放在一个容器（zip）里分发。([W3C][1])
所以引擎要解决的关键问题一般是：

1. **打开与解析**：容器 → OPF → manifest/spine → reading order
2. **资源访问**：相对路径/跨文件引用/字体/图片都要能稳定加载
3. **渲染与排版**：Reflow（可重排） vs FXL（固定版式），以及 paginated vs scrolling
4. **定位与进度**：字体/行距/屏幕变了也要能回到“同一段内容”
5. **目录/跳转/书签/标注**：TOC + Range + Decorations
6. **搜索与索引**：按章节/全书搜索，最好还能做全文索引

你现在的模块架构非常适合把这 6 件事全部关在 `engines/epub/internal/*`，对外只暴露 `EpubEngine : ReaderEngine`。

---

## 2) 我最推荐的路线：Readium 做底座 + 你自己的 API 适配层

### 为什么推荐 Readium（工程收益最大）

Readium Kotlin Toolkit 已经覆盖了 EPUB2/EPUB3，并且对“阅读器必备功能”做了比较完整的实现：分页/滚动、RTL、搜索、高亮（Decoration API）、以及阅读偏好设置等。([GitHub][2])
它还提供：

* **EPUB 解析器**（`EpubParser`）以及 **PositionsService**（用于生成“稳定的定位点/页码感知”）。([readium.org][3])
* **Navigator 可动态提交偏好**（字体、字号、publisherStyles、滚动/分页、页边距等），并且偏好是可编辑/可合并/可存储的。([readium.org][4])
* **ContentService**：可迭代抽取出版物内容（文本/图片/段落等），用于搜索、TTS、索引等（注意它也提醒这是昂贵操作，需要缓存）。([readium.org][5])

> 这意味着：你不用从 0 自己踩“EPUB 兼容性 + WebView 各种坑 + CFI/位置计算”的雷；你主要做“对齐你自己的契约”和“产品体验层”。

## 3) 在你现有 `core/reader/api` 下，EPUB 引擎怎么分层最稳

你目录里已经给了一个很理想的模板（`internal/open/parser/locator/render/provider/cache`）。我建议 EPUB 具体这样落：

### 3.1 open：EpubOpener（最关键的“边界层”）

职责：**把 DocumentSource 变成“可用的 Publication + Resource container + metadata/cover”**

推荐做法：

* **把 SAF/Uri 输入先落到 app 私有目录**（或至少拿到稳定的 FileDescriptor + 可随机读）
  原因：

  * zip 内资源读取需要随机访问
  * WebView 资源加载、缓存策略、权限隔离都更可控
* 用 Readium Streamer 打开并解析成 Publication（你可以在内部直接使用 `EpubParser`），并生成 positions。([readium.org][3])

> 你在 `core/reader/runtime` 已经有 `BookFormatDetector`，所以 opener 只需要专心“打开并组织资源”。

### 3.2 doc：EpubDocument（ReaderDocument 实现）

职责：把 Readium 的 Publication 模型映射到你 `core/model` 的：

* `DocumentMetadata`（title/authors/language/identifier/description）
* `DocumentCapabilities`（reflow/fxl、search、outline、annotations…）
* `Outline`（nav document / ncx 的映射）

Readium 本身明确 EPUB 支持搜索、高亮等能力，你可以直接反映到 capabilities。([GitHub][2])

### 3.3 session/controller：把“导航与渲染”收口为一个 Controller

对 EPUB 来说，我建议你把 `ReaderController` 的核心抽象定成：

* `goTo(locator)`：跳到某个位置（最重要）
* `goForward()/goBackward()`：翻页/滚动下一屏
* `setPreferences(RenderConfig)`：提交字体/字号/主题/滚动/分页等
* `observeCurrentLocator()`：持续输出当前 Locator（用来存进度）

**关键点**：
EPUB 的“页”是**强烈依赖设置和屏幕的**，所以不要把“页码”当成主定位；主定位一定要是**内容锚点**（CFI / href#fragment）。

## 4) Locator：EPUB 必须用 CFI（别用 pageIndex/charOffset）

### 4.1 为什么 CFI 是最稳的主定位

EPUB CFI（Canonical Fragment Identifier）就是为“引用 EPUB 内任意内容位置/范围”设计的标准方案。([W3C GitHub][6])
它能跨：

* 字体大小变化
* 行距/页边距变化
* 横竖屏变化
  依然尽量指向同一段 DOM 文本节点/范围。

### 4.2 推荐你的 Locator 结构（与你 core/model 对齐）

我推荐 EPUB Locator 里最少包含：

* `href`：spine item 的相对路径（如 `Text/chapter01.xhtml`）
* `cfi`：`epubcfi(...)`
* `progression`：0..1（全书进度，做进度条/“跳转百分比”很方便）
* （可选）`position`：Readium PositionsService 的序号/位置点（用于“全书页码感知”）

其中 “position/progression” 你可以用 Readium 的 positions 服务生成，它专门用于“生成稳定位置点”。([readium.org][3])

---

## 5) 渲染：WebView 是 EPUB Reflow 的正确姿势（不要硬上 ComposeText）

### 5.1 Reflowable EPUB：两种阅读模式都要支持

* **Paginated**（仿 iBooks 的左右翻页）：本质是 CSS columns（多栏布局）+ 横向滚动控制
* **Scrolling**：纵向滚动（更像网页）

Readium 的 Navigator 体系里已经覆盖了分页/滚动，并且提供偏好提交与设置约束（例如 reflow vs fixed-layout、publisherStyles、scroll vs paginated 等）。([readium.org][4])

### 5.2 FXL EPUB：当成“固定页面/画布”处理

FXL（固定版式）要点：

* spine item 常常就是一页（或一跨页）
* 重点是 fit（宽/高/contain/cover）和缩放策略
* 位置通常更接近“页 + 内部锚点”，但仍建议保留 CFI

Readium 同时支持 reflow 与 FXL。([GitHub][2])

---

## 6) 偏好设置：一定要走“提交 preferences → 引擎重算 presentation”的模式

你如果自己写 WebView 注入 CSS，也能做，但会非常容易出现：

* 注入顺序问题（publisher css vs user css）
* 章节切换后设置丢失
* 某些设置互斥（例如 publisherStyles 开启时某些 spacing 失效）

Readium 的做法是：**你提交 Preferences，Navigator 根据规则计算“有效 settings”，再更新 presentation**；并且提供 `PreferencesEditor` 帮你做范围/互斥规则处理。([readium.org][4])
这和你 `core/datastore` 的“设置存储”可以天然配合：

* datastore 保存“用户偏好”（字体、字号、主题、行距、publisherStyles…）
* `feature/reader` 只更新偏好
* 引擎 Controller 接收偏好并 `submitPreferences`

---

## 7) 搜索/全文索引：推荐“两段式”落地（先可用，再做强）

### 7.1 MVP：章节内/全书基础搜索

最简单可用的策略：

* 在当前 spine item 的 WebView 里做查找（或通过引擎的 search 能力）
* 返回命中结果为 Locator（cfi），支持 next/prev

Readium 本身标注“EPUB 支持搜索文本内容”。([GitHub][2])

### 7.2 进阶：用 ContentService 做离线全文索引（配合 core/work）

你如果想做“书架搜索/全书搜索/跨书搜索”，不要在 WebView 里硬爬 DOM。推荐：

* 用 Readium 的 `ContentService` 迭代抽取内容元素，每个元素自带 locator，可直接跳转/做 decoration。([readium.org][5])
* 在 `core/work` 后台做全文索引（SQLite FTS/自建倒排）
* 结果返回 Locator 列表给 UI

注意：Readium 也提醒 `content.text()` 这种整书抽取很昂贵，**要缓存**。([readium.org][5])

---

## 8) 标注/高亮：存 CFI Range，渲染用 Decoration（或你自己的注入）

推荐的数据契约：

* `Annotation.anchor = (startCfi, endCfi, href)`（必要时再加 snippet）
* 渲染时在当前页面加载完成后，应用 decorations

Readium 明确支持 Highlighting（Decoration API）。([GitHub][2])

## 9) 性能与坑点清单（你做 EPUB 一定会遇到）

1. **解压/缓存策略**：

   * 首次打开可以解压到 `cache/epub/{id}/`
   * 二次打开复用，避免重复 unzip
   * 做一个 CachePolicy：LRU + 按 bookId 清理

2. **资源加载的“base URL”问题**：

   * 相对路径、字体、svg、css import 会让 `file://` 方案很痛苦
   * 推荐走“本地资源服务器/统一拦截器”的方式（Readium 体系里通常会把资源组织成可读取的 container；你也可以自己做统一 ResourceProvider）

3. **字体与字体回退**：

   * EPUB 内嵌字体、用户自定义字体、系统字体 fallback 都要兼容
   * 最容易出 bug 的是“字体替换后位置漂移”，所以一定用 CFI 做定位

4. **脚本与安全**：

   * EPUB 可能带脚本或外链资源
   * 建议默认禁用网络加载（拦截所有 http/https），只允许引擎提供的本地资源

5. **分章/超大章**：

   * 有些书一个 XHTML 超大，WebView 首次布局会很慢
   * 可做“后台预处理拆分/压缩 css”、或者至少做预取下一章

---

## 10) 最小闭环（我建议你这样落地，最快出可用产品）

1. `EpubOpener`：打开 + metadata + cover + outline（nav/ncx）
2. `Locator`：`href + epubcfi + progression`（能保存/恢复进度）([W3C GitHub][6])
3. `Renderer`：WebView / EpubNavigator（先实现 scroll 或 paginated 其一）
4. `Preferences`：字号/主题/publisherStyles（提交后即时生效）([readium.org][4])
5. Search MVP：next/prev 命中返回 Locator
6. 标注：CFI range + decoration

---

### 最终结论（最推荐的一句话）

> **EPUB 引擎最推荐的实现是：WebView/Navigator 渲染 + CFI 定位 + positions/progression 进度 + content 抽取做搜索/索引；工程上优先用 Readium Kotlin Toolkit 做解析与导航底座，然后在 `engines/epub` 里把它适配到你的 `core/reader/api`。**([GitHub][2])
