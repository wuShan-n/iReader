下面给你一套**“专业级、可长期演进”的 EPUB 引擎实现方案**，并且会严格贴合你现有的模块/契约设计（`core/reader/api` + `core/reader/runtime` + `engines/epub`），同时指出**为了做到真正专业**你需要在契约层做的两处关键升级（否则 EPUB 只能做成“能看但不好用”的半成品）。

---

# 最推荐方案（结论）

## 方案选择：**Readium Kotlin Toolkit 做底座 + 你自有契约做统一门面 + 引擎内部封装 WebView 导航器**

原因很直接：

* EPUB3 生态里最难的不是“能打开”，而是：

  * **EPUB CFI 定位/选区/高亮**（跨 DOM、跨 spine item 的稳定定位）
  * **WebView 渲染与分页/滚动、RTL、字体/主题/行距等偏好落地**
  * **资源服务（ZIP 内资源、相对路径、媒体类型、缓存、跨域限制）**
  * **高亮 decoration / selection 与 UI 交互**
* Readium Kotlin Toolkit 正是为这些低层能力而生，且是 Kotlin/Android 方向的成熟方案（工具箱式，不替你写 UI），并且是宽松的 BSD-3-Clause 许可。([GitHub][1])
* 它包含明确的 **WebView 集成体系**（JS bridge、资源拦截/服务、分页/滚动行为等），避免你自己踩一年的坑。([DeepWiki][2])
* 还能在需要时拓展到 LCP（若你未来要 DRM），Readium 官方也明确了 LCP 的依赖形态（需要额外二进制 liblcp）。([Readium][3])

> 你仍然保留你这套“多格式引擎统一契约”的架构优势：feature 永远不直接依赖 Readium（也不直接依赖 engines），而是只依赖 `core/reader/api` 或 `core/reader/runtime`。

# 你现有契约里，要做的两处“专业级”升级（非常关键）

你现在的 `RenderContent.Html(contentUri)` 设计可以做到“显示章节”，但要做到**专业阅读体验**（页内翻页/滚动、selection、annotation decorations、内部链接跳转、准确的当前位置同步），引擎必须能**驱动**渲染表面（WebView）或至少能和它形成稳定的双向通道。

我建议你在 `core/reader/api` 增加两项能力（都保持“引擎不暴露具体 WebView 类型”，feature 仍然只面向契约）：

## 1) 增加 RenderSurface 抽象（引擎可绑定渲染表面）

新增：

```kotlin
interface RenderSurface : Closeable {
    suspend fun load(url: String)
    suspend fun evalJs(script: String): String?
    fun setMessageListener(listener: (String) -> Unit)
}
```

并在 `ReaderController` 增加：

```kotlin
suspend fun bindSurface(surface: RenderSurface): ReaderResult<Unit>
```

这样：

* EPUB 引擎可以用 JS 来做分页/滚动/定位/选区/高亮
* feature 只需要提供一个 WebViewSurface 实现（放在 `core/common-android` 或 `feature/reader`），不会破坏你的依赖规则

## 2) 扩展 ReaderEvent（至少加入 SelectionChanged / LinkActivated）

你现在 `ReaderEvent` 是 sealed 且只有 3 个事件，**引擎无法扩展事件类型**（sealed 限制），这会卡死 EPUB 的交互（比如点链接、选中文本）。

推荐改法：把 `ReaderEvent` 从 `sealed interface` 改成普通 `interface`，或者保留 sealed 但预留通用事件：

```kotlin
data class Custom(val type: String, val payload: String) : ReaderEvent
data class SelectionChanged(val range: LocatorRange?, val rect: NormalizedRect?) : ReaderEvent
data class LinkActivated(val target: LinkTarget) : ReaderEvent
```

Readium 本身就有 selection 概念（含 locator + rect）([Readium][4])，以及 decoration 概念([Readium][5])；你只要把它们映射进你的事件/模型体系即可。

> 这两处升级是“专业 EPUB”与“能打开就行”的分水岭。否则你会被迫把大量交互逻辑塞进 feature 的 WebView 里，最后契约形同虚设。

# engines/epub 的专业实现蓝图（按你目录结构落地）

下面按你建议的 `engines/epub/internal/*` 给出“应当实现什么、怎么实现、关键边界”。

## 0) 依赖选型（Readium 模块）

在 `engines/epub` 里引入 Readium Kotlin Toolkit 的核心模块（至少 shared + streamer/opening + navigator）：

* 打开/解析：Readium 的 `AssetRetriever` + `PublicationOpener` 体系([Readium][6])
* 定位：Readium Locator 的 Locations 里就有 partial CFI 等信息([Readium][7])
* 渲染交互：Readium 的 navigator / WebView integration 体系([DeepWiki][2])
* 许可：BSD-3-Clause（适合商用集成）([Readium][8])

（版本建议跟 Readium 最新稳定线走，至少 3.x 系列；你项目已有 Version Catalog 管理很合适。Readium 仓库显示 3.1.2 在 2025-09 左右发布。([GitHub][9])）

---

## 1) internal/open：EpubOpener（专业级打开策略）

### 目标

* 支持 SAF Uri（content://）与 file://
* 避免 ZipFile 只能读文件路径带来的限制
* 做到：

  * 快速校验是不是 EPUB（mimetype + container.xml）
  * 稳定缓存（避免每次打开都解压/扫描）
  * 安全（Zip Slip、防路径穿越）

### 推荐实现

**把 DocumentSource 先落地到引擎缓存目录的“只读镜像文件”**，然后基于该文件走 ZipFile / Readium Asset。

流程：

1. `DocumentSource.openInputStream()` -> copy 到 `cacheDir/epub/{docId}.epub`
2. 生成 `DocumentId`：建议 sha-256(file bytes) 或 (size + hash prefix)
3. 校验 ZIP：

   * `mimetype` entry 必须存在且内容为 `application/epub+zip`
   * `META-INF/container.xml` 必须存在
4. 建立 `ResourceIndex`：

   * spine readingOrder
   * manifest href -> entryName
   * mediaType 映射（优先 OPF manifest，其次按扩展名猜）
5. 缓存策略：

   * 不要全量解压；按需解压资源到 LRU（图片/字体/章节 xhtml）更合理
   * 对“章节 xhtml”可做预解压（prefetch）以提升翻页体验

> Readium 的 opening/asset 体系就是围绕“先取 Asset、再 open Publication”设计的，你可直接套用。([Readium][6])

---

## 2) internal/parser：EpubParser（解析与统一模型）

你如果用了 Readium，解析层的核心是：**把 Readium 的 Publication 模型转换为你自己的 DocumentMetadata、Outline、Capabilities**，并保留足够的 extras 让后续渲染/定位正确。

### 必须提取的专业字段

* metadata：

  * title/author/language/identifier
  * reading progression direction（LTR/RTL）
  * layout hints（reflow/fixed layout）——EPUB3 可能是 fixed-layout
  * accessibility metadata（有就展示/存储）([W3C][10])
* spine/readingOrder：

  * href 列表
  * 每一项 mediaType
  * nav doc / ncx

### Capabilities（建议）

* EPUB 一般：

  * reflowable = true
  * outline = true
  * search = true（你实现 SearchProvider）
  * textExtraction = true
  * links = true
  * annotations = true
* 若检测到 fixed-layout EPUB（可选支持）：

  * fixedLayout = true（并在 runtime 默认 RenderConfig 走 FixedPage 或专用配置）

---

## 3) internal/locator：EpubCfi + LocatorMapper（专业的定位体系）

### 为什么必须用 CFI（而不是 href#id 或纯 percent）

* href#id 对选区/高亮不稳定
* percent 无法恢复到“字级位置”
* CFI 是标准、可跨排版恢复的定位方式（并且 W3C 还在维护 CFI 1.1 规范）([W3C GitHub][11])

### 你的 Locator 设计与 Readium 的对齐

你的 `Locator(scheme="epub.cfi", value=...)` 是对的。

建议 extras 至少携带：

* `href`：spine item href
* `progressionInSpine`：0..1（页内 progression）
* `totalProgression`：0..1（全书 progression）
* `titleInView`（可选）

Readium 的 Locator 本身就有 `href` 和 `locations`（含 progression/fragment/partialCfi 等）([Readium][12])，你的 `LocatorMapper` 做双向映射即可。

### 专业级“normalize”

实现 `LocatorNormalizer`：

* 处理：

  * CFI 指向已被修复/清理后的 DOM（容错）
  * 同一个逻辑位置的多种 CFI 表达
* 建议策略：

  * normalize 时用 Readium/JS 在 DOM 中“resolve CFI -> nearest visible text node -> regenerate CFI”

---

## 4) internal/render：EpubRenderer（WebView + JS + 资源服务）

### 总体策略

* **渲染引擎就是 WebView（Chromium）**：这是专业阅读器的主流选择，因为 EPUB 的 HTML/CSS 复杂度远超自研排版引擎。
* 引擎负责：

  * 给 WebView 提供稳定的 base URL
  * 注入 CSS（主题/字号/行距/边距/字体）
  * 注入 JS（分页/定位/选区/高亮/链接拦截）
  * 将 WebView 的“当前位置”转成 CFI Locator 并持续同步到 `ReaderController.state`

Readium 已经有成熟的 WebView 集成体系（专用 WebView 类、JS bridge、资源服务机制、分页/滚动）([DeepWiki][2])，你最推荐的做法是：**引擎内部封装 Readium 的 navigator**，对外仍然只暴露你的 `ReaderController + Providers`。

---

## 5) internal/provider：Outline/Text/Search/Resource/Annotation

### 5.1 OutlineProvider（目录）

* 用 nav doc（EPUB3）或 NCX（EPUB2）生成树
* 每个节点的 locator：用对应章节位置的 CFI（或 href + fragment，再 normalize 到 CFI）

Readium Publication 模型里通常能取到 TOC（table of contents），你做映射即可。

### 5.2 ResourceProvider（资源）

**专业点在于：同一份资源路径必须对 WebView 与引擎 API 一致**：

* `openResource(path)` 既要给 WebView 拦截请求用，也要给业务（比如导出封面）用
* MIME 类型：优先 OPF manifest，fallback 扩展名

并做好：

* 路径规范化（去掉 `../`、防 Zip Slip）
* 缓存（按 entry size 分类：小资源内存缓存，大资源磁盘缓存）

### 5.3 TextProvider（复制/分享/导出）

Readium 有 “content extraction” 指南：Content iterator 会产出段落/标题/图片等语义元素，每个元素都带 locator，可用于导航和 decoration。([Readium][13])
你可以：

* `getTextAround(locator)`：定位到最近 Content.Element，拼接附近若干元素的 text
* `getText(range)`：更难，需要按 CFI range 截取 DOM 文本（建议走 JS resolve range，然后取 textContent）

### 5.4 SearchProvider（全书搜索，专业实现）

这里不要走“遍历字符串 contains”这种玩具实现。

推荐两层体系：

**A. 即时搜索（不建索引，流式返回）**

* 基于 Content iterator（段落粒度）扫描
* 每段落找到匹配就 emit SearchHit
* SearchHit.range：

  * 最理想是精确 CFI range（需要 DOM 级定位）
  * 若阶段 1 先给段落级 locator，也可以先把 range start/end 设为同一 locator，并在 UI 点击后再二次精确定位（但我建议你把 contract 改成 `SearchHit.target: Locator` + `range: LocatorRange?` 更合理）

**B. 索引搜索（专业性能）**

* 在 `core/work` 或 `core/data` 做“后台索引任务”
* 索引内容建议来自 Content iterator（语义块），存：

  * chunkId
  * chunkText
  * chunkLocator（CFI）
* Room FTS（或 SQLite FTS5）做全文检索，命中后再把 chunkLocator + offset 映射回 CFI range（可选）

Readium 侧也在推进 SearchService 抽象（`search(query) -> SearchIterator`）([Readium][14])，但不同版本/实现细节你需要评估是否直接可用；我更建议你把“索引”当成你自己掌控的能力，Readium 只负责“可靠地吐出可定位的文本块”。

### 5.5 AnnotationProvider（高亮/笔记/书签）

你契约里 anchor 已经很好：reflow 用 `LocatorRange`。

专业 EPUB 高亮链路：

1. WebView JS 获取 selection -> CFI range（start/end）
2. 引擎返回 `AnnotationDraft(anchor=ReflowRange(range))`
3. 存储层保存
4. 渲染时 `decorationsFor()` 把 range 转为 Decoration.Reflow
5. WebView JS 根据 CFI range 渲染高亮（span/mark overlay）

Readium 的 navigator 本身就有 Decoration 概念([Readium][5])，也有 Selection（含 locator + rect）([Readium][4])；你可以直接桥接：

* Readium DecorationId <-> 你的 AnnotationId
* Readium Locator <-> 你的 Locator(epub.cfi)

# EpubEngine / EpubDocument / EpubSession 的“专业实现”职责边界

按照你的目录结构落地（结合 Readium）建议是：

## `EpubEngine`（唯一入口）

* `supportedFormats = setOf(EPUB)`
* `open(source, options)`：

  * 调 `EpubOpener`：镜像文件 + 校验 + 建立 Publication
  * 返回 `EpubDocument`

## `EpubDocument`（持有 Publication + Asset 生命周期）

* `metadata()`：映射 Readium metadata -> DocumentMetadata
* `capabilities`：按解析结果设置（reflow / fxl）
* `createSession()`：

  * 构建 `EpubSession`：

    * controller（可 bind RenderSurface）
    * providers（outline/search/text/resources/annotations）

## `EpubSession`

* 负责把：

  * 阅读位置 state（locator、progression、titleInView）
  * 渲染偏好（RenderConfig）
  * 资源服务（ResourceProvider）
  * JS bridge 通道
    统一编排起来

# WebView 安全与兼容性（专业项目必须写进方案）

EPUB 本质是“可执行 HTML 内容”，专业阅读器必须默认做到：

* 禁止外部网络请求（避免加载跟踪像素/远程脚本）

  * 所有请求必须被 `ResourceProvider`/拦截器接管
* 限制 file:// 权限与跨域
* 对 `javascript:` URL、外链做策略：外链只能走 `LinkTarget.External`，交给系统浏览器/自定义 tab
* ZIP entry 路径必须做 normalize，防 `../` 跳目录
* 对超大图片/字体做解码与缓存上限，避免 OOM

Readium 的 WebView 集成里对资源拦截、JS bridge、分页行为已经有系统化设计，你封装它能少走很多弯路。([DeepWiki][2])

# 若你“必须完全自研”（不依赖 Readium）的备选路线（不推荐，但给你专业路线图）

如果你强约束“不能引入 Readium”，那你至少要自研/集成以下组件，工作量会爆炸式上升：

1. **EPUB 包解析**（container.xml + OPF + nav/ncx + manifest/spine）
2. **资源服务层**（ZIP random access + MIME + cache + 安全）
3. **CFI 全量实现**（生成/解析/resolve/容错恢复）——按 W3C CFI 1.1 实现([W3C GitHub][11])
4. **WebView 分页与定位系统**（JS + CSS columns/scroll + progress 计算）
5. **选区转 CFI range**（DOM Range -> CFI）+ 高亮渲染（CFI range -> DOM wrap）
6. **全文索引与 range 映射**（offset -> DOM text node -> CFI）

你会基本复刻一套 Readium 生态里已经多年打磨的底层系统，因此“最推荐”仍是用 Readium 做底座。

# 最后：与你现有架构的对齐总结

* `engines/epub` 只实现 `core/reader/api`，内部依赖 Readium（对外不可见）
* `app` 做 DI 组装，把 `EpubEngine` 多绑定进 `EngineRegistry`

同时，为了做到专业体验：

* **在 `core/reader/api` 加 RenderSurface + 扩展 ReaderEvent**（两处升级）
