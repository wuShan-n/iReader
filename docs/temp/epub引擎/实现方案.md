下面是**在你现有契约完全不改（或只做极小补充）的前提下**，实现 **EPUB 引擎**我最推荐的一套方案：**WebView 驱动的 Reflow 渲染（稳定、快落地）+ 以 CFI 为核心定位（可逐步补全）+ Providers 分层（outline/search/text/resources/annotations）**。它能很好适配你现在的 `RenderContent.Html`、`Locator(epub.cfi)`、`ReaderController` 等接口。

## 0) 结论先说：EPUB 最推荐路线

### 推荐方案（MVP → 可演进）

1. **MVP（最稳、最短路径）**

* 渲染：**WebView + 章节（spine item）级别**加载
* `next/prev`：章节切换（把“章节”当 RenderPage）
* Locator：先用“伪 CFI / href#fragment”过渡，后续再正规化成真 CFI
* Outline：支持 EPUB3 `nav.xhtml` + EPUB2 `toc.ncx`
* Resources：解压缓存 + `ResourceProvider` +（可选）ContentProvider 提供 `content://` baseUri
* Search/Text：先做“按章节提取纯文本 + 流式匹配”，命中后返回范围 locator（先粗后细）

2. **增强（用户体验更像真阅读器）**

* 渲染：WebView **CSS columns 分页**（一章多“页”）
* `next/prev`：章内翻页，翻完再跨章
* 增加 `LocatorSchemes.REFLOW_PAGE` 作为“稳定分页缓存定位”（你模型里已经预留）
* CFI：逐步把 “href#id → CFI” 做到可用（配合 `LocatorNormalizer`）

3. **高级（标注、精确定位、跨设备一致）**

* 真正的 **CFI 生成/解析/归一化**（支持 selection、链接跳转、TOC 精确落点）
* Search/Text 提供精确 `LocatorRange`（CFI range），标注锚点稳定

## 2) 渲染实现：为什么我强推 WebView（且如何与你 API 对齐）

你已经有 `RenderContent.Html(inlineHtml/contentUri/baseUri)`，这几乎就是为 EPUB 准备的。

### MVP 渲染策略（章节级）

* 每个 spine item（一个 xhtml/html）对应一个 `RenderPage`
* `render()` 返回当前章节
* `next()`：spineIndex + 1，`prev()`：spineIndex - 1
* `goTo(locator)`：先把 locator 解析到 spineIndex（先粗略到章节顶部），后续再精确到 fragment/CFI

**RenderPage 示例（章节级）**

```kotlin
val page = RenderPage(
  id = PageId("epub:${docId.value}:spine:$index"),
  locator = currentLocator, // epub.cfi（先可粗略）
  content = RenderContent.Html(
    contentUri = chapterUri,    // content://epub/<docId>/spine/<index>
    baseUri = baseUriForDoc     // content://epub/<docId>/
  ),
  links = emptyList(),
  decorations = emptyList()
)
```

### 推荐的资源加载方式（强烈建议）

**用 ContentProvider（或等价机制）提供 `content://` 资源**，让 WebView 能稳定加载图片/css/font：

* baseUri：`content://com.xxx.epub/<docId>/`
* spine html：`content://com.xxx.epub/<docId>/spine/<i>.xhtml`
* 资源：`content://com.xxx.epub/<docId>/res/...`

> 你引擎模块本身可以带一个 Provider（清单合并到 app），实现里只需要访问你解压目录（cacheDir/epub/docId/…）。这能避免 `file://` 权限坑，也避免 UI 里写一堆拦截逻辑。

## 3) Locator/CFI：最推荐的“渐进式”做法（与你现有模型完美契合）

你已经定义 `LocatorNormalizer`，这正好解决“先粗后精”的迁移问题。

### 3.1 立刻可用的定位表示（MVP）

* 统一 `scheme = LocatorSchemes.EPUB_CFI`
* `value` 先允许：

  * `spine:<index>`（章节顶部）
  * `href:Text/chap01.xhtml#p3`（TOC/链接常见）
* 后续由 `EpubLocatorNormalizer.normalize()` 逐步把它变成真 CFI（`epubcfi(...)`）

```kotlin
// 过渡期 locator（仍然 scheme=epub.cfi）
Locator(
  scheme = LocatorSchemes.EPUB_CFI,
  value = "href:Text/ch01.xhtml#p3",
  extras = mapOf("kind" to "href")
)
```

### 3.2 增强分页后：引入稳定“页定位”

你模型里已经预留 `LocatorSchemes.REFLOW_PAGE`，分页后强烈建议用它做“缓存页锚点”：

* `reflow.page` value = `spineIndex:pageIndex`（或加 configHash）
* 同时保留一个对应的 `epub.cfi`（用于跨设备、标注锚点）

**这能解决 WebView columns 分页时“CFI 精确落点很难”**的问题：页翻稳定、书签也稳定；等 CFI 做强了再逐步切回。

## 4) Search/Text/Outline/Resources：按你 Providers 契约的最佳实现方式

### 4.1 ResourceProvider（必做）

* 直接从解压目录按相对路径开流
* MIME 优先从 OPF manifest 的 media-type；否则按后缀猜

```kotlin
class EpubResourceProvider(...) : ResourceProvider {
  override suspend fun openResource(path: String): ReaderResult<InputStream> = ...
  override suspend fun getMimeType(path: String): ReaderResult<String?> = ...
}
```

### 4.2 OutlineProvider（强烈建议做全）

支持两种：

* EPUB3：`nav.xhtml`（manifest properties=nav）
* EPUB2：`toc.ncx`

OutlineNode 的 locator：

* MVP：用 `href:...#...` 过渡（交给 normalizer / controller 解释）
* 增强：解析 fragment → 生成 CFI

### 4.3 TextProvider（建议 MVP 先做“章节文本”）

实现最稳的版本：

* `getTextAround(locator)`：定位到 spineIndex 后，提取该章节纯文本（限制 maxChars）
* `getText(range)`：若 range 暂时只有章节级定位，先返回章节文本片段（后续再精确）

纯文本提取建议：

* 用 HTML/XHTML 解析库（比如 jsoup）提取 `body.text()`（对搜索/复制足够）
* 保留一个 `sectionTitle` 可来自 TOC 或 spine item title

### 4.4 SearchProvider（推荐“流式 + 分章索引 + 可中断”）

你的接口是 `Flow<SearchHit>`，非常适合边搜边出结果：

* open 后异步构建索引（按 spine item 抽文本 + 建倒排或简单 contains）
* `search(query)`：依次扫描章节，命中就 emit
* `startFrom`：定位到 spineIndex，从那一章开始搜

**命中 range 怎么给？**

* MVP：range 用“章节级 locator + 章节内偏移放 extras”（后续升级）
* 增强：可以改成 `LocatorRange(start/end)` 里真正 CFI range

示例（MVP 粗范围）：

```kotlin
SearchHit(
  range = LocatorRange(
    start = Locator(EPUB_CFI, "spine:12", extras = mapOf("charStart" to "1032")),
    end   = Locator(EPUB_CFI, "spine:12", extras = mapOf("charEnd" to "1066"))
  ),
  excerpt = "...",
  sectionTitle = "Chapter 5"
)
```

## 5) ReaderController：最推荐的两档实现（你可以先上 A，再迭代 B）

### A 档（MVP）：章节级 Controller（强推先做这个）

* state.locator：`spine:i`
* progression：`current=i+1, total=spineCount`（label=“12/60”）
* render()：返回当前章节 RenderPage（Html contentUri/baseUri）
* next/prev：切章节
* goToProgress：按 spine 比例跳

> 优点：实现简单、稳定、几乎不会踩 WebView 分页坑，能快速把 EPUB 打通所有业务链路（目录/搜索/复制/标注数据结构都能跑）。

### B 档（增强）：columns 分页 Controller

* 在 wrapper html 注入 CSS columns + JS 计算页数
* next/prev：

  * 章内 pageIndex++/--
  * 越界再跨章
* state.progression：用 `reflow.page` 做页进度（更贴近真实阅读器）
* prefetchNeighbors：预解压/预读相邻章节、预生成 wrapper uri

> 这一档需要 UI(WebView)配合：加载后按 URL 参数滚到指定列；必要时通过 JS bridge 回传“实际页数/当前页”给 controller 更新 state（否则你只能估算）。

## 6) 打开/缓存：EPUB opener 的关键细节（非常容易踩坑）

### 必须做的 4 件事

1. **安全解压**：禁止 `../` 路径穿越；只允许写入 doc 专属目录
2. **容器解析**：`META-INF/container.xml` -> OPF path
3. **OPF 解析**：manifest/spine/metadata（至少 spine+href+mediaType）
4. **缓存策略**：`cacheDir/epub/<docId>/`，docId 建议 hash(uri + sizeBytes + displayName)

### Document.close() 建议策略

* 不要立刻删解压目录（会影响返回阅读）
* 用 LRU/WorkManager 做后台清理（你 core/work 已预留）

## 7) 引擎对外类的最小骨架（贴合你当前接口）

### EpubEngine

```kotlin
class EpubEngine(
  private val opener: EpubOpener
) : ReaderEngine {

  override val supportedFormats = setOf(BookFormat.EPUB)

  override suspend fun open(source: DocumentSource, options: OpenOptions): ReaderResult<ReaderDocument> {
    return opener.open(source, options) // Ok(EpubDocument) / Err(...)
  }
}
```

### EpubDocument（createSession 组装 providers + controller）

```kotlin
internal class EpubDocument(
  override val id: DocumentId,
  private val container: EpubContainer,
  override val openOptions: OpenOptions
) : ReaderDocument {

  override val format = BookFormat.EPUB

  override val capabilities = DocumentCapabilities(
    reflowable = true,
    fixedLayout = false,
    outline = container.hasToc,
    search = true,          // 做不到就 false
    textExtraction = true,  // 能抽纯文本就 true
    annotations = true,     // 业务支持就 true
    links = true
  )

  override suspend fun metadata(): ReaderResult<DocumentMetadata> = ReaderResult.Ok(container.metadata)

  override suspend fun createSession(initialLocator: Locator?, initialConfig: RenderConfig): ReaderResult<ReaderSession> {
    val session = EpubSession(container, initialLocator, initialConfig)
    return ReaderResult.Ok(session)
  }

  override fun close() { /* 释放引用/关闭索引句柄等 */ }
}
```
