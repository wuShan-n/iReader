## ✅ 新增 1：页数度量持久化存储（内存 + properties 文件）

### `engines/epub/.../internal/pagination/EpubPageMetricsStore.kt`（新增）

```kotlin
package com.example.reader.engines.epub.internal.pagination

import java.io.File
import java.io.FileInputStream
import java.io.FileOutputStream
import java.util.Properties
import java.util.concurrent.ConcurrentHashMap

internal class EpubPageMetricsStore(
    private val file: File
) {
    // key = "spine:sig" -> pages
    private val map = ConcurrentHashMap<String, Int>()
    @Volatile private var dirty: Boolean = false

    init {
        load()
    }

    fun getPages(spine: Int, sig: Int): Int? = map["$spine:$sig"]

    fun putPages(spine: Int, sig: Int, pages: Int) {
        val v = pages.coerceIn(1, 5000)
        val key = "$spine:$sig"
        val old = map.put(key, v)
        if (old != v) dirty = true
    }

    fun flush() {
        if (!dirty) return
        synchronized(this) {
            if (!dirty) return
            val props = Properties()
            for ((k, v) in map.entries) props.setProperty(k, v.toString())
            file.parentFile?.mkdirs()
            FileOutputStream(file).use { props.store(it, "epub page metrics") }
            dirty = false
        }
    }

    private fun load() {
        if (!file.exists()) return
        runCatching {
            val props = Properties()
            FileInputStream(file).use { props.load(it) }
            for (name in props.stringPropertyNames()) {
                val v = props.getProperty(name)?.toIntOrNull() ?: continue
                map[name] = v.coerceIn(1, 5000)
            }
        }
    }
}
```

#### 5.4 goTo(REFLOW_PAGE) 支持 sig mismatch 迁移

在 `LocatorSchemes.REFLOW_PAGE` 分支里，把解析后赋值逻辑替换为：

```kotlin
val parsed = parseReflowPageFull(locator.value)
    ?: return ReaderResult.Err(ReaderError.CorruptOrInvalid("Invalid reflow.page: ${locator.value}"))
val (s, p, oldSig) = parsed

val targetSpine = s.coerceIn(0, container.spineCount - 1)
val newSig = sig() // 当前 layout/config 的 signature

spineIndex = targetSpine

val newCount = paginator.pageCount(spineIndex, constraints, config)
val mappedPage = if (oldSig != null && oldSig != newSig) {
    // ✅ sig 不匹配：按旧页数比例迁移到新页数
    val oldCount = metricsStore.getPages(spineIndex, oldSig) ?: newCount
    if (oldCount <= 1 || newCount <= 1) 0
    else {
        val ratio = p.toDouble() / (oldCount - 1).toDouble()
        (ratio * (newCount - 1).toDouble()).roundToInt()
    }
} else p

pageIndex = mappedPage.coerceIn(0, (newCount - 1).coerceAtLeast(0))
pendingAnchorId = null
```

并新增解析函数（放在 controller 内部）：

```kotlin
private data class ReflowKey(val spine: Int, val page: Int, val sig: Int?)

private fun parseReflowPageFull(value: String): ReflowKey? {
    val parts = value.split(':')
    if (parts.size < 2) return null
    val s = parts[0].toIntOrNull() ?: return null
    val p = parts[1].toIntOrNull() ?: return null
    val sig = parts.getOrNull(2)?.toIntOrNull()
    return ReflowKey(s, p, sig)
}
```

> 兼容旧的 `"spine:page"`（sig 为 null）。

#### 5.5 goTo() 支持 metricsOnly（只更新 store/state，不重渲染）

在 goTo() 一开始加这段（在 when 分支前/或分支内部都行）：

```kotlin
val metricsOnly = locator.extras["metricsOnly"] == "1"
val pagesExtra = locator.extras["pages"]?.toIntOrNull()
val sigExtra = locator.extras["sig"]?.toIntOrNull()
if (pagesExtra != null && sigExtra != null) {
    metricsStore.putPages(
        spine = locator.value.substringBefore(':').toIntOrNull() ?: spineIndex,
        sig = sigExtra,
        pages = pagesExtra
    )
}
```

然后在处理完 spineIndex/pageIndex 更新、`_state.value = buildState()` 之后，如果 `metricsOnly`：

```kotlin
if (metricsOnly) {
    // ✅ 只对齐 state，不触发 renderCurrent（WebView 已在正确位置）
    _state.value = buildState()
    return ReaderResult.Ok(lastRendered ?: run {
        // 没有历史渲染就兜底渲染一次
        return render(policy)
    })
}
```

> 这样 metrics 高频上报不会造成“重渲染风暴”。

## ✅ 修改 6：Feature 侧 metrics 路由：不再 setConfig，只用 goTo(metricsOnly) 更新引擎 store

### `feature/reader/.../web/ReaderWebViewLinkRouter.kt`（修改 metrics 分支）

把你之前 `metrics` 分支里“setConfig 写 extra”的部分删掉，改成：

```kotlin
"metrics" -> {
    val spine = uri.getQueryParameter("spine")?.toIntOrNull() ?: return true
    val sig = uri.getQueryParameter("sig")?.toIntOrNull() ?: return true
    val pages = uri.getQueryParameter("pages")?.toIntOrNull() ?: return true
    val page = uri.getQueryParameter("page")?.toIntOrNull() ?: return true

    // ✅ 防抖：只处理当前章节的 metrics（避免切章加载旧页面的回调污染）
    val cur = controller.state.value.locator
    val curSpine = cur.value.split(':').getOrNull(0)?.toIntOrNull()
    if (cur.scheme == LocatorSchemes.REFLOW_PAGE && curSpine != null && curSpine != spine) return true

    scope.launch {
        controller.goTo(
            Locator(
                scheme = LocatorSchemes.REFLOW_PAGE,
                value = "$spine:$page:$sig",
                extras = mapOf(
                    "pages" to pages.toString(),
                    "sig" to sig.toString(),
                    "metricsOnly" to "1"
                )
            )
        )
    }
    true
}
```

# ✅ 这轮优化实现了什么

1. **真实页数不再依赖 RenderConfig.extra**

* 不会因为 metrics 写入导致 config/hash 变化、缓存失效或 setConfig 循环。

2. **页数可持久化**

* `metrics.properties` 写到 `cacheDir/epub/<docId>/metrics.properties`，下次打开同书同布局签名可直接用真实页数。

3. **sig 不匹配也能“迁移定位”**

* 用户改字体/旋转屏幕（sig 变了）后，用旧的 `reflow.page` 定位不会直接跳错很多页，会按旧页数比例映射到新页数。

4. **metrics 高频上报不触发重渲染**

* `metricsOnly=1` 只对齐 state + 更新 store，不走 renderCurrent。

---

如果你要继续下一步（仍不改 core API），最建议做两件：

* **把“当前页/pageCount 的真实值”反哺 Progression.percent**（现在 percent 仍可能用估算总页，下一步可用 store + 全书 mapper 缓存）。
* **内部链接 bounds（热区）可选增强**：JS 计算 anchor rects 回传，填充 `RenderPage.links.bounds`（reflow 下也能有坐标）。
